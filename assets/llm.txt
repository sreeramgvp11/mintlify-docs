# HydraDB Complete Reference for LLMs

> **Graph-first, unified state infrastructure for AI.** HydraDB replaces Neo4j + VectorDB + Redis with a single API that combines graph traversal, semantic search, and recency scoring to find what's useful, not just what's similar.

**Base URL:** `https://api.hydradb.com`  
**Authentication:** `Authorization: Bearer <your_api_key>` on every request  
**SDKs:** TypeScript `npm install @hydradb/sdk` | Python `pip install hydradb-sdk`

**Current SDK Versions:**
- TypeScript: Check npm for `@hydradb/sdk` latest version
- Python: Check PyPI for `hydradb-sdk` latest version
- MCP Server: `@hydradb/mcp@0.0.1` (verify latest on npm)
- CLI: `hydradb-cli` on PyPI

---

## Critical Rules for LLMs

1. **Always poll status endpoints** — Tenant creation and ingestion are async operations
2. **Vectorstore index mapping** — Index `[0]` = Memories (recall_preferences), Index `[1]` = Knowledge (full_recall)
3. **Endpoint selection** — Use `recall_preferences` for user state, `full_recall` for documents
4. **Scope consistency** — Write and read must use the same `sub_tenant_id`
5. **Schema immutability** — Tenant metadata schema cannot be changed after creation
6. **Check package versions** — Don't assume SDK versions; always verify npm/PyPI for latest

## HTTP Status Codes & Error Handling

| Code | Meaning | Common Causes | Fix |
|------|---------|---------------|-----|
| **200** | Success | Request completed | Check response body for partial failures in bulk operations |
| **401** | Unauthorized | Missing/invalid API key | Verify `Authorization: Bearer <key>` header format |
| **404** | Not Found | Tenant doesn't exist or still provisioning | Poll `/tenants/infra/status` until ready |
| **422** | Validation Error | Invalid request body | Check: file_metadata array length, app_knowledge stringified, metadata schema match |
| **429** | Rate Limited | Too many requests | Use bulk endpoints, implement exponential backoff, cache repeated queries |
| **500** | Internal Error | Server-side issue | Retry with backoff, check status.hydradb.com |

**Common 422 Causes:**
- `file_metadata` array length doesn't match `files` count
- `app_knowledge` sent as JSON object instead of JSON-encoded string
- Metadata field not declared in tenant schema
- `MemoryItem` with no `text` and no `user_assistant_pairs`
- Boolean_recall called with `alpha`, `mode`, or `graph_context` params

## Philosophy & Differentiators

**Graph-first, not graph-optional** — The graph is the primary substrate; vectors are entry points for traversal

**Memories vs Knowledge as distinct primitives:**
- **Memories** = Dynamic, per-user state (preferences, conversation history, behavioral traits)
- **Knowledge** = Static, tenant-wide documents (PDFs, Slack, Notion, CSV, emails)
- Separate indices, separate endpoints, one unified API

**Composable & unopinionated:**
- Control embeddings model and dimensions
- Define graph schema and relationship types
- Parameterize retrieval (recency, alpha, forced relations, metadata)
- Use custom chunking strategies

**One endpoint replaces:** Neo4j + Pinecone/Weaviate + Redis

**Key metrics:**
- 90% on LongMemEvals
- Sub-200ms p95 retrieval latency
- In-memory graph traversal
- Multi-tenant isolation enforced at API layer

## Core Primitives

### Tenants & Sub-Tenants

**Tenant** — Top-level isolated workspace. No cross-tenant data access. `tenant_id` required on every API call.

**Sub-Tenant** — Logical partition within tenant (user, team, workspace, project). Pass `sub_tenant_id` to scope operations. If omitted, defaults to tenant's default sub-tenant.

**Design patterns:**

| Pattern | Configuration |
|---------|---------------|
| B2B SaaS | One tenant per customer; sub-tenants for teams/workspaces/users |
| B2C app | One tenant for app; sub-tenants per end user |
| Internal tools | One tenant per company; sub-tenants per department/project |
| Separate environments | Different `tenant_id` per env (prod/staging/dev) — NEVER share across envs |

### Knowledge

Shared, tenant-wide context from documents and app sources.

- **Sources:** PDF, DOCX, PPTX, Markdown, CSV, TXT, HTML, Slack, Notion, Gmail, Linear, GitHub
- **Processing:** Parsed → Chunked → Embedded → Graph-linked at write time
- **Mutability:** Versioned — replaced or deleted explicitly
- **Recall endpoint:** `POST /recall/full_recall`
- **Vectorstore index:** `[1]`

### Memories

User-scoped context — preferences, conversation history, inferred traits. Each memory scoped to `sub_tenant_id`.

- **Mutability:** Dynamic — evolves with every interaction
- **Recall endpoint:** `POST /recall/recall_preferences`
- **Vectorstore index:** `[0]`
- **Inference:** Set `infer: true` to extract implicit preferences from raw input
- **Upsert:** Set `upsert: true` to update existing memories by text match

### Recall Pipeline

Multi-stage retrieval combining:

1. **Metadata filtering** — Deterministic narrowing by tenant schema fields
2. **Hybrid retrieval** — Semantic (vector) + Keyword (BM25)
3. **Graph traversal** — Follow entity relationships
4. **Personalized ranking** — Recency, frequency, relevance scoring

### Metadata

**`metadata`** — Filterable fields defined in tenant schema at creation. Use `metadata_filters` at recall time. Schema is immutable after tenant creation.

**`additional_metadata`** — Free-form fields per document. Stored but NOT filterable at recall time. Use for flexible, unstructured context.

## API Reference

### Tenants

#### POST /tenants/create

Create isolated workspace (async operation).

**Request:**
```json
{
  "tenant_id": "acme_corp",
  "tenant_metadata_schema": [
    {
      "name": "category",
      "data_type": "VARCHAR",
      "enable_match": true,
      "enable_dense_embedding": false,
      "enable_sparse_embedding": false
    },
    {
      "name": "timestamp",
      "data_type": "VARCHAR",
      "enable_match": true,
      "enable_dense_embedding": false,
      "enable_sparse_embedding": false
    }
  ]
}
```

**Optional fields:**
- `is_embeddings_tenant: true` — Bring your own embeddings (requires `embeddings_dimension`)
- `embeddings_dimension: 1536` — Vector dimension for BYO embeddings tenants

**Response:**
```json
{
  "status": "accepted",
  "tenant_id": "acme_corp",
  "message": "Tenant creation started in the background..."
}
```

**Critical:** Always poll `/tenants/infra/status` before ingesting data.

#### GET /tenants/infra/status

Poll tenant provisioning status until ready.

**Request:**
```
GET /tenants/infra/status?tenant_id=acme_corp
```

**Response:**
```json
{
  "tenant_id": "acme_corp",
  "graph_status": true,
  "vectorstore_status": [true, true],
  "infra_status": "ready"
}
```

**Wait for:**
- `graph_status: true`
- `vectorstore_status: [true, true]` (both indices ready)
- `infra_status: "ready"`

#### GET /tenants/monitor

Get object counts and vector dimensions.

**Request:**
```
GET /tenants/monitor?tenant_id=acme_corp
```

**Response:**
```json
{
  "tenant_id": "acme_corp",
  "total_chunks": 12847,
  "total_sources": 432,
  "total_memories": 1523,
  "embedding_dimension": 1024,
  "graph_nodes": 8934,
  "graph_relationships": 15672
}
```

#### GET /tenants/sub_tenant_ids

List sub-tenants with indexed data.

**Request:**
```
GET /tenants/sub_tenant_ids?tenant_id=acme_corp
```

**Response:**
```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_ids": ["user_alice", "user_bob", "team_eng"]
}
```

#### DELETE /tenants/delete

Permanently delete tenant and all its data. Irreversible.

**Request (query param):**
```
DELETE /tenants/delete?tenant_id=acme_corp
```

**Response:**
```json
{
  "status": "accepted",
  "tenant_id": "acme_corp",
  "message": "Tenant deletion started..."
}
```

**Note:** Cannot re-create tenant with same ID until deletion completes.

### Ingestion

#### POST /ingestion/upload_knowledge

Ingest files or app sources (async, multipart/form-data).

**Endpoints used:** `POST /ingestion/upload_knowledge`, `POST /ingestion/verify_processing`

**Request (files):**
```bash
curl -X POST https://api.hydradb.com/ingestion/upload_knowledge \
  -H "Authorization: Bearer $API_KEY" \
  -F "tenant_id=acme_corp" \
  -F "sub_tenant_id=team_eng" \
  -F "files=@/path/to/doc.pdf" \
  -F "files=@/path/to/report.docx" \
  -F 'file_metadata=[
    {
      "file_id": "doc_001",
      "metadata": { "category": "legal", "timestamp": "2024-01-15" }
    },
    {
      "file_id": "doc_002",
      "metadata": { "category": "finance", "timestamp": "2024-02-20" }
    }
  ]'
```

**Request (app sources):**
```bash
curl -X POST https://api.hydradb.com/ingestion/upload_knowledge \
  -H "Authorization: Bearer $API_KEY" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "source_id": "slack_001",
      "source_name": "Slack: #engineering",
      "knowledge_items": [
        {
          "text": "Deployment completed successfully for v2.3.1",
          "metadata": { "category": "deployment", "timestamp": "2024-03-10" },
          "additional_metadata": {
            "channel": "engineering",
            "author": "alice@acme.com",
            "thread_ts": "1710086400"
          }
        }
      ]
    }
  ]'
```

**Critical rules:**
- `file_metadata` array length MUST match `files` count
- `app_knowledge` must be JSON-encoded STRING in multipart form
- `metadata` fields must match tenant schema
- `additional_metadata` is stored but NOT filterable

**Response:**
```json
{
  "status": "success",
  "message": "Processing started",
  "source_ids": ["doc_001", "doc_002"]
}
```

#### POST /ingestion/verify_processing

Poll indexing status until completed.

**Request:**
```json
{
  "tenant_id": "acme_corp",
  "file_ids": ["doc_001", "doc_002"]
}
```

**Response:**
```json
{
  "results": [
    {
      "file_id": "doc_001",
      "indexing_status": "completed",
      "chunks_created": 47
    },
    {
      "file_id": "doc_002",
      "indexing_status": "processing",
      "chunks_created": 0
    }
  ]
}
```

**Status values:**
- `queued` — Waiting to process
- `processing` — Currently parsing/chunking/embedding
- `graph_creation` — Vector indexing done, graph creation still running
- `completed` — Ready for recall
- `success` — Alias for completed
- `errored` — Processing failed

**Typical indexing time:** 30-90 seconds per document

### Memories

#### POST /memories/add_memory

Store user preferences, conversation history, behavioral traits.

**Request:**
```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "user_alice",
  "memories": [
    {
      "text": "Prefers dark mode",
      "infer": true,
      "user_name": "Alice",
      "metadata": { "category": "preference", "timestamp": "2024-03-15" }
    },
    {
      "user_assistant_pairs": [
        {
          "user": "What's my coffee order?",
          "assistant": "Large oat milk latte with one shot of vanilla"
        }
      ],
      "infer": true,
      "upsert": true
    }
  ]
}
```

**Memory formats:**

1. **Text-based:** `{ "text": "...", "infer": true/false }`
2. **Conversation-based:** `{ "user_assistant_pairs": [...], "infer": true/false }`

**Parameters:**
- `infer: true` — Extract implicit preferences from raw input (e.g., "I like dark mode" → "User prefers dark mode UI setting")
- `infer: false` — Store exactly as written
- `upsert: true` — Update existing memory if similar text found
- `upsert: false` — Always create new memory

**Response:**
```json
{
  "status": "success",
  "message": "Memories added successfully",
  "memory_ids": ["mem_001", "mem_002"]
}
```

#### POST /list/data (kind: "memories")

List all memories for a sub-tenant, with pagination.

**Request:**
```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "user_alice",
  "kind": "memories",
  "page": 1,
  "page_size": 50
}
```

**Response:**
```json
{
  "user_memories": [
    {
      "id": "mem_001",
      "text": "User prefers dark mode UI setting",
      "timestamp": "2024-03-15T10:30:00Z",
      "metadata": { "category": "preference" }
    }
  ],
  "pagination": { "page": 1, "page_size": 50, "total": 1, "total_pages": 1, "has_next": false, "has_previous": false }
}
```

#### DELETE /memories/delete_memory

Delete a single memory by ID. Uses query parameters, not a request body.

**Request:**
```
DELETE /memories/delete_memory?tenant_id=acme_corp&sub_tenant_id=user_alice&memory_id=mem_001
```

**Response:**
```json
{
  "success": true,
  "user_memory_deleted": true
}
```

**Note:** Deletes one memory per call. Loop over IDs to delete multiple.

### Recall

#### POST /recall/full_recall

Retrieve knowledge (documents) with hybrid semantic + graph + keyword.

**Request:**
```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "team_eng",
  "query": "What are the Q4 deployment procedures?",
  "max_results": 10,
  "alpha": 0.8,
  "recency_bias": 0.3,
  "mode": "fast",
  "graph_context": true,
  "metadata_filters": {
    "category": "deployment"
  }
}
```

**Parameters:**

- `query` (required) — Natural language query
- `max_results` (default: 5) — Number of chunks to return
- `alpha` (default: 0.8, range: 0-1 or `"auto"`) — Semantic vs keyword balance (1.0 = pure semantic, 0.0 = pure keyword, `"auto"` = backend chooses)
- `recency_bias` (default: 0.0, range: 0-1) — Boost recent content (0 = no bias, 1.0 = heavy recency)
- `mode` (default: "fast") — `"fast"` for speed, `"thinking"` for graph context + forceful relations
- `graph_context` (default: false) — Include entity relationships in response
- `metadata_filters` — Deterministic filtering by tenant schema fields
- `forceful_relations` — Array of entity names to explicitly include (requires `mode: "thinking"`)

**Response:**
```json
{
  "results": [
    {
      "chunk_uuid": "chunk_001",
      "text": "Q4 deployment requires approval from tech lead and security review...",
      "source_id": "doc_deployment_guide",
      "source_name": "Q4 Deployment Guide.pdf",
      "score": 0.94,
      "metadata": {
        "category": "deployment",
        "timestamp": "2024-10-01"
      },
      "additional_metadata": {
        "author": "alice@acme.com",
        "version": "2.1"
      },
      "graph_context": {
        "entities": ["Tech Lead", "Security Review"],
        "relationships": [
          {
            "source": "Deployment",
            "target": "Security Review",
            "type": "REQUIRES"
          }
        ]
      }
    }
  ]
}
```

#### POST /recall/recall_preferences

Retrieve memories (user state) with personalization.

**Request:**
```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "user_alice",
  "query": "What are my communication preferences?",
  "max_results": 5,
  "mode": "fast"
}
```

**Response:**
```json
{
  "results": [
    {
      "memory_id": "mem_001",
      "text": "User prefers concise email updates over long reports",
      "score": 0.91,
      "timestamp": "2024-03-15T10:30:00Z"
    }
  ]
}
```

**Note:** `recall_preferences` queries vectorstore index `[0]` (memories), while `full_recall` queries index `[1]` (knowledge).

#### POST /recall/boolean_recall

Exact phrase or boolean keyword search.

**Request:**
```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "team_eng",
  "query": "mechanical engineer",
  "operator": "phrase",
  "search_mode": "sources",
  "max_results": 10
}
```

**Operators:**
- `"phrase"` — Exact phrase match
- `"and"` — All terms must appear
- `"or"` — Any term can appear

**`search_mode`:**
- `"sources"` — Search knowledge sources (default)
- `"memories"` — Search memories

**Critical:** Do NOT send `alpha`, `mode`, or `graph_context` to boolean_recall — these parameters are invalid for this endpoint.

**Response:**
```json
{
  "results": [
    {
      "chunk_uuid": "chunk_123",
      "text": "Senior mechanical engineer with 10 years of experience...",
      "source_id": "resume_001",
      "score": 1.0
    }
  ]
}
```

## SDK Reference

### TypeScript SDK

**Installation:**
```bash
npm install @hydradb/sdk
```

**Initialization:**
```typescript
import { HydraDBClient } from '@hydradb/sdk';

const client = new HydraDBClient({
  token: process.env.HYDRA_DB_API_KEY,
  baseUrl: 'https://api.hydradb.com' // optional
});
```

**Tenants:**
```typescript
// Create tenant
await client.tenant.create({
  tenant_id: 'acme_corp',
  tenant_metadata_schema: [
    { name: 'category', data_type: 'VARCHAR', enable_match: true, enable_dense_embedding: false, enable_sparse_embedding: false }
  ]
});

// Poll status
const status = await client.tenant.getInfraStatus({ tenant_id: 'acme_corp' });

// Monitor
const stats = await client.tenant.monitor({ tenant_id: 'acme_corp' });

// Delete
await client.tenant.deleteTenant({ tenant_id: 'acme_corp' });
```

**Ingestion:**
```typescript
// Upload files
await client.upload.knowledge({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'team_eng',
  files: [{ path: './doc.pdf', filename: 'doc.pdf', contentType: 'application/pdf' }],
  file_metadata: JSON.stringify([
    { file_id: 'doc_001', metadata: { category: 'legal' } }
  ]),
  upsert: true,
});

// Upload app sources
await client.upload.knowledge({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'team_eng',
  app_knowledge: JSON.stringify([
    {
      id: 'slack_001',
      title: 'Slack: #engineering',
      content: 'Deployment completed',
      metadata: { category: 'deployment' }
    }
  ]),
  upsert: true,
});

// Verify processing
const status = await client.upload.verifyProcessing({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'team_eng',
  file_ids: ['doc_001']
});
```

**Memories:**
```typescript
// Add memories
await client.upload.addMemory({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'user_alice',
  upsert: true,
  memories: [
    { source_id: 'mem_001', text: 'Prefers dark mode', infer: true }
  ]
});

// List memories
const memories = await client.fetch.listData({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'user_alice',
  kind: 'memories'
});

// Delete memory
await client.upload.deleteMemory({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'user_alice',
  memory_id: 'mem_001'
});
```

**Recall:**
```typescript
// Full recall (knowledge)
const results = await client.recall.fullRecall({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'team_eng',
  query: 'deployment procedures',
  max_results: 10,
  alpha: 0.8,
  recency_bias: 0.3,
  mode: 'fast',
  graph_context: true,
  metadata_filters: { category: 'deployment' }
});

// Preferences recall (memories)
const prefs = await client.recall.recallPreferences({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'user_alice',
  query: 'communication preferences',
  max_results: 5
});

// Boolean recall
const exact = await client.recall.booleanRecall({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'team_eng',
  query: 'mechanical engineer',
  operator: 'phrase',
  search_mode: 'sources',
  max_results: 10
});
```

**Knowledge:**
```typescript
// List knowledge sources
const sources = await client.fetch.listData({
  tenant_id: 'acme_corp',
  sub_tenant_id: 'team_eng',
  kind: 'knowledge',
  page: 1,
  page_size: 50

// Fetch content

// Delete knowledge

```

```python
boolean_results = client.recall.boolean_recall(
```

## Cookbook Patterns

### Glean Clone — Unified Enterprise Search

**Goal:** Search across Slack, Notion, Gmail, Linear, GitHub with cited answers

**Endpoints used:** `POST /ingestion/upload_knowledge`, `POST /recall/full_recall`

**Pattern:**
1. Ingest all app sources into single tenant
2. Use metadata for source type, timestamp, author
3. Recall with high alpha (0.9) for semantic understanding
4. Present results with source citations and links

**Implementation:**
```python
# Ingest from multiple sources
sources = {
    'slack': slack_channels,
    'notion': notion_pages,
    'gmail': emails,
    'linear': issues
}

for source_type, items in sources.items():
    client.upload.knowledge(
        tenant_id='enterprise',
        app_knowledge=[{
            'source_id': f'{source_type}_{item.id}',
            'source_name': item.title,
            'knowledge_items': [{
                'text': item.content,
                'metadata': {
                    'source_type': source_type,
                    'timestamp': item.created_at,
                    'author': item.author
                },
                'additional_metadata': {
                    'url': item.url
                }
            }]
        } for item in items]
    )

# Search across all sources
results = client.recall.full_recall(
    tenant_id='enterprise',
    query='Q4 OKRs',
    max_results=10,
    alpha=0.9,
    graph_context=True
)

# Format with citations
for result in results:
    print(f"[{result['metadata']['source_type']}] {result['source_name']}")
    print(f"{result['text']}")
    print(f"Link: {result['additional_metadata']['url']}\n")
```

### AI LinkedIn Recruiter

**Goal:** Search candidates by skills, experience, cultural fit

**Endpoints used:** `POST /ingestion/upload_knowledge`, `POST /recall/full_recall`

**Pattern:**
1. Ingest resumes and profiles as knowledge
2. Use metadata for skills, years_experience, location
3. Recall with moderate alpha (0.7) for balanced semantic + keyword
4. Filter by required skills in metadata_filters

**Implementation:**
```python
# Ingest candidate profiles

    client.upload.knowledge(
        tenant_id='recruiting',

            'source_id': f'candidate_{candidate.id}',

                'text': f"{candidate.summary}\n{candidate.experience}",
                'metadata': {
                    'skills': ','.join(candidate.skills),
                    'years_experience': str(candidate.years),
                    'location': candidate.location
                }
            }]
        }]
    )

# Search for senior React engineers in SF
results = client.recall.full_recall(
    tenant_id='recruiting',
    query='senior frontend engineer React TypeScript',
    max_results=20,
    alpha=0.7,
    metadata_filters={
        'location': 'San Francisco'
    }
)

```

### Customer Support Agent

**Goal:** Per-user memory of plan, history, past failures

**Endpoints used:** `POST /memories/add_memory`, `POST /recall/recall_preferences`, `POST /recall/full_recall`

**Pattern:**
1. Store user plan, preferences as memories
2. Store conversation history as memories with inference
3. Ingest knowledge base as shared knowledge
4. Recall both memories and knowledge, prioritize memories

**Implementation:**
```python
# Store user context
client.upload.add_memory(
    tenant_id='support',

    memories=[
        {'text': f'User is on {user.plan} plan'},
        {'text': f'User timezone: {user.timezone}'},
        {'text': 'Past issue: Payment failed on 2024-02-15, resolved with card update'}
    ]
)

# Store conversation with inference
client.upload.add_memory(
    tenant_id='support',
    sub_tenant_id=f'user_{user.id}',
    memories=[{
        'user_assistant_pairs': [{
            'user_message': user_message,
            'assistant_message': assistant_response
        }],
        'infer': True
    }]
)

# Recall user context + knowledge base
user_context = client.recall.recall_preferences(
    tenant_id='support',

    max_results=5

knowledge = client.recall.full_recall(

# Combine and prioritize user context
```

## Common Mistakes Checklist

| Symptom | Cause | Fix |
|---------|-------|-----|
| Empty recall results | Content not yet indexed | Poll `verify_processing` until `completed` |
| Memories missing from `full_recall` | Wrong endpoint for store | Use `recall_preferences` for memories, `full_recall` for knowledge |
| `metadata_filters` ignored | Field not declared with `enable_match: true` | Declare in `tenant_metadata_schema` before ingestion |
| `graph_context` empty | Not set to `true` | Set `graph_context: true` on recall request |
| Recency doesn't matter | `recency_bias` defaults to 0 | Set explicitly between 0.1 and 1.0 |
| Wrong data surfaces | Wrong `sub_tenant_id` on recall | Match the scope used at write time |
| Forceful relations not fetching | Not in `thinking` mode | Set `mode: "thinking"` |
| 422 on upload | `file_metadata` array length mismatch | Match array length to files count |
| Schema field can't be changed | Tenant metadata schema is immutable | Plan before creation, use `additional_metadata` for flexible fields |
| 422 on app_knowledge | Sent as JSON object instead of JSON-encoded string | Stringify `app_knowledge` before sending in multipart form |
| boolean_recall returning 400 | Sent `alpha`, `mode`, or `graph_context` params | Drop those params — boolean_recall doesn't accept them |
| Tenant creation failure on re-create | Previous deletion still in progress | Use a different `tenant_id` or wait for cleanup |
| Data not scoped correctly | `sub_tenant_id` omitted on per-user data | Always pass `sub_tenant_id` for user-specific writes and reads |
| Inference not running | `infer` not set to `true` | Set `infer: true` explicitly for conversational/behavioral input |
| Partial deletion unnoticed | Assumed `success: true` means all items deleted | Always inspect `results[].deleted` per item |

## Troubleshooting Guide

### Authentication Failures

**Symptom:** 401 Unauthorized

**Check:**
1. Header format: `Authorization: Bearer <key>` with space after "Bearer"
2. Key validity in dashboard at app.hydradb.com
3. Key permissions (read vs read-write)

**Fix:**
```bash
# Correct header format
curl -H "Authorization: Bearer $API_KEY" https://api.hydradb.com/tenants/monitor?tenant_id=test
```

### Tenant Not Found After Creation

**Symptom:** 404 on ingestion/recall immediately after tenant creation

**Cause:** Tenant creation is async

**Fix:**
```python
# Create tenant
client.tenant.create(tenant_id='acme_corp')

# Poll until ready
import time
while True:
    status = client.tenant.get_infra_status(tenant_id='acme_corp')
    if (status['graph_status'] and 
        status['vectorstore_status'] == [True, True]):
        break
    time.sleep(2)

# Now safe to ingest
client.upload.knowledge(...)
```

### 422 on Ingestion

**Common causes:**

1. **file_metadata array mismatch:**
```python
# WRONG: 2 files, 1 metadata item
files = [file1, file2]
file_metadata = [{'file_id': 'doc_001'}]

# CORRECT: 2 files, 2 metadata items
files = [file1, file2]
file_metadata = [
    {'file_id': 'doc_001'},
    {'file_id': 'doc_002'}
]
```

2. **app_knowledge not stringified:**
```python
# WRONG: Sent as JSON object
app_knowledge = [{'source_id': 'slack_001', ...}]

# CORRECT: Stringify before sending
app_knowledge = json.dumps([{'source_id': 'slack_001', ...}])
```

3. **MemoryItem with no content:**
```python
# WRONG: No text and no user_assistant_pairs
{'infer': True}

# CORRECT: At least one content field
{'text': 'User prefers dark mode', 'infer': True}
```

### Empty Recall Results

**Diagnostic checklist:**

1. **Content still processing:**
```python
status = client.upload.verify_processing(
    tenant_id='acme_corp',
    file_ids=['doc_001']
)
# Wait until indexing_status == 'completed'
```

2. **Wrong endpoint:**
```python
# WRONG: Using full_recall for memories
client.recall.full_recall(sub_tenant_id='user_alice', query='preferences')

# CORRECT: Use recall_preferences for memories
client.recall.recall_preferences(sub_tenant_id='user_alice', query='preferences')
```

3. **Wrong sub_tenant_id:**
```python
# Write with sub_tenant_id
client.upload.knowledge(sub_tenant_id='team_eng', ...)

# Read with SAME sub_tenant_id
client.recall.full_recall(sub_tenant_id='team_eng', ...)
```

4. **Too restrictive filters:**
```python
# Try without filters first
results = client.recall.full_recall(
    tenant_id='acme_corp',
    query='deployment'
    # No metadata_filters
)
```

### Rate Limiting

**Symptoms:**
- 429 status code
- Throttling errors

**Solutions:**

1. **Use bulk endpoints:**
```python
# INEFFICIENT: Loop with individual calls
for memory in memories:
    client.upload.add_memory(memories=[memory])

# EFFICIENT: Single bulk call
client.upload.add_memory(memories=memories)
```

2. **Implement exponential backoff:**
```python
import time
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(
    stop=stop_after_attempt(5),
    wait=wait_exponential(multiplier=1, min=2, max=60)
)
def recall_with_retry():
    return client.recall.full_recall(...)
```

3. **Cache repeated patterns:**
```python
from functools import lru_cache

@lru_cache(maxsize=100)
def cached_recall(query):
    return client.recall.full_recall(query=query)
```

## Multi-Tenant Design Patterns

### B2B SaaS Pattern

**Configuration:**
- One `tenant_id` per customer company
- `sub_tenant_id` for teams, workspaces, or users within customer
- Shared knowledge at tenant level (no sub_tenant_id)
- User-specific memories at sub-tenant level

**Example:**
```python
# Customer Acme Corp
tenant_id = 'acme_corp'

# Shared company knowledge (no sub_tenant_id)
client.upload.knowledge(
    tenant_id='acme_corp',
    # sub_tenant_id omitted
    files=[company_handbook]
)

# Team-specific data
client.upload.knowledge(
    tenant_id='acme_corp',
    sub_tenant_id='team_engineering',
    files=[tech_docs]
)

# User-specific memories
client.upload.add_memory(
    tenant_id='acme_corp',
    sub_tenant_id='user_alice',
    memories=[{'text': 'Prefers TypeScript'}]
)
```

### B2C App Pattern

**Configuration:**
- One `tenant_id` for entire app
- `sub_tenant_id` for each end user
- Shared knowledge at tenant level (FAQs, docs)
- All user data at sub-tenant level

**Example:**
```python
# Single tenant for app
tenant_id = 'my_app'

# Shared knowledge for all users
client.upload.knowledge(
    tenant_id='my_app',
    app_knowledge=[{
        'source_id': 'faq',
        'source_name': 'FAQ',
        'knowledge_items': [{'text': faq_content}]
    }]
)

# User-specific memories
for user in users:
    client.upload.add_memory(
        tenant_id='my_app',
        sub_tenant_id=f'user_{user.id}',
        memories=[
            {'text': f'User name: {user.name}'},
            {'text': f'User preferences: {user.preferences}'}
        ]
    )
```

### Internal Tools Pattern

**Configuration:**
- One `tenant_id` per company
- `sub_tenant_id` for departments or projects
- Use metadata for finer-grained access control

**Example:**
```python
# Company tenant
tenant_id = 'internal'

# Department-specific data
client.upload.knowledge(
    tenant_id='internal',
    sub_tenant_id='dept_engineering',
    files=[tech_specs]
)

client.upload.knowledge(
    tenant_id='internal',
    sub_tenant_id='dept_finance',
    files=[financial_reports]
)

# Query scoped to department
results = client.recall.full_recall(
    tenant_id='internal',
    sub_tenant_id='dept_engineering',
    query='Q4 roadmap'
)
```

### Environment Separation

**CRITICAL: Never share tenant_id across prod/staging/dev**

**Example:**
```python
# WRONG: Same tenant for all environments
tenant_id = 'my_app'  # Used in prod, staging, dev

# CORRECT: Separate tenants per environment
prod_tenant = 'my_app_prod'
staging_tenant = 'my_app_staging'
dev_tenant = 'my_app_dev'
```

### Scoping Hierarchy

**Order of operations:**
1. `tenant_id` — Primary isolation boundary (customers, environments)
2. `sub_tenant_id` — Logical partition inside tenant (users, teams, workspaces)
3. `metadata` — Deterministic filters inside partition (category, project, department)

**Example:**
```python
# Narrow from tenant → sub-tenant → metadata
results = client.recall.full_recall(
    tenant_id='acme_corp',           # Customer
    sub_tenant_id='team_mobile',     # Team
    query='sprint status',
    metadata_filters={                # Project
        'project': 'phoenix'
    }
)
```

```python
        }]
```

### Custom Chunking

**Use case:** Need control over chunk boundaries (e.g., section-level chunking)

**Endpoints used:** `POST /ingestion/upload_knowledge` (with pre-chunked content)

**Pattern:**
```python
# Pre-chunk before ingestion
def custom_chunk(document):
    # Your chunking logic
    return [
        {'text': section1, 'metadata': {'section': '1'}},
        {'text': section2, 'metadata': {'section': '2'}}
    ]

chunks = custom_chunk(document)

# Ingest pre-chunked content
client.upload.knowledge(

```

### Multi-Index Retrieval

**Use case:** Need to query both memories and knowledge, merge results

**Endpoints used:** `POST /recall/recall_preferences`, `POST /recall/full_recall`

**Pattern:**
```python
# Query both indices in parallel
import asyncio

async def multi_index_recall(tenant_id, sub_tenant_id, query):
    memories = await client.recall.recall_preferences(
        tenant_id=tenant_id,
        sub_tenant_id=sub_tenant_id,
        query=query,
        max_results=5
    )
    
    knowledge = await client.recall.full_recall(
        tenant_id=tenant_id,
        query=query,
        max_results=10
    )
    
    # Merge and dedupe
    combined = memories['results'] + knowledge['results']
    return combined

results = asyncio.run(multi_index_recall('acme', 'user_alice', 'preferences'))
```

### Batch Processing

**Use case:** High-volume ingestion from data pipeline

**Endpoints used:** `POST /ingestion/upload_knowledge`, `POST /ingestion/verify_processing`

**Pattern:**
```python
# Batch files
batch_size = 50
for i in range(0, len(files), batch_size):
    batch = files[i:i+batch_size]
    
    client.upload.knowledge(
        tenant_id='batch_ingest',
        files=batch,
        file_metadata=[
            {'file_id': f'doc_{j}', 'metadata': {'batch': str(i)}}
            for j in range(len(batch))
        ]
    )
    
    # Verify batch completed before next
    file_ids = [f'doc_{j}' for j in range(i, i+len(batch))]
    wait_for_completion(tenant_id='batch_ingest', file_ids=file_ids)
```

### Temporal Reasoning

**Use case:** Time-aware search (recent vs historical)

**Endpoints used:** `POST /tenants/create` (with timestamp schema), `POST /recall/full_recall`

**Pattern:**
```python
# Add timestamp to metadata schema
client.tenant.create(
    tenant_id='temporal',
    tenant_metadata_schema={
        'timestamp': {
            'enabled': True,
            'type': 'str',
            'enable_match': True
        }
    }

# Query recent content
recent = client.recall.full_recall(
    tenant_id='temporal',
    query='incidents',
    recency_bias=0.8,
    max_results=10

historical = client.recall.full_recall(
    tenant_id='temporal',
    query='incidents',
    recency_bias=0.0,

        'timestamp': '2023-01-01'  # Before this date
    }

```

### Hybrid Reranking

**Use case:** Combine HydraDB recall with external reranker (Cohere, Jina)

**Endpoints used:** `POST /recall/full_recall` + external reranker API

**Pattern:**
```python
# Get more candidates from HydraDB
candidates = client.recall.full_recall(
    tenant_id='rerank',
    query='deployment procedures',
    max_results=50
)

# Rerank with Cohere
import cohere
co = cohere.Client(api_key=cohere_key)

reranked = co.rerank(
    query='deployment procedures',
    documents=[c['text'] for c in candidates['results']],
    top_n=10
)

# Map back to original results
final = [candidates['results'][r.index] for r in reranked.results]
```

## Performance Guidelines

### Latency Benchmarks

- **Recall (fast mode):** p50 ~80ms, p95 ~180ms, p99 ~250ms
- **Recall (thinking mode):** p50 ~150ms, p95 ~300ms, p99 ~400ms
- **Ingestion processing:** 30-90 seconds per document (typical)
- **Tenant creation:** 10-30 seconds
- **Memory add:** p50 ~50ms, p95 ~120ms

### Scaling Guidelines

**Per tenant:**
- **Documents:** Tested up to 100K documents per tenant
- **Chunks:** Tested up to 5M chunks per tenant
- **Memories:** Tested up to 500K memories per sub-tenant
- **Sub-tenants:** No hard limit, tested with 10K+ sub-tenants

**Rate limits:**
- **Recall:** 1000 requests/minute per tenant
- **Ingestion:** 100 files/minute per tenant
- **Memories:** 500 adds/minute per sub-tenant

### Caching Strategies

**When to cache:**
- Repeated queries (e.g., FAQ, common patterns)
- User preference lookups
- Metadata-heavy filters

**TTL recommendations:**
- **User preferences:** 5-15 minutes
- **Knowledge search:** 1-5 minutes
- **Real-time data:** No cache

**Example:**
```python
from functools import lru_cache
import hashlib

@lru_cache(maxsize=1000)
def cached_recall(tenant_id, query, metadata_str):
    return client.recall.full_recall(
        tenant_id=tenant_id,
        query=query,
        metadata_filters=json.loads(metadata_str)
    )

# Use
results = cached_recall(
    'acme',
    'deployment',
    json.dumps({'category': 'ops'})
)
```

## Security & Compliance

### Tenant Isolation

- **API-level enforcement:** No cross-tenant data access at API layer
- **Database isolation:** Separate graph and vector indices per tenant
- **Sub-tenant isolation:** Enforced via query filters, not separate infrastructure

### Data Encryption

- **At rest:** AES-256 encryption for all stored data
- **In transit:** TLS 1.3 for all API communication
- **Keys:** Managed by HydraDB infrastructure

### Audit Logging

**Available events:**
- Tenant creation/deletion
- Data ingestion (files uploaded, sources added)
- Recall queries (query text, results count)
- Memory operations (add, delete)
- Authentication events

**Log retention:** 90 days

### Compliance

- **SOC 2 Type II:** In progress
- **GDPR:** Compliant, supports right to deletion
- **Data residency:** US-only currently, EU region coming Q3 2024
