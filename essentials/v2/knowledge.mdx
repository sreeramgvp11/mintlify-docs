## 1. What it is

Knowledge is the shared, tenant-wide context that every user and agent in a tenant can search  -  product documents, internal wikis, policy PDFs, Slack threads, Notion pages, CSVs, emails, and anything else that ought to be reusable across the workspace.

Knowledge is the counterpart to [Memories](/essentials/v2/memories), which holds per-user context. Both are retrievable through the same [`POST /search`](/api-reference/v2/endpoint/search) endpoint, but they live in separate stores; you pick which one to search via the `source` parameter (see [Search](/essentials/v2/search)).

| | Knowledge | Memories |
|---|---|---|
| **Content** | Documents, files, app-generated content | User preferences, conversations, inferred traits |
| **Scope** | Shared across all users in a tenant | Per-user, scoped by `sub_tenant_id` |
| **Mutability** | Versioned  -  replaced or deleted explicitly | Dynamic  -  evolves with every interaction |
| **Search via** | [`POST /search`](/api-reference/v2/endpoint/search) with `source: "knowledge"` | [`POST /search`](/api-reference/v2/endpoint/search) with `source: "memories"` |
| **`vectorstore_status`** | `vectorstore_status.knowledge` | `vectorstore_status.memories` |

<Info>
  **One endpoint, two stores.** Knowledge and memories are stored separately, but a single [`POST /search`](/api-reference/v2/endpoint/search) call can hit both via `source: "all"`  -  results are merged and re-ranked together. Pick `source: "knowledge"` for shared-doc RAG, `source: "memories"` for per-user context, or `source: "all"` for personalized answers grounded in both. See [Search](/essentials/v2/search) for the full picture.
</Info>

## 2. What it does

When you call [`POST /source/ingest`](/api-reference/v2/endpoint/ingest-content) with `type=knowledge`, HydraDB walks each source through a pipeline:

1. **Parses** the input  -  PDF, DOCX, Markdown, CSV, plain text, or app-source JSON.
2. **Chunks** the content into semantically coherent segments.
3. **Extracts entities and relationships**, building graph nodes and edges (see [Context Graphs](/essentials/v2/context-graphs)).
4. **Embeds** every chunk into the dense and sparse vector stores.
5. **Indexes** the result so it becomes searchable via [`POST /search`](/api-reference/v2/endpoint/search).

The pipeline runs **asynchronously**. The ingest call returns `202 Accepted` with `source_id`s; the content isn't searchable until `indexing_status` reaches at least `graph_creation`. Poll [`GET /source/status`](/api-reference/v2/endpoint/source-status) to track progress  -  see the full status table at [Source Status](/api-reference/v2/endpoint/source-status).

## 3. When to use it

Reach for Knowledge ingestion when the content is:

- **Shared across all users** in your tenant  -  product docs, policy PDFs, runbooks, wikis, internal tooling.
- **Static or infrequently updated**  -  versioned by replacing the source rather than appending.
- **Document-structured** rather than conversational  -  files, threads, pages, articles.

Reach for [Memories](/essentials/v2/memories) instead when the content is:

- User preferences or behavioral signals tied to one person.
- Per-user conversation history that should shape future answers for that user.
- Anything that should personalize a single user's experience.

| If the content is… | Store it as… |
|---|---|
| Shared across all users | Knowledge |
| Specific to one user or session | [Memory](/essentials/v2/memories) |

When in doubt, ask "would a different user benefit from seeing this?"  -  if yes, Knowledge; if no, Memory.

## 4. Two ingestion paths

[`POST /source/ingest`](/api-reference/v2/endpoint/ingest-content) with `type=knowledge` accepts two content formats in the same request  -  you can send either, or both at once. Pick the path that matches what you have on hand:

- **Path A** when you have **binary files** HydraDB should parse for you.
- **Path B** when you have **pre-extracted text** from an app or connector and want to skip parsing.

### Path A  -  Files

Binary files HydraDB should parse and chunk: PDF, DOCX, Markdown, CSV, TXT.

<CodeGroup>
```python Python SDK
import json

with open("/path/to/policy.pdf", "rb") as f1, open("/path/to/runbook.pdf", "rb") as f2:
    result = client.source.ingest(
        type="knowledge",
        tenant_id="acme_corp",
        files=[
            ("policy.pdf", f1, "application/pdf"),
            ("runbook.pdf", f2, "application/pdf"),
        ],
        file_metadata=json.dumps([
            {"source_id": "policy_v2", "tenant_metadata": {"document_type": "policy", "status": "approved"}},
            {"source_id": "runbook_deploy", "tenant_metadata": {"document_type": "runbook"}},
        ]),
    )
```
```typescript TypeScript SDK
const result = await client.source.ingest({
  type: "knowledge",
  tenant_id: "acme_corp",
  files: [
    { path: "/path/to/policy.pdf", filename: "policy.pdf", contentType: "application/pdf" },
    { path: "/path/to/runbook.pdf", filename: "runbook.pdf", contentType: "application/pdf" },
  ],
  file_metadata: JSON.stringify([
    { source_id: "policy_v2", tenant_metadata: { document_type: "policy", status: "approved" } },
    { source_id: "runbook_deploy", tenant_metadata: { document_type: "runbook" } },
  ]),
});
```
```bash cURL
curl -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F "files=@/path/to/policy.pdf" \
  -F "files=@/path/to/runbook.pdf" \
  -F 'file_metadata=[
    { "source_id": "policy_v2", "tenant_metadata": { "document_type": "policy", "status": "approved" } },
    { "source_id": "runbook_deploy", "tenant_metadata": { "document_type": "runbook" } }
  ]'
```
</CodeGroup>

### Path B  -  App sources

Pre-parsed text from connected apps. You supply the text, [metadata](/essentials/v2/metadata), and an optional URL. Use this when your application already extracts content (Slack messages, Notion pages, Gmail threads, webpages) and you want HydraDB to skip the parsing step.

<CodeGroup>
```python Python SDK
import json

result = client.source.ingest(
    type="knowledge",
    tenant_id="acme_corp",
    app_knowledge=json.dumps([
        {
            "tenant_id": "acme_corp",
            "sub_tenant_id": "default",
            "id": "slack_thread_001",
            "title": "Pricing discussion - Slack #product",
            "type": "slack",
            "content": {"text": "We agreed on three tiers: Starter at $29, Pro at $79, Enterprise at $199."},
            "tenant_metadata": {"channel": "product", "status": "finalized"},
            "document_metadata": {"slack_ts": "1715012345.000100"},
        }
    ]),
)
```
```typescript TypeScript SDK
const result = await client.source.ingest({
  type: "knowledge",
  tenant_id: "acme_corp",
  app_knowledge: JSON.stringify([
    {
      tenant_id: "acme_corp",
      sub_tenant_id: "default",
      id: "slack_thread_001",
      title: "Pricing discussion - Slack #product",
      type: "slack",
      content: { text: "We agreed on three tiers: Starter at $29, Pro at $79, Enterprise at $199." },
      tenant_metadata: { channel: "product", status: "finalized" },
      document_metadata: { slack_ts: "1715012345.000100" },
    },
  ]),
});
```
```bash cURL
curl -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "tenant_id": "acme_corp",
      "sub_tenant_id": "default",
      "id": "slack_thread_001",
      "title": "Pricing discussion - Slack #product",
      "type": "slack",
      "content": { "text": "We agreed on three tiers: Starter at $29, Pro at $79, Enterprise at $199." },
      "tenant_metadata": { "channel": "product", "status": "finalized" },
      "document_metadata": { "slack_ts": "1715012345.000100" }
    }
  ]'
```
</CodeGroup>

<Info>
Both paths are available. We *recommend* storing the content as a markdown file and using Path A (file upload via `POST /source/ingest`) for better parsing and chunking.
</Info>

**Response (both paths):**

```json
{
  "success": true,
  "data": {
    "success": true,
    "message": "Knowledge uploaded successfully",
    "results": [
      { "source_id": "policy_v2", "filename": "policy.pdf", "status": "queued", "error": null }
    ],
    "success_count": 1,
    "failed_count": 0
  },
  "error": null,
  "meta": {
    "request_id": "9d13aef4-02f4-4e73-8c62-4c2601d04f9d",
    "latency_ms": 12.3
  }
}
```

All endpoints return this envelope shape. Access the payload via `response.data` (e.g. `response.data.results`), not at the top level.

## 5. Verifying processing

Ingestion is async, so you need to confirm each source actually finished before it'll show up in [search](/essentials/v2/search). Poll [`GET /source/status`](/api-reference/v2/endpoint/source-status) with the `source_id`s you got back from ingestion. The snippets below loop until every item reaches a terminal state (`completed` or `errored`):

<CodeGroup>
```python Python SDK
import time

source_ids = ["policy_v2", "runbook_deploy"]
terminal = {"completed", "errored"}

while True:
    status = client.source.status(
        tenant_id="acme_corp",
        sub_tenant_id="default",
        source_ids=source_ids,
    )
    if all(s.indexing_status in terminal for s in status.statuses):
        break
    time.sleep(5)
```
```typescript TypeScript SDK
const sourceIds = ["policy_v2", "runbook_deploy"];
const terminal = new Set(["completed", "errored"]);

let status;
while (true) {
  status = await client.source.status({
    tenant_id: "acme_corp",
    sub_tenant_id: "default",
    source_ids: sourceIds,
  });

  const allDone = status.statuses.every((s) => terminal.has(s.indexing_status));
  if (allDone) break;

  await new Promise((r) => setTimeout(r, 5_000));
}
```
```bash cURL
while true; do
  response=$(curl -s -G 'https://api.hydradb.com/source/status' \
    -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
    -H "API-Version: 2" \
    --data-urlencode "tenant_id=acme_corp" \
    --data-urlencode "sub_tenant_id=default" \
    --data-urlencode "source_ids=policy_v2" \
    --data-urlencode "source_ids=runbook_deploy")

  statuses=$(echo "$response" | jq -r '.data.statuses[].indexing_status')
  if echo "$statuses" | grep -qv '^\(completed\|errored\)$'; then
    sleep 5
    continue
  fi
  echo "$response" | jq
  break
done
```
</CodeGroup>

```json
{
  "statuses": [
    {
      "source_id": "policy_v2",
      "indexing_status": "completed",
      "success": true
    }
  ]
}
```

Status values move forward through `queued` → `processing` → `graph_creation` → `completed` (or land in `errored` on failure). Poll every few seconds; most documents complete in 1–5 minutes. The full table  -  including the `success` alias that occasionally appears for `completed`  -  is on [Source Status](/api-reference/v2/endpoint/source-status).

<Info>
**`graph_creation` is already searchable.** Items in this state are retrievable via [`POST /search`](/api-reference/v2/endpoint/search). Wait for `completed` only when you specifically need full graph traversal (`graph_context: true`).
</Info>

## 6. Key parameters

The full schema lives on [`POST /source/ingest`](/api-reference/v2/endpoint/ingest-content). Below are the fields you'll touch most often, grouped by where they live in the request.

### File metadata (`file_metadata` array items)

| Field | Description |
|---|---|
| <Field name="source_id" type="string" /> | Optional stable identifier. Used as upsert key when `upsert: true`. |
| <Field name="tenant_metadata" type="object" /> | Filterable fields. Keys must match the tenant metadata schema. Also accepted as `metadata`. |
| <Field name="document_metadata" type="object" /> | Free-form fields for display and bookkeeping. Filterable only when nested under `document_metadata` in `metadata_filters`. Also accepted as `additional_metadata`. |
| <Field name="title" type="string" /> | Override the source title shown in search results. |
| <Field name="type" type="string" /> | Override the source type shown in search results. |
| <Field name="url" type="string" /> | Override the canonical URL. |
| <Field name="timestamp" type="ISO-8601 datetime" /> | Override the source timestamp. |
| <Field name="relations" type="object" /> | Forcefully connect this source to others with `{ "source_ids": [...] }`. |

### App source model (`app_knowledge` array items)

| Field | Description |
|---|---|
| <Field name="id" type="string" recommended /> | Unique identifier for the source. Upsert key. |
| <Field name="title" type="string" recommended /> | Short title or subject. |
| <Field name="type" type="string" recommended /> | Source category, such as `slack`, `notion`, `gmail`, or `webpage`. |
| <Field name="description" type="string" /> | Optional long-form description. |
| <Field name="url" type="string" /> | Canonical URL or reference link. |
| <Field name="timestamp" type="ISO-8601 datetime" /> | Creation or last-updated time. |
| <Field name="content" type="object" required /> | Content payload. Use `{ "text": "..." }` for plain text or `{ "markdown": "..." }` for Markdown. |
| <Field name="tenant_metadata" type="object" /> | Filterable fields matching the tenant schema. Also accepted as `metadata`. |
| <Field name="document_metadata" type="object" /> | Free-form fields. Filterable only when nested under `document_metadata` in `metadata_filters`. Also accepted as `additional_metadata`. |
| <Field name="attachments" type="array" /> | Optional related attachments. |
| <Field name="relations" type="object" /> | Forcefully connect this source to others. |

### Request-level fields

| Field | Default | Description |
|---|---|---|
| <Field name="tenant_id" type="string" required /> |  -  | Target tenant. |
| <Field name="sub_tenant_id" type="string" /> | none | Scope this knowledge to a sub-tenant. If omitted, knowledge is available to all sub-tenants. |
| <Field name="upsert" type="boolean" /> | `true` | Replace existing items with the same `source_id`. |

## 7 - ForceFul relations

Forceful relations let you **declare** links between Knowledge sources at ingestion time. When a query matches a source that has forceful relations attached, HydraDB pulls the linked sources into the `additional_context` field of the search response alongside the primary chunks.

This is separate from the entity and relationship [context graph](/essentials/v2/context-graphs) HydraDB builds automatically. Graph extraction is *derived* from content; forceful relations are *declared* by you  -  useful when you know two documents belong together (a contract and its addendum, a runbook and its troubleshooting guide) but the text doesn't make the connection explicit.

Set the `relations` field on any item in `file_metadata` or `app_knowledge`:

```json
{
  "relations": {
    "source_ids": ["source-id-1", "source-id-2"]
  }
}
```

<CodeGroup>
```python Python SDK
client.source.ingest(
    type="knowledge",
    tenant_id="acme_corp",
    app_knowledge=json.dumps([
        {
            "tenant_id": "acme_corp",
            "sub_tenant_id": "default",
            "id": "auth-guide",
            "title": "Authentication Guide",
            "content": {"text": "..."},
            "relations": {"source_ids": ["auth-troubleshooting", "auth-faq"]},
        }
    ]),
)
```
```typescript TypeScript SDK
await client.source.ingest({
  type: "knowledge",
  tenant_id: "acme_corp",
  app_knowledge: JSON.stringify([
    {
      id: "auth-guide",
      title: "Authentication Guide",
      content: { text: "..." },
      relations: { source_ids: ["auth-troubleshooting", "auth-faq"] },
    },
  ]),
});
```
```bash cURL
curl -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "id": "auth-guide",
      "title": "Authentication Guide",
      "content": { "text": "..." },
      "relations": { "source_ids": ["auth-troubleshooting", "auth-faq"] }
    }
  ]'
```
</CodeGroup>

At search time, linked sources are returned in the `additional_context` field of the [search response](/api-reference/v2/endpoint/search). `search_forceful_relations` defaults to `true`  -  you only need to set it explicitly if you want to disable forceful-relation expansion.

<Warning>
**Knowledge sources only.** Forceful relations work within the same store. Pointing a Knowledge source at a Memory source ID (or vice versa) will silently return nothing at search time.
</Warning>

<Warning>
**`thinking` mode only.** Forceful-relation context is fetched only when `mode: "thinking"` is set on the search request. In `fast` mode, `search_forceful_relations` is ignored even if set to `true`.
</Warning>

## 8. Metadata on knowledge

[Metadata](/essentials/v2/metadata) is how you bridge structured filters and semantic search. Attach `tenant_metadata` at ingestion to enable deterministic narrowing at search time:

```json
{
  "tenant_metadata": { "document_type": "runbook", "environment": "production" }
}
```

Then at search time:

```json
{
  "query": "How do I rotate API keys?",
  "metadata_filters": { "document_type": "runbook", "environment": "production" }
}
```

`tenant_metadata` keys must be defined in the tenant metadata schema with `enable_match: true` to be filterable  -  undeclared keys are silently ignored at search time. `document_metadata` is stored alongside the source for display and bookkeeping; to filter on it, nest the keys under `document_metadata` in `metadata_filters` (e.g. `{"metadata_filters": {"document_metadata": {"author": "Alice"}}}`).

See [Metadata](/essentials/v2/metadata) for schema design, filter examples, and the `tenant_metadata` vs `document_metadata` decision rule. See [Create Tenant](/api-reference/v2/endpoint/create-tenant) for the schema field configuration (`enable_match`, `enable_dense_embedding`, `enable_sparse_embedding`).

## 9. Common mistakes

| Mistake | What goes wrong | Fix |
|---|---|---|
| Searching Knowledge with `source: "memories"` | Nothing returned  -  wrong store | Use `source: "knowledge"` (or `"all"`) on [`POST /search`](/api-reference/v2/endpoint/search). |
| Using `document_metadata` for filterable fields | Filters are silently ignored unless nested correctly | Put filterable keys in `tenant_metadata` and declare them in the [tenant schema](/api-reference/v2/endpoint/create-tenant), or filter under `metadata_filters: {"document_metadata": {...}}`  -  see [Metadata](/essentials/v2/metadata). |
| Not polling [`/source/status`](/api-reference/v2/endpoint/source-status) before searching | Chunks aren't indexed yet  -  empty search results | Wait for `indexing_status: completed` (or `graph_creation` for partial availability). |
| Sending `file_metadata` as a JSON object instead of a JSON string | `multipart/form-data` validation fails | Stringify `file_metadata` before sending in the `-F` field. |
| Omitting `tenant_id` | `422` validation error | Required on every call. |
| Pointing forceful relations at memory IDs from a knowledge source | Silently returns nothing at search | Forceful relations work **within the same store**  -  knowledge→knowledge, memory→memory. |
