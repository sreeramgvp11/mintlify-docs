## 1. What it is

Knowledge is shared, tenant-wide context that all users and agents in a tenant can recall - product documents, internal wikis, policy PDFs, Slack threads, Notion pages, CSVs, emails, and any other source content.

Knowledge is the counterpart to [Memories](/essentials/v2/memories). Both are retrievable through the same `POST /search` endpoint, but they live in separate stores and you pick which to search via the `source` parameter:

| | Knowledge | Memories |
|---|---|---|
| **Content** | Documents, files, app-generated content | User preferences, conversations, inferred traits |
| **Scope** | Shared across all users in a tenant | Per-user, scoped by `sub_tenant_id` |
| **Mutability** | Versioned  -  replaced or deleted explicitly | Dynamic  -  evolves with every interaction |
| **Search via** | `POST /search` with `source: "sources"` | `POST /search` with `source: "memories"` |
| **`vectorstore_status`** | Index `1` | Index `0` |

<Info>
  **One endpoint, two stores.** Knowledge and memories are stored separately, but `POST /search` can hit both in one call via `source: "all"`  -  results are merged and re-ranked together. Use `source: "sources"` for shared-doc RAG, `source: "memories"` for per-user context, `source: "all"` for personalized answers grounded in both. See [Recall](/essentials/v2/recall).
</Info>

---

## 2. What it does

When you upload Knowledge, HydraDB:

1. Parses the source (PDF, DOCX, Markdown, CSV, plain text, or app-source JSON).
2. Chunks the content into semantically coherent segments.
3. Runs entity and relationship extraction - building graph nodes and edges.
4. Embeds chunks into the dense and sparse vector stores.
5. Makes the content searchable through [`POST /search`](/api-reference/v2/endpoint/search-overview).

This pipeline runs **asynchronously**. Content is not searchable until `indexing_status` is `completed`. Poll [`GET /source/status`](/api-reference/v2/endpoint/sources-overview) with the returned `source_id`.

---

## 3. When to use it

Use Knowledge ingestion when the content is:

- Shared across all users in your tenant (product docs, policy PDFs, runbooks, wikis)
- Static or infrequently updated (versioned by replacing the source)
- Document-structured rather than conversational (files, threads, pages)

Use [Memories](/essentials/v2/memories) instead for:

- User preferences and behavioral signals
- Per-user conversation history
- Content that should personalize a single user's experience

| If the content is... | Store it as... |
|---|---|
| Shared across all users | Knowledge |
| Specific to one user or session | Memory |

## 4. Two ingestion paths

The `POST /source/ingest` endpoint (`type=knowledge`) accepts two content formats in one request. Send either or both.

### Path A - Files

Binary files HydraDB should parse and chunk: PDF, DOCX, Markdown, CSV, TXT.

<CodeGroup>
```bash cURL
curl -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F "files=@/path/to/policy.pdf" \
  -F "files=@/path/to/runbook.pdf" \
  -F 'file_metadata=[
    { "file_id": "policy_v2", "tenant_metadata": { "document_type": "policy", "status": "approved" } },
    { "file_id": "runbook_deploy", "tenant_metadata": { "document_type": "runbook" } }
  ]'
```
```typescript TypeScript SDK
const result = await client.source.ingest({
  type: "knowledge",
  tenant_id: "acme_corp",
  files: [
    { path: "/path/to/policy.pdf", filename: "policy.pdf", contentType: "application/pdf" },
    { path: "/path/to/runbook.pdf", filename: "runbook.pdf", contentType: "application/pdf" },
  ],
  file_metadata: JSON.stringify([
    { file_id: "policy_v2", tenant_metadata: { document_type: "policy", status: "approved" } },
    { file_id: "runbook_deploy", tenant_metadata: { document_type: "runbook" } },
  ]),
});
```
```python Python SDK
import json

with open("/path/to/policy.pdf", "rb") as f1, open("/path/to/runbook.pdf", "rb") as f2:
    result = client.source.ingest(
        type="knowledge",
        tenant_id="acme_corp",
        files=[
            ("policy.pdf", f1, "application/pdf"),
            ("runbook.pdf", f2, "application/pdf"),
        ],
        file_metadata=json.dumps([
            {"file_id": "policy_v2", "tenant_metadata": {"document_type": "policy", "status": "approved"}},
            {"file_id": "runbook_deploy", "tenant_metadata": {"document_type": "runbook"}},
        ]),
    )
```
</CodeGroup>

### Path B - App sources

Pre-parsed text from connected apps. You supply the text, metadata, and an optional URL.

<CodeGroup>
```bash cURL
curl -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "tenant_id": "acme_corp",
      "sub_tenant_id": "default",
      "id": "slack_thread_001",
      "title": "Pricing discussion - Slack #product",
      "type": "slack",
      "content": { "text": "We agreed on three tiers: Starter at $29, Pro at $79, Enterprise at $199." },
      "tenant_metadata": { "channel": "product", "status": "finalized" },
      "document_metadata": { "slack_ts": "1715012345.000100" }
    }
  ]'
```
```typescript TypeScript SDK

```
```python Python SDK

    type="knowledge",
    tenant_id="acme_corp",
    app_knowledge=json.dumps([
        {
            "tenant_id": "acme_corp",
            "sub_tenant_id": "default",
            "id": "slack_thread_001",
            "title": "Pricing discussion - Slack #product",
            "type": "slack",

```
</CodeGroup>

<Info>
Both paths are available. We *recommend* storing the content as a markdown file and using Path A (file upload via `POST /source/ingest`) for better parsing and chunking.
</Info>

**Response (both paths):**

```json
{
  "success": true,
  "data": {
    "success": true,
    "message": "Knowledge uploaded successfully",
    "results": [
      { "source_id": "policy_v2", "filename": "policy.pdf", "status": "queued", "error": null }
    ],
    "success_count": 1,
    "failed_count": 0
  },
  "error": null,
  "meta": {
    "request_id": "9d13aef4-02f4-4e73-8c62-4c2601d04f9d",
    "latency_ms": 12.3
  }
}
```

All v2 endpoints return this envelope shape. Access the payload via `response.data` (e.g. `response.data.results`), not at the top level.

## 5. Verifying processing

Poll `GET /source/status` with the `source_id`s you got back from ingestion.

<CodeGroup>
```bash cURL
curl -G 'https://api.hydradb.com/source/status' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  --data-urlencode "tenant_id=acme_corp" \
  --data-urlencode "sub_tenant_id=default" \
  --data-urlencode "file_ids=policy_v2" \
  --data-urlencode "file_ids=runbook_deploy"
```
```typescript TypeScript SDK
const status = await client.source.status({
  tenant_id: "acme_corp",
  sub_tenant_id: "default",
  file_ids: ["policy_v2", "runbook_deploy"],
});
```
```python Python SDK
status = client.source.status(
    tenant_id="acme_corp",
    sub_tenant_id="default",
    file_ids=["policy_v2", "runbook_deploy"],
)
```
</CodeGroup>

```json
{
  "statuses": [
    {
      "file_id": "policy_v2",
      "indexing_status": "completed",
      "success": true
    }
  ]
}
```

Status values: `queued` → `processing` → `graph_creation` → `completed` (or `errored`).

Poll every few seconds. Most documents complete in 1–5 minutes. Items in `graph_creation` are already retrievable via `/search`; wait for `completed` only when you specifically need full graph traversal.

## 6. Key parameters

### File metadata (`file_metadata` array items)

| Field | Type | Description |
|---|---|---|
| `file_id` | string | Optional stable identifier. Used as upsert key when `upsert: true`. |
| `metadata` | object | Filterable fields. Keys must match the tenant metadata schema. |
| `additional_metadata` | object | Free-form fields for display and bookkeeping. Not filterable. |

### App source model (`app_knowledge` array items)

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique identifier for the source. Upsert key. |
| `title` | string | Short title or subject. |
| `type` | string | Source category (e.g., `slack`, `notion`, `gmail`, `webpage`). |
| `description` | string | Optional long-form description. |
| `url` | string | Canonical URL or reference link. |
| `timestamp` | string | ISO-8601 timestamp (creation or last-updated). |
| `content` | object | Content payload. Use `{"text": "..."}` for plain text. |
| `metadata` | object | Filterable fields matching the tenant schema. |
| `additional_metadata` | object | Free-form fields. Not filterable at recall time. |
| `attachments` | array | Optional related attachments. |
| `relations` | object | Forcefully connect this source to others. |

### Request-level fields

| Field | Type | Default | Description |
|---|---|---|---|
| `tenant_id` | string | - *(required)* | Target tenant. |
| `sub_tenant_id` | string | none | Scope this knowledge to a sub-tenant. If omitted, knowledge is available to all sub-tenants. |
| `upsert` | boolean | `true` | Replace existing items with the same `source_id` / `file_id`. |

## 7 - ForceFul relations

Forceful relations let you explicitly link Knowledge sources to each other at ingestion time. When a query matches a source that has forceful relations declared, HydraDB pulls the linked sources into `additional_context` alongside the primary results.

This is separate from the entity and relationship graph HydraDB builds automatically during ingestion. Graph extraction is derived from content; forceful relations are declared by you.

Set the `relations` field on any item in `file_metadata` or `app_knowledge`:

```json
{
  "relations": {
    "cortex_source_ids": ["source-id-1", "source-id-2"]
  }
}
```

<CodeGroup>
```bash cURL
curl -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "id": "auth-guide",
      "title": "Authentication Guide",
      "content": { "text": "..." },
      "relations": { "cortex_source_ids": ["auth-troubleshooting", "auth-faq"] }
    }
  ]'
```
```typescript TypeScript SDK
await client.source.ingest({
  type: "knowledge",
  tenant_id: "acme_corp",
  app_knowledge: JSON.stringify([
    {
      id: "auth-guide",
      title: "Authentication Guide",
      content: { text: "..." },
      relations: { cortex_source_ids: ["auth-troubleshooting", "auth-faq"] },
    },
  ]),
});
```
```python Python SDK
client.source.ingest(
    type="knowledge",
    tenant_id="acme_corp",
    app_knowledge=json.dumps([
        {
            "tenant_id": "acme_corp",
            "sub_tenant_id": "default",
            "id": "auth-guide",
            "title": "Authentication Guide",
            "content": {"text": "..."},
            "relations": {"cortex_source_ids": ["auth-troubleshooting", "auth-faq"]},
        }
    ]),
)
```
</CodeGroup>

At recall, linked sources are returned in the `additional_context` field. Set `search_forceful_relations: true` (the default) on the request to include them.

<Warning>
**Knowledge sources only.** Forceful relations work within the same store. Pointing a Knowledge source at a Memory source ID (or vice versa) will silently return nothing at recall time.
</Warning>

<Warning>
**`thinking` mode only.** Forceful-relation context is fetched only when `mode: "thinking"` is set on the recall request. In `fast` mode, `search_forceful_relations` is ignored even if set to `true`.
</Warning>

## 8. Metadata on knowledge

Attach `metadata` at ingestion to enable deterministic filtering at recall time:

```json
{
  "metadata": { "document_type": "runbook", "environment": "production" }
}
```

Then at recall time:

```json
{
  "query": "How do I rotate API keys?",
  "metadata_filters": { "document_type": "runbook", "environment": "production" }
}
```

`metadata` keys must be defined in the tenant metadata schema with `enable_match: true` to be filterable. `additional_metadata` is stored but cannot be used in `metadata_filters`.

See [Essentials → Metadata](/essentials/v2/metadata) for schema design and filtering patterns.

## 9. Common mistakes

| Mistake | What goes wrong | Fix |
|---|---|---|
| Searching Knowledge with `source: "memories"` | Nothing returned  -  wrong store | Use `source: "sources"` (or `"all"`) on `POST /search`. |
| Using `document_metadata` for filterable fields | Filters are silently ignored unless nested correctly | Put filterable keys in `tenant_metadata` and declare them in the tenant schema, or filter under `metadata_filters: {"document_metadata": {...}}`. |
| Not polling `/source/status` before searching | Chunks aren't indexed yet  -  empty search results | Wait for `indexing_status: completed` (or `graph_creation` for partial availability). |
| Sending `file_metadata` as a JSON object instead of a JSON string | `multipart/form-data` validation fails | Stringify `file_metadata` before sending in the `-F` field. |
| Omitting `tenant_id` | `422` validation error | Required on every call. |
