App sources are pre-parsed records from business applications  -  emails, chat messages, tickets, wiki pages, CRM objects, comments, and attachments  -  where you already have the text and structured fields. Ship them through the same [`POST /source/ingest`](/api-reference/v2/endpoint/ingest-content) endpoint as files, using the `app_knowledge` field instead of multipart uploads.

Every app item becomes one [Knowledge](/essentials/v2/knowledge) source under a flat shape. There are no typed `kind`/`provider`/`fields` discriminators. Threads, replies, and comments are modeled as **separate sources** linked together with [`relations.source_ids`](#7-forceful-relations).

<Info>
  App sources land in the [Knowledge](/essentials/v2/knowledge) store and are searchable through [`POST /search`](/api-reference/v2/endpoint/search) with **`search_apps: true`** and `source: "knowledge"` (or `source: "all"` to merge with [Memories](/essentials/v2/memories)).
</Info>

## 1. What it is

| | App source | File upload |
|---|---|---|
| **Content** | Pre-extracted text (Slack message, Notion page, Jira ticket, etc.) | Binary file (PDF, DOCX, MD, CSV, TXT) |
| **Sent in** | `app_knowledge` (stringified JSON array) | `files` (multipart) |
| **HydraDB does** | Skips parsing; chunks the text you provide | Parses, then chunks |
| **Best for** | App connectors with their own extractors | Documents in their original binary format |

Both paths run through the same pipeline after parsing  -  chunk â†’ embed â†’ graph â†’ index  -  and both return a `source_id` you poll on [`GET /source/status`](/api-reference/v2/endpoint/source-status).

## 2. The mental model

Every app-source item maps to one HydraDB source with the fields below. The shape is the same regardless of provider  -  use `type` as a free-form category tag, not a discriminator.

| Field | Purpose |
|---|---|
| `id` | Stable HydraDB source ID. Upsert key. |
| `title` | Short display title (subject, channel + timestamp, page name). |
| `type` | Source category tag (`slack`, `notion`, `gmail`, `jira`, `salesforce`, etc.). Used for display and filtering. |
| `description` | Optional long-form summary. |
| `url` | Canonical link back to the source item. |
| `timestamp` | ISO-8601 creation or last-updated time. |
| `kind` | Type of app object: `email`, `message`, `ticket`, `knowledge_base`, `comment`, or `custom`. |
| `provider` | System the item came from, such as `slack`, `gmail`, `jira`, `notion`, or `salesforce`. |
| `external_id` | Stable provider ID for this exact item. |
| `fields` | Structured content and app-specific fields for the selected `kind`. |
| `tenant_metadata` | Schema-aligned filterable fields (channel, project, space, region). See [Scoping using metadata](/essentials/v2/metadata). |
| `document_metadata` | Free-form display/bookkeeping fields (author, message ts, ticket key, etc.). |
| `attachments` | Pre-extracted files attached to this item. |
| `relations` | Forceful links to other HydraDB sources by `source_ids`. |

Top-level identifiers like `kind`, `provider`, and `external_id` along with the structured `fields` payload are normalized for indexing and graph creation. Put the main searchable item text in `fields` (e.g. `fields.body` or `fields.description`). Top-level generic `content` remains supported for legacy untyped payloads.

## 3. When to use it

Reach for `app_knowledge` when:

- Your app or connector already extracts text (Slack, Notion, Gmail, Confluence, Jira, Salesforce, HubSpot, Zendesk, Intercom).
- You want to preserve provider IDs as the HydraDB `id` so re-ingests upsert cleanly.
- You want chunking driven by structured text rather than file parsing.

Stick with the file-upload path when you have the original binary (PDF/DOCX/MD/CSV) and want HydraDB's parser to handle layout, tables, and OCR. See [Knowledge - Path A](/essentials/v2/knowledge#path-a-files).

## 4. Ingest an app source

Send `app_knowledge` as a **stringified JSON** value on the multipart form. Each entry becomes one source.

<CodeGroup>
```bash cURL
curl -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "id": "slack_C01_1716213600_000100",
      "title": "Auth rollback discussion - #engineering",
      "type": "slack",
      "url": "https://acme.slack.com/archives/C01/p1716213600000100",
      "timestamp": "2026-05-20T10:00:00Z",
      "content": { "text": "Can someone review the auth rollback? Bob, you free to pair?" },
      "tenant_metadata": { "channel": "engineering", "workspace": "acme" },
      "document_metadata": { "slack_ts": "1716213600.000100", "author": "alice" }
    }
  ]'
```
```typescript TypeScript SDK

```json

See [Knowledge - Verifying processing](/essentials/v2/knowledge#5-verifying-processing) for the polling loop.

<Info>
  **Multipart, not JSON.** [`POST /source/ingest`](/api-reference/v2/endpoint/ingest-content) is one of the few endpoints that uses `multipart/form-data`. Stringify `app_knowledge` before sending it in the form field.
</Info>

## 5. Incremental ingestion

Use the same [`POST /source/ingest`](/api-reference/v2/endpoint/ingest-content) call for both initial backfill and live updates. The choice that matters is whether the provider event is the **same source being updated** or a **new source that arrived later**.

| Provider event | What to send | Why |
|---|---|---|
| Ticket title/description/status changed | Re-ingest with the same `id`, `upsert: true` | The ticket itself changed; replace the snapshot |
| New ticket comment (Linear/Jira/Zendesk) | Ingest a **new** source with its own `id` and `relations.source_ids` pointing to the parent ticket | The comment is a distinct provider object |
| Edited ticket comment | Re-ingest the comment source with the same `id`, `upsert: true` | Same comment changed |
| Deleted ticket comment | Call [`DELETE /source`](/api-reference/v2/endpoint/delete-source) with the comment's `id` | The comment should no longer surface |
| New Slack/Teams message | Ingest a **new** `type: "slack"` source; link to the thread root via `relations.source_ids` | Messages are standalone events |
| New email reply | Ingest a **new** `type: "gmail"` source; link to the original via `relations` | Each email has its own provider ID |
| New Notion/Confluence page in a space | Ingest a **new** source; space goes in `tenant_metadata` | A page is a standalone document |
| Existing Notion/Confluence page edited | Re-ingest with the same `id`, `upsert: true` | The page document changed |

<Warning>
  `upsert: true` (the default) replaces the entire existing source with the new payload. To add a comment to a ticket without losing the ticket body, model the comment as its own source and link via `relations.source_ids`  -  don't mutate the parent.
</Warning>

### Slack message (with thread + attachment)

A Slack thread is several sources linked together. Use the message timestamp inside a stable composite ID so re-ingests upsert cleanly.

  "type": "slack",
  "kind": "message",
  "provider": "slack",
  "external_id": "1716213600.000100",
  "url": "https://acme.slack.com/archives/C01/p1716213600000100",
  "timestamp": "2026-05-20T10:00:00Z",
  "fields": {
    "kind": "message",
    "body": "The auth rollback notes are attached here.",
    "author": "alice",
    "thread_id": "1716213600.000100",

## 7. Forceful relations

`relations.source_ids` declares explicit links between sources. When a query matches one side of the link, HydraDB pulls the linked sources into the `additional_context` field of the [search response](/api-reference/v2/endpoint/search) alongside the primary chunks.

Use forceful relations for connections the text alone can't express:

- Reply / comment / message â†’ parent thread or ticket
- Linked tickets (`linked_to`, `blocks`, `child_of`)
- CRM opportunity â†’ support ticket
- Page â†’ referenced runbook or design doc

```json
{
  "relations": {
    "source_ids": ["jira_AUTH-100", "jira_AUTH-124"],
    "properties": { "reason": "linked issues" }
  }
}
```

<Warning>
- Forceful relations work **within the same store**  -  Knowledge â†’ Knowledge or Memory â†’ Memory. Pointing a Knowledge source at a Memory source ID (or vice versa) silently returns nothing at search.
- Forceful-relation context is only fetched when `mode: "thinking"` is set on the [search request](/api-reference/v2/endpoint/search). In `fast` mode, `search_forceful_relations` is ignored.
</Warning>

See [Knowledge - Forceful relations](/essentials/v2/knowledge#7-forceful-relations) for the full retrieval contract.

## 8. Attachments

Use `attachments` when you've already extracted file text and want it indexed alongside the parent source. Each attachment is preserved with the source and surfaced in search results.

```json
{
  "attachments": [
    {
      "id": "att-har-1",
      "title": "staging-login.har",
      "url": "https://...",
      "content_type": "application/json",
      "content": { "text": "HAR trace shows repeated 302 redirects after token refresh." },
      "misc": { "source": "linear", "comment_id": "c123" }
    }
  ]
}
```

| Attachment field | Effect |
|---|---|
| `content.text` / `content.markdown` | Indexed as searchable text linked to the parent source. |
| `url`, `title`, `content_type` | Preserved for display and citations. |
| `content_url` | Direct content URL distinct from the reference URL. |
| `misc` | Free-form metadata (provider IDs, comment IDs, etc.). |

A URL by itself is treated as metadata; HydraDB does **not** fetch and parse arbitrary attachment URLs from app-source payloads. Extract the text yourself and put it in `content.text`.

## 9. Metadata: scoping vs display

Two fields, two jobs.

| Use `tenant_metadata` when | Use `document_metadata` when |
|---|---|
| You want to **filter** at search time (`metadata_filters`) | The field is for display, citation, or bookkeeping only |
| The field is declared in the [tenant metadata schema](/api-reference/v2/endpoint/create-tenant) | The field is free-form and changes per source |
| Examples: `channel`, `project`, `space`, `pipeline`, `environment`, `region`, `doc_type` | Examples: `slack_ts`, `author`, `ticket_key`, `comment_id`, `gmail_message_id`, `stage`, `amount` |

Filterable keys must be declared in `tenant_metadata_schema` with `enable_match: true` at tenant creation. Undeclared keys in `tenant_metadata` are silently ignored at search time. To filter on `document_metadata`, nest under `document_metadata` in `metadata_filters`:

```json
{
  "query": "auth rollback discussion",
  "metadata_filters": {
    "channel": "engineering",
    "document_metadata": { "author": "alice" }
  }
}
```

See [Scoping using metadata](/essentials/v2/metadata) for the full schema design rules and filter shapes.

## 10. How app sources affect search

When `search_apps: true` is passed to the [`POST /search`](/api-reference/v2/endpoint/search) endpoint, HydraDB runs an app-aware search lane alongside normal semantic and lexical retrieval.

The app-aware lane does more than search raw text. It parses the query for app-specific hints (e.g., provider names, object types, exact external IDs, authors, thread keywords, or relationship terms like "linked to", "blocked by", "reply", or "parent") and uses these clues to build candidate sets and expand context around matching sources.

| Search signal | Uses |
| :--- | :--- |
| `kind`, `provider` | Routes queries to specific types, such as "Slack messages" or "Jira tickets". |
| `external_id` | Performs exact ID lookups like `AUTH-123` or page IDs. |
| Workflow state (`status`, `priority`) | Applies structured filters when the query implies them (e.g., "open tickets"). |
| Actors (`author`, `assignee`, etc.) | Answers queries like "tickets assigned to Alice" or "emails from Bob". |
| Forceful Relations | Follows `reply_to`, `comment_on`, `child_of`, `linked_to`, `blocks`, or `related_to` connections. |
| Attachments | Incorporates pre-extracted attachment text via auxiliary vector search. |

This means a query like *"show me Slack replies from Alice about AUTH-123"* combines text relevance with Slack scoping, actor matching, exact ticket ID matching, and thread/reply expansion.

### Parent and child search

Parent/child and hierarchical search is based on provider-scoped external IDs and relationships. When HydraDB ingests a source with a parent relation or `parent_id` hierarchy link, it indexes that connection in the context graph.

At recall time, `search_apps: true` lets the app-aware lane find an initial source. In `thinking` mode, if the query asks for parents, children, ancestors, descendants, epics, subtasks, accounts, deals, or similar hierarchy wording, HydraDB automatically expands around the matched source:

* **To find a parent:** HydraDB matches the source's parent relation to another source's `external_id` under the same `provider`.
* **To find children:** HydraDB looks for sources that point to the matched source's `external_id` as their parent.
* **Granular expansion:** Context expansion is bounded and can pull in grandparent/grandchild nodes when highly relevant to the query.

These expanded sources are brought into the retrieved results as connected context. Chunks retrieved through this path are tagged with `_retrieval_source: "app_relation_expansion"` in their metadata and may include an `_app_relation_type` property (e.g., `parent`, `child`, `grandparent`, `grandchild`).

<Note>
  Set `mode: "thinking"` and `search_apps: true` on your [search request](/api-reference/v2/endpoint/search) for optimal app-source behavior. This gives the relation-expansion and reranking layers the space to assemble complete conversational and workflow context.
</Note>

```json
{
  "tenant_id": "acme_corp",
  "query": "show me tickets linked to AUTH-123 and comments from Alice",
  "mode": "thinking",
  "search_apps": true,
  "max_results": 10
}
```

## 11. TLDR field/type reference

Use this as the compact contract for `app_knowledge` ingestion in API v2. Unlike v1 app sources, v2 uses a flat Knowledge-source shape: `type` is a source category tag, and provider-specific structure goes in `tenant_metadata`, `document_metadata`, `attachments`, and `relations`.

### Top-level app source fields

| Field | Why send it | Search / ingestion effect |
|---|---|---|
| <Field name="id" type="string" recommended /> | Stable HydraDB source ID for this app item. Compose it from provider IDs, such as `slack_<channel>_<ts>` or `jira_<key>`. | Used as the `source_id`, upsert key, status/deletion ID, and forceful-relation target. If omitted, HydraDB generates one. |
| <Field name="title" type="string" recommended /> | Human-readable title, subject, page name, ticket title, or message label. | Improves source display, snippets, citations, and semantic matching. |
| <Field name="type" type="string" recommended /> | Source category tag, such as `slack`, `gmail`, `jira`, `notion`, `salesforce`, `linear_comment`, or `custom`. | Preserved for display/filtering and useful for app-level scoping. This is not a discriminator; any stable string is allowed. |
| <Field name="description" type="string" /> | Optional long-form summary or context. | Adds additional searchable/display context. |
| <Field name="url" type="string" /> | Canonical link back to the provider item. | Improves citations and source navigation. |
| <Field name="timestamp" type="ISO-8601 datetime" /> | Creation, sent, or last-updated time. | Helps chronology, freshness, and display. |
| <Field name="content" type="object" required /> | Searchable body. Use `content.text` for plain text or `content.markdown` for Markdown. | Primary text that HydraDB chunks, embeds, graphs, and indexes. |
| <Field name="tenant_metadata" type="object" /> | Schema-aligned, filterable fields such as channel, project, space, mailbox, pipeline, region, or environment. | Searchable via `metadata_filters` when keys are declared in `tenant_metadata_schema`. |
| <Field name="document_metadata" type="object" /> | Free-form display/bookkeeping fields such as author, ticket key, provider IDs, status, assignee, thread ID, or stage. | Preserved on the source and searchable/filterable under `metadata_filters.document_metadata`. |
| <Field name="kind" type="string" recommended /> | App-source object kind, such as `email`, `message`, `ticket`, `knowledge_base`, `comment`, or `custom`. | Used by `search_apps: true` to route queries and enable specialized app search behavior. |
| <Field name="provider" type="string" recommended /> | App system/namespace the source came from (e.g., `slack`, `gmail`, `jira`, `notion`, `salesforce`, `linear`). | Prevents ID collisions across different systems and scopes searches to specific providers. |
| <Field name="external_id" type="string" recommended /> | Stable provider ID for this exact item in the external system. | Used for exact ID matching, resolving relations, and thread/hierarchy traversal. |
| <Field name="fields" type="object" recommended /> | Structured kind-specific fields. | Houses author, body, thread and parent identifiers. Put primary text inside `fields` instead of raw top-level `content`. |
| <Field name="attachments" type="array" /> | Pre-extracted files attached to this item. | Attachment text is indexed alongside the source. |
| <Field name="relations" type="object" /> | Forceful links to other sources: `{ "source_ids": ["..."] }`. | In `thinking` mode, linked Knowledge sources can be returned in `additional_context`. |

### Content fields

| Field | Why send it | Search / ingestion effect |
|---|---|---|
| <Field name="content.text" type="string" recommended /> | Plain-text body. | Indexed as the main source text. |
| <Field name="content.markdown" type="string" recommended /> | Markdown body when you have structure, headings, lists, or tables. | Indexed as the main source text with Markdown-aware structure. |

Send at least one of `content.text` or `content.markdown`. If you have both, keep them consistent and prefer the version that best preserves useful structure.

### Attachment fields

| Field | Why send it | Search / ingestion effect |
|---|---|---|
| <Field name="id" type="string" /> | Stable attachment ID. | Helps identify and deduplicate attachments. |
| <Field name="title" type="string" /> | File name or display title. | Improves attachment display and matching. |
| <Field name="url" type="string" /> | Link back to the provider attachment. | Improves citations and navigation. |
| <Field name="content_url" type="string" /> | Direct content URL, distinct from the reference URL. | Preserved as attachment metadata. |
| <Field name="content_type" type="string" /> | MIME type such as `application/pdf`, `text/markdown`, or `image/png`. | Helps classify the attachment. |
| <Field name="content.text" type="string" /> | Pre-extracted plain text. | Indexed as attachment text. |
| <Field name="content.markdown" type="string" /> | Pre-extracted Markdown. | Indexed as attachment text while preserving structure. |
| <Field name="misc" type="object" /> | Provider-specific attachment metadata. | Preserved for display/bookkeeping. |

### Relation fields

| Field | Why send it | Search / ingestion effect |
|---|---|---|
| <Field name="relations.source_ids" type="string[]" required /> | Source IDs of related Knowledge sources. | Enables forceful relation expansion in `mode: "thinking"`. |
| <Field name="relations.properties" type="object" /> | Optional metadata about the relationship, such as reason, relation type, or confidence. | Preserved with the declared relation. |

Only link to source IDs in the same store. For app sources, that means Knowledge â†’ Knowledge. If the related item may arrive later, still use the stable source ID you plan to ingest for that item.

### Recommended `type` values and mappings

`type` is free-form, but using consistent values makes filtering, analytics, and source rendering easier.

| App concept | Recommended `type` | `content` | `tenant_metadata` | `document_metadata` | Link with `relations.source_ids` |
|---|---|---|---|---|---|
| Email message | `gmail`, `outlook`, `email` | Subject/body text or thread excerpt | Mailbox, team, workspace | Provider message ID, thread ID, from/to/cc/bcc, reply/forward IDs | Parent email, thread root, forwarded source |
| Chat message | `slack`, `teams`, `discord`, `message` | Message body | Channel, workspace, team | Message timestamp, thread timestamp, author, mentions | Thread root or replied-to message |
| Ticket / issue / support case | `jira`, `linear`, `zendesk`, `github_issue`, `ticket` | Title + description + current status summary | Project, queue, product, environment | Ticket key, status, priority, assignee, reporter, parent epic ID | Parent epic, linked issues, blockers, related tickets |
| Knowledge-base page | `notion`, `confluence`, `google_docs`, `knowledge_base` | Page body as text/Markdown | Space, database, collection, doc type | Page ID, parent page ID, created_by, updated_by | Parent page, referenced docs, related runbooks |
| First-class comment | `<provider>_comment`, such as `linear_comment` or `jira_comment` | Comment body | Project, ticket queue, channel, workspace | Comment ID, author, parent ticket/page/source ID | Parent ticket/page/source |
| CRM/business record | `salesforce`, `hubspot`, `crm`, `custom` | Record summary plus important fields | Pipeline, region, segment, account tier | Record type, record ID, stage, amount, owner, account ID | Account, deal, contact, support ticket |
| Web page / CMS item | `webpage`, `cms`, `blog` | Page/article body | Site, collection, locale, topic | Slug, author, published_at | Canonical/related pages |
| Internal workflow record | `custom`, `workflow`, `<system_name>` | Summary, status, and key details | Workflow, team, environment | Record ID, owner, status, parent record ID | Parent/child records or related sources |

<Note>
  For provider-specific IDs, prefer stable composite `id` values and mirror the raw provider IDs in `document_metadata`. For scope fields you need to filter on, use `tenant_metadata` and declare the keys in the tenant schema.
</Note>

## 12. Common mistakes

| Mistake | Why it breaks | Fix |
|---|---|---|
| Sending `app_knowledge` as a JSON object instead of a stringified JSON | `multipart/form-data` validation fails | `JSON.stringify(...)` (TS) or `json.dumps(...)` (Python) before sending |
| Putting filterable fields in `document_metadata` | Filters silently ignored | Move them to `tenant_metadata` and declare in the [tenant schema](/api-reference/v2/endpoint/create-tenant) |
| Re-ingesting a Slack thread on every new reply | Loses message-level authorship and timestamps; wastes indexing | Ingest each reply as a new source; link via `relations.source_ids` |
| Mutating a ticket source to add a comment | `upsert` replaces the whole snapshot | Ingest the comment as its own source with a link back to the ticket |
| Using a non-stable string for `id` | Re-ingests create duplicate sources instead of upserting | Compose `id` from stable provider IDs (e.g. `slack_<channel>_<ts>`, `jira_<key>`) |
| Pointing `relations.source_ids` at memory IDs from a Knowledge source | Silently returns nothing at search | Forceful relations stay within one store - Knowledge â†” Knowledge, Memory â†” Memory |
| Skipping [`/source/status`](/api-reference/v2/endpoint/source-status) before searching | Source not indexed yet; empty results | Poll until `indexing_status: "completed"` (or `graph_creation` for partial availability) |
| Forgetting to set `search_apps: true` on [`POST /search`](/api-reference/v2/endpoint/search) | Returns only text/vector matches without the app-aware lane expansion | Always set `search_apps: true` (and use `mode: "thinking"`) when querying app-source data |
