---
title: "Scoping using metadata"
description: "How HydraDB uses metadata to scope and rank query results deterministically — schema-aligned tenant fields, free-form per-document fields, and how they combine at query time."
---

Metadata is structured data attached to [Knowledge](/essentials/v2/knowledge) and [Memories](/essentials/v2/memories) so you can scope query to a known set **before** semantic retrieval runs. It bridges the gap between vector similarity (great for "things like X") and exact scoping (essential for "must be in environment=production, must be customer=acme, must be a runbook"). HydraDB exposes two metadata layers  -  one declared up front and fast, one free-form and slower  -  and lets you combine them in a single [`POST /query`](/api-reference/v2/endpoint/query) call.

For the full request and response schema, see [Query  -  API Reference](/api-reference/v2/endpoint/query).

---

## 1. Use-case recipes

Pick the row that matches your goal and use the parameters as a starting point:

| I want to… | Put the value in… | Scope at query with… | Notes |
| --- | --- | --- | --- |
| Scope by a field used on every query (department, region, plan) | `metadata` on ingest | Top-level key in `metadata_filters` | Fast path. Must declare in `tenant_metadata_schema` with `enable_match: true`. |
| Scope by a one-off or per-document field (author, slack channel, doc ID) | `additional_metadata` on ingest | Nested under `additional_metadata` in `metadata_filters` | No schema change. ~3× over-fetch, slower than the fast path. |
| Query inside a metadata field (semantic match on a description) | `metadata` with `enable_dense_embedding: true` | Just include the field's content in your `query` | Requires the schema field to be `VARCHAR`. |
| Keyword match on a metadata field | `metadata` with `enable_sparse_embedding: true` | `query_by: "text"` on `/query` | BM25 over that field. |
| Combine deterministic scoping with semantic search | Both metadata kinds | `metadata_filters` + your query | Scope runs before ranking  -  keeps query predictable. |
| Keep draft / archived / region-specific content out of user answers | `metadata.status` (or similar) | `metadata_filters: { status: "published" }` | Classic published/unpublished pattern. |

If a field will be scoped on **every** query, it belongs in `metadata` and the [tenant schema](/api-reference/v2/endpoint/create-tenant). If it's ad-hoc or unique to one document, use `additional_metadata` instead.

## 2. Parameter reference

[Follow this for when to use `tenant_id` and `sub_tenant_id`](./multi-tenant#2-when-to-use-each)

### Schema declaration (set at tenant creation)

Defined once in `tenant_metadata_schema` on [`POST /tenants`](/api-reference/v2/endpoint/create-tenant). Immutable after creation  -  plan your scope fields up front.

| Field | Type / values | Purpose |
| --- | --- | --- |
| `name` <sup style={{ color: "red" }}>*</sup> | string | Field key. Must start with a letter or `_`. Reserved system names are rejected. |
| `data_type` | `"string"` \| `"integer"` \| `"float"` \| `"boolean"` \| `"array"` \| `"object"` \| `VARCHAR` \| `INT*` \| `JSON` … | Field type. Default: `VARCHAR`. Friendly names are coerced server-side. |
| `enable_match` | boolean | Set `true` to use this field as a scope key in `metadata_filters`. Default: `false`. |
| `enable_dense_embedding` | boolean | Index this `VARCHAR` field for semantic similarity search. Default: `false`. |
| `enable_sparse_embedding` | boolean | Index this `VARCHAR` field for BM25 keyword search. Default: `false`. |
| `max_length` | integer | Max length for string fields. Default: `1024`. Increase for long text. |
| `searchable` *(shorthand)* | boolean | Equivalent to `enable_dense_embedding: true` + `enable_sparse_embedding: true`. |
| `filterable` *(shorthand)* | boolean | Equivalent to `enable_match: true`. |

### Ingest-time metadata (per source)

Sent on each item in `app_knowledge[]`, `document_metadata[]`, or `memories[]` on [`POST /context/ingest`](/api-reference/v2/endpoint/ingest-context):

| Field | Type | Purpose |
| --- | --- | --- |
| `metadata` | object for knowledge; JSON-encoded string for memory items | Tenant-schema fields. Keys must match `tenant_metadata_schema`. The fast scoping path. |
| `additional_metadata` | object | Free-form per-document or per-memory fields. No schema. The flexible, slower path. |

<Warning>
For `type=memory`, the `memories` multipart field is already JSON-stringified, and each memory item's `metadata` is currently validated as a JSON-encoded string. Keep `additional_metadata` as an object. For knowledge ingestion (`document_metadata` and `app_knowledge`), `metadata` remains an object.
</Warning>

### Query-time scoping (in `metadata_filters`)

Sent on [`POST /query`](/api-reference/v2/endpoint/query). Two scopes in the same object, mixed freely:

| Where the key lives | What it scopes | Performance |
| --- | --- | --- |
| **Top level** of `metadata_filters` | `metadata` fields | Applied in the vector store before ranking. Fast. |
| Nested under **`additional_metadata`** (canonical) or `additional_metadata` (deprecated alias) | `additional_metadata` fields | Post-retrieval, with the engine over-fetching ~3× to compensate. Slower. |

## 3. Tuning heuristics

Most of the time the defaults are right. When they aren't, here's where to start:

- **Plan scoping fields before first ingest.** Schema is immutable, and undeclared scope keys are silently ignored at query time. If you'll scope on it more than once, declare it.
- **Pick `metadata` for hot paths, `additional_metadata` for cold ones.** Tenant-level scopes are pre-applied in the vector store; document-level scopes force a post-retrieval pass with over-fetch.
- **Use exact-equality only.** `metadata_filters` are equality constraints across all categories. Range, contains, or fuzzy matching belong in the query, query mode, or downstream reranking  -  not here.
- **Keep keys stable.** Renaming a metadata key requires re-ingesting affected sources. Same goes for changing `enable_match` flags.
- **Ingested metadata is immutable.** Once a source is indexed, its `metadata` and `additional_metadata` values are locked. To change them, re-ingest the source with the new values (use `upsert: true` and the same `id`).
- **Don't substitute `metadata_filters` for `sub_tenant_id`.** `metadata_filters` scopes results *inside* a partition. For partitioning by user, team, or workspace, use [`sub_tenant_id`](/essentials/v2/multi-tenant).

## 4. Minimal working example

Two phases: **set up metadata** (declare the schema, then attach values at ingest), then **scope at query**.

### Step 1  -  Create metadata

The schema lives at the tenant level; values land on each source at ingest time. Both happen before any query.

#### Step 1a  -  Declare the schema at tenant creation

<CodeGroup>
```bash cURL
curl -X POST 'https://api.hydradb.com/tenants' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "acme_corp",
    "tenant_metadata_schema": [
      {
        "name": "compliance_framework",
        "data_type": "VARCHAR",
        "enable_match": true
      }
    ]
  }'
```
```typescript TypeScript SDK
await client.tenants.create({
  tenantId: "acme_corp",
  tenantMetadataSchema: [
    {
      name: "compliance_framework",
      dataType: "VARCHAR",
      enableMatch: true,
    },
  ],
});
```
```python Python SDK
client.tenants.create(
    tenant_id="acme_corp",
    tenant_metadata_schema=[
        {
            "name": "compliance_framework",
            "data_type": "VARCHAR",
            "enable_match": True,
        }
    ],
)
```
</CodeGroup>

#### Step 1b  -  Attach metadata at ingest

Drop schema-aligned values into `metadata` and any free-form per-document values into `additional_metadata`:

<CodeGroup>
```bash cURL
curl -X POST 'https://api.hydradb.com/context/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "tenant_id": "acme_corp",
      "sub_tenant_id": "default",
      "id": "auth-controls-001",
      "kind": "knowledge_base",
      "fields": {
        "title": "Authentication controls",
        "body": "Authentication controls are reviewed quarterly for SOC2."
      },
      "metadata": { "compliance_framework": "SOC2" },
      "additional_metadata": { "author": "alice", "doc_version": 3 }
    }
  ]'
```
```typescript TypeScript SDK
await client.context.ingest({
  type: "knowledge",
  tenantId: "acme_corp",
  appKnowledge: JSON.stringify([
    {
      tenant_id: "acme_corp",
      sub_tenant_id: "default",
      id: "auth-controls-001",
      kind: "knowledge_base",
      fields: {
        title: "Authentication controls",
        body: "Authentication controls are reviewed quarterly for SOC2.",
      },
      metadata: { compliance_framework: "SOC2" },
      additional_metadata: { author: "alice", doc_version: 3 },
    },
  ]),
});
```
```python Python SDK
import json

client.context.ingest(
    type="knowledge",
    tenant_id="acme_corp",
    app_knowledge=json.dumps([
        {
            "tenant_id": "acme_corp",
            "sub_tenant_id": "default",
            "id": "auth-controls-001",
            "kind": "knowledge_base",
            "fields": {
                "title": "Authentication controls",
                "body": "Authentication controls are reviewed quarterly for SOC2.",
            },
            "metadata": {"compliance_framework": "SOC2"},
            "additional_metadata": {"author": "alice", "doc_version": 3},
        }
    ]),
)
```
</CodeGroup>

<Warning>
**Ingested metadata is immutable.** Once a source is indexed, its `metadata` and `additional_metadata` values are locked. To change them, re-ingest the source with the new values using `upsert: true` and the same `id`.
</Warning>

### Step 2  -  Scope at query

Mix tenant-level (top-level) and document-level (nested) scopes in the same `metadata_filters` object:

<CodeGroup>
```bash cURL
curl -X POST 'https://api.hydradb.com/query' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "acme_corp",
    "query": "How does authentication control review work?",
    "type": "knowledge",
    "query_by": "hybrid",
    "metadata_filters": {
      "compliance_framework": "SOC2",
      "additional_metadata": { "author": "alice" }
    }
  }'
```
```typescript TypeScript SDK
const result = await client.query({
  tenantId: "acme_corp",
  query: "How does authentication control review work?",
  type: "knowledge",
  queryBy: "hybrid",
  metadataFilters: {
    compliance_framework: "SOC2",
    additional_metadata: { author: "alice" },
  },
});
```
```python Python SDK
result = client.query(
    tenant_id="acme_corp",
    query="How does authentication control review work?",
    type="knowledge",
    query_by="hybrid",
    metadata_filters={
        "compliance_framework": "SOC2",
        "additional_metadata": {"author": "alice"},
    },
)
```
</CodeGroup>

Top-level `compliance_framework` is the fast scope, applied in the vector store before ranking. Nested `additional_metadata.author` is matched post-retrieval on the free-form field.

## 5. Common mistakes

| Symptom | Cause | Fix |
| --- | --- | --- |
| `metadata_filters` doesn't scope results | Key isn't declared in `tenant_metadata_schema`, or `enable_match` is `false` | Re-create the tenant with the field declared (`enable_match: true`), or move the field to `additional_metadata` and nest the scope under `additional_metadata`. |
| Scope on an `additional_metadata` key silently ignored | Scope passed as top-level key | Nest it: `metadata_filters: { additional_metadata: { author: "alice" } }`. |
| Schema field changes don't take effect | Schema is immutable after tenant creation | Create a new tenant with the corrected schema. There's no in-place schema migration. |
| Metadata values on a source don't update | Ingested metadata is immutable | Re-ingest the source with `upsert: true` and the same `id`. |
| Query returns 0 results after adding a scope | Over-scoping  -  combined constraints exclude everything | Start with the minimum hard constraints; add scopes one at a time and re-check counts. |
| Range / contains / fuzzy scope doesn't work | `metadata_filters` is equality-only | Move that constraint into the `query` text, use a different `query_by`, or apply it in your own post-scoping. |
| Fields used in `metadata` but missing from response objects | Field declared as embedding-only without `enable_match` | Embedding-enabled doesn't imply scoping-enabled. Add `enable_match: true` to use it in `metadata_filters`. |
| `additional_metadata` scope feels slow | It is  -  over-fetches ~3× to compensate for post-retrieval matching | Move the hot field into `metadata` and declare it in the schema. |

## 6. Advanced patterns

**Stacked scopes with sub-tenant partitioning.** Use `sub_tenant_id` for the partition (per-user, per-workspace), and use `metadata_filters` to scope *inside* that partition. They're complementary, not interchangeable. See [Multi-Tenant](/essentials/v2/multi-tenant).

**Published vs draft.** Add a `status` field with `enable_match: true` to your schema; tag every source with `metadata.status = "draft" | "published"`; pass `metadata_filters: { status: "published" }` on user-facing queries. Keeps work-in-progress out of customer answers automatically.

**Multi-language corpora.** Add a `language` field with `enable_match: true`; route each query to the right language by passing `metadata_filters: { language: detect_language(query) }`.

**Schema-as-product.** Treat `tenant_metadata_schema` as part of your data contract  -  review it like a database migration. The cost of getting it wrong (immutability + re-ingest) is real; the cost of getting it right is one extra meeting.
