---
title: "Scoping using metadata"
description: "How HydraDB uses metadata to scope and rank search results deterministically — schema-aligned tenant fields, free-form per-document fields, and how they combine at search time."
---

Metadata is structured data attached to [Knowledge](/essentials/v2/knowledge) and [Memories](/essentials/v2/memories) so you can scope search to a known set **before** semantic retrieval runs. It bridges the gap between vector similarity (great for "things like X") and exact scoping (essential for "must be in environment=production, must be customer=acme, must be a runbook"). HydraDB exposes two metadata layers  -  one declared up front and fast, one free-form and slower  -  and lets you combine them in a single [`POST /search`](/api-reference/v2/endpoint/search) call.

For the full request and response schema, see [Search  -  API Reference](/api-reference/v2/endpoint/search).

---

## 1. Use-case recipes

Pick the row that matches your goal and use the parameters as a starting point:

| I want to… | Put the value in… | Scope at search with… | Notes |
| --- | --- | --- | --- |
| Scope by a field used on every query (department, region, plan) | `tenant_metadata` on ingest | Top-level key in `metadata_filters` | Fast path. Must declare in `tenant_metadata_schema` with `enable_match: true`. |
| Scope by a one-off or per-document field (author, slack channel, doc ID) | `additional_metadata` on ingest | Nested under `additional_metadata` in `metadata_filters` | No schema change. ~3× over-fetch, slower than the fast path. |
| Search inside a metadata field (semantic match on a description) | `tenant_metadata` with `enable_dense_embedding: true` | Just include the field's content in your `query` | Requires the schema field to be `VARCHAR`. |
| Keyword match on a metadata field | `tenant_metadata` with `enable_sparse_embedding: true` | `search_by: "text"` on `/search` | BM25 over that field. |
| Combine deterministic scoping with semantic search | Both metadata kinds | `metadata_filters` + your query | Scope runs before ranking  -  keeps search predictable. |
| Keep draft / archived / region-specific content out of user answers | `tenant_metadata.status` (or similar) | `metadata_filters: { status: "published" }` | Classic published/unpublished pattern. |

If a field will be scoped on **every** query, it belongs in `tenant_metadata` and the [tenant schema](/api-reference/v2/endpoint/create-tenant). If it's ad-hoc or unique to one document, use `additional_metadata` instead.

## 2. Parameter reference

[Follow this for when to use `tenant_id` and `sub_tenant_id`](./multi-tenant#2-when-to-use-each)

### Schema declaration (set at tenant creation)

Defined once in `tenant_metadata_schema` on [`POST /tenants`](/api-reference/v2/endpoint/create-tenant). Immutable after creation  -  plan your scope fields up front.

| Field | Type / values | Purpose |
| --- | --- | --- |
| `name` <sup style={{ color: "red" }}>*</sup> | string | Field key. Must start with a letter or `_`. Reserved system names are rejected. |
| `data_type` | `"string"` \| `"integer"` \| `"float"` \| `"boolean"` \| `"array"` \| `"object"` \| `VARCHAR` \| `INT*` \| `JSON` … | Field type. Default: `VARCHAR`. Friendly names are coerced server-side. |
| `enable_match` | boolean | Set `true` to use this field as a scope key in `metadata_filters`. Default: `false`. |
| `enable_dense_embedding` | boolean | Index this `VARCHAR` field for semantic similarity search. Default: `false`. |
| `enable_sparse_embedding` | boolean | Index this `VARCHAR` field for BM25 keyword search. Default: `false`. |
| `max_length` | integer | Max length for string fields. Default: `1024`. Increase for long text. |
| `searchable` *(shorthand)* | boolean | Equivalent to `enable_dense_embedding: true` + `enable_sparse_embedding: true`. |
| `filterable` *(shorthand)* | boolean | Equivalent to `enable_match: true`. |

### Ingest-time metadata (per source)

Sent on each item in `app_knowledge[]`, `file_metadata[]`, or `memories[]` on [`POST /source/ingest`](/api-reference/v2/endpoint/ingest-content):

| Field | Type | Purpose |
| --- | --- | --- |
| `tenant_metadata` (alias `metadata`) | object | Tenant-schema fields. Keys must match `tenant_metadata_schema`. The fast scoping path. |
| `additional_metadata` (deprecated alias `document_metadata`) | object | Free-form per-document fields. No schema. The flexible, slower path. |

### Search-time scoping (in `metadata_filters`)

Sent on [`POST /search`](/api-reference/v2/endpoint/search). Two scopes in the same object, mixed freely:

| Where the key lives | What it scopes | Performance |
| --- | --- | --- |
| **Top level** of `metadata_filters` | `tenant_metadata` fields | Applied in the vector store before ranking. Fast. |
| Nested under **`additional_metadata`** (canonical) or `document_metadata` (deprecated alias) | `additional_metadata` fields | Post-retrieval, with the engine over-fetching ~3× to compensate. Slower. |

## 3. Tuning heuristics

Most of the time the defaults are right. When they aren't, here's where to start:

- **Plan scoping fields before first ingest.** Schema is immutable, and undeclared scope keys are silently ignored at search time. If you'll scope on it more than once, declare it.
- **Pick `tenant_metadata` for hot paths, `additional_metadata` for cold ones.** Tenant-level scopes are pre-applied in the vector store; document-level scopes force a post-retrieval pass with over-fetch.
- **Use exact-equality only.** `metadata_filters` are equality constraints across all categories. Range, contains, or fuzzy matching belong in the query, search mode, or downstream reranking  -  not here.
- **Keep keys stable.** Renaming a metadata key requires re-ingesting affected sources. Same goes for changing `enable_match` flags.
- **Ingested metadata is immutable.** Once a source is indexed, its `tenant_metadata` and `additional_metadata` values are locked. To change them, re-ingest the source with the new values (use `upsert: true` and the same `source_id`).
- **Don't substitute `metadata_filters` for `sub_tenant_id`.** `metadata_filters` scopes results *inside* a partition. For partitioning by user, team, or workspace, use [`sub_tenant_id`](/essentials/v2/multi-tenant).

## 4. Minimal working example

Two phases: **set up metadata** (declare the schema, then attach values at ingest), then **scope at search**.

### Step 1  -  Create metadata

The schema lives at the tenant level; values land on each source at ingest time. Both happen before any search.

#### Step 1a  -  Declare the schema at tenant creation

<CodeGroup>
```python Python SDK
client.tenants.create(
    tenant_id="acme_corp",
    tenant_metadata_schema=[
        {
            "name": "compliance_framework",
            "data_type": "VARCHAR",
            "enable_match": True,
        }
    ],
)
```
```typescript TypeScript SDK
await client.tenants.create({
  tenant_id: "acme_corp",
  tenant_metadata_schema: [
    {
      name: "compliance_framework",
      data_type: "VARCHAR",
      enable_match: true,
    },
  ],
});
```
```bash cURL
curl -X POST 'https://api.hydradb.com/tenants' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "acme_corp",
    "tenant_metadata_schema": [
      {
        "name": "compliance_framework",
        "data_type": "VARCHAR",
        "enable_match": true
      }
    ]
  }'
```
</CodeGroup>

#### Step 1b  -  Attach metadata at ingest

Drop schema-aligned values into `tenant_metadata` and any free-form per-document values into `additional_metadata`:

<CodeGroup>
```python Python SDK
import json

client.source.ingest(
    type="knowledge",
    tenant_id="acme_corp",
    app_knowledge=json.dumps([
        {
            "tenant_id": "acme_corp",
            "sub_tenant_id": "default",
            "id": "auth-controls-001",
            "kind": "knowledge_base",
            "fields": {
                "title": "Authentication controls",
                "body": "Authentication controls are reviewed quarterly for SOC2.",
            },
            "tenant_metadata": {"compliance_framework": "SOC2"},
            "additional_metadata": {"author": "alice", "doc_version": 3},
        }
    ]),
)
```
```typescript TypeScript SDK
await client.source.ingest({
  type: "knowledge",
  tenant_id: "acme_corp",
  app_knowledge: JSON.stringify([
    {
      tenant_id: "acme_corp",
      sub_tenant_id: "default",
      id: "auth-controls-001",
      kind: "knowledge_base",
      fields: {
        title: "Authentication controls",
        body: "Authentication controls are reviewed quarterly for SOC2.",
      },
      tenant_metadata: { compliance_framework: "SOC2" },
      additional_metadata: { author: "alice", doc_version: 3 },
    },
  ]),
});
```
```bash cURL
curl -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=knowledge" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "tenant_id": "acme_corp",
      "sub_tenant_id": "default",
      "id": "auth-controls-001",
      "kind": "knowledge_base",
      "fields": {
        "title": "Authentication controls",
        "body": "Authentication controls are reviewed quarterly for SOC2."
      },
      "tenant_metadata": { "compliance_framework": "SOC2" },
      "additional_metadata": { "author": "alice", "doc_version": 3 }
    }
  ]'
```
</CodeGroup>

<Warning>
**Ingested metadata is immutable.** Once a source is indexed, its `tenant_metadata` and `additional_metadata` values are locked. To change them, re-ingest the source with the new values using `upsert: true` and the same `source_id`.
</Warning>

### Step 2  -  Scope at search

Mix tenant-level (top-level) and document-level (nested) scopes in the same `metadata_filters` object:

<CodeGroup>
```python Python SDK
result = client.search.query(
    tenant_id="acme_corp",
    query="How does authentication control review work?",
    source="knowledge",
    search_by="hybrid",
    metadata_filters={
        "compliance_framework": "SOC2",
        "additional_metadata": {"author": "alice"},
    },
)
```
```typescript TypeScript SDK
const result = await client.search.query({
  tenant_id: "acme_corp",
  query: "How does authentication control review work?",
  source: "knowledge",
  search_by: "hybrid",
  metadata_filters: {
    compliance_framework: "SOC2",
    additional_metadata: { author: "alice" },
  },
});
```
```bash cURL
curl -X POST 'https://api.hydradb.com/search' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "acme_corp",
    "query": "How does authentication control review work?",
    "source": "knowledge",
    "search_by": "hybrid",
    "metadata_filters": {
      "compliance_framework": "SOC2",
      "additional_metadata": { "author": "alice" }
    }
  }'
```
</CodeGroup>

Top-level `compliance_framework` is the fast scope, applied in the vector store before ranking. Nested `additional_metadata.author` is matched post-retrieval on the free-form field.

## 5. Common mistakes

| Symptom | Cause | Fix |
| --- | --- | --- |
| `metadata_filters` doesn't scope results | Key isn't declared in `tenant_metadata_schema`, or `enable_match` is `false` | Re-create the tenant with the field declared (`enable_match: true`), or move the field to `additional_metadata` and nest the scope under `additional_metadata`. |
| Scope on an `additional_metadata` key silently ignored | Scope passed as top-level key | Nest it: `metadata_filters: { additional_metadata: { author: "alice" } }`. |
| Schema field changes don't take effect | Schema is immutable after tenant creation | Create a new tenant with the corrected schema. There's no in-place schema migration. |
| Metadata values on a source don't update | Ingested metadata is immutable | Re-ingest the source with `upsert: true` and the same `source_id`. |
| Search returns 0 results after adding a scope | Over-scoping  -  combined constraints exclude everything | Start with the minimum hard constraints; add scopes one at a time and re-check counts. |
| Range / contains / fuzzy scope doesn't work | `metadata_filters` is equality-only | Move that constraint into the `query` text, use a different `search_by`, or apply it in your own post-scoping. |
| Fields used in `metadata` but missing from response objects | Field declared as embedding-only without `enable_match` | Embedding-enabled doesn't imply scoping-enabled. Add `enable_match: true` to use it in `metadata_filters`. |
| `additional_metadata` scope feels slow | It is  -  over-fetches ~3× to compensate for post-retrieval matching | Move the hot field into `tenant_metadata` and declare it in the schema. |

## 6. Advanced patterns

**Stacked scopes with sub-tenant partitioning.** Use `sub_tenant_id` for the partition (per-user, per-workspace), and use `metadata_filters` to scope *inside* that partition. They're complementary, not interchangeable. See [Multi-Tenant](/essentials/v2/multi-tenant).

**Published vs draft.** Add a `status` field with `enable_match: true` to your schema; tag every source with `tenant_metadata.status = "draft" | "published"`; pass `metadata_filters: { status: "published" }` on user-facing queries. Keeps work-in-progress out of customer answers automatically.

**Multi-language corpora.** Add a `language` field with `enable_match: true`; route each query to the right language by passing `metadata_filters: { language: detect_language(query) }`.

**Schema-as-product.** Treat `tenant_metadata_schema` as part of your data contract  -  review it like a database migration. The cost of getting it wrong (immutability + re-ingest) is real; the cost of getting it right is one extra meeting.
