## When to use it

`POST /source/ingest` is asynchronous. Use this endpoint to know when content is ready to be searched. Common patterns:

- **Polling after ingest** — wait until status is `completed` before running searches
- **Bulk progress tracking** — pass multiple `source_id`s in one call to check a whole batch
- **Error inspection** — see why a specific item failed via `error_code` and `message`

Works for both knowledge sources and memory items — pass the `source_id` you got from the ingest response.

## Endpoint

- **Auth:** Bearer token + `API-Version: 2`
- **Idempotency:** Read-only
- **Async:** No

## Query parameters

| Name | Type | Required | Default | Description |
|---|---|---|---|---|
| `file_id` | string | No | – | **Deprecated** — single ID. Use `file_ids` instead. |
| `file_ids` | string[] | No | `[]` | One or more `source_id` values returned at ingestion. Despite the name, accepts IDs of any source — files, app sources, memories. |
| `tenant_id` | string | No | `""` | Tenant the items belong to. |
| `sub_tenant_id` | string \| null | No | `null` | Sub-tenant scope. If omitted, the default sub-tenant is used. |

<Info>
The query parameter is named `file_ids` for legacy reasons. It accepts the `source_id` returned by both file uploads and app sources and memories — not only file-backed sources.
</Info>

## Response

`200` response body matches OpenAPI schema **`BatchProcessingStatus`** with one `ProcessingStatus` per requested ID.

```json
{
  "statuses": [
    {
      "file_id": "policy_v2",
      "indexing_status": "completed",
      "error_code": "",
      "success": true,
      "message": "Processing status retrieved successfully"
    },
    {
      "file_id": "runbook_deploy",
      "indexing_status": "graph_creation",
      "error_code": "",
      "success": true,
      "message": "Processing status retrieved successfully"
    }
  ]
}
```

| Field | Description |
|---|---|
| `statuses[]` | Array — one entry per requested ID. |
| `statuses[].file_id` | The ID being reported on (string). |
| `statuses[].indexing_status` | Current status (enum). See [Status values](#status-values). |
| `statuses[].error_code` | Diagnostic code if `errored`; empty string otherwise. |
| `statuses[].success` | Boolean — `true` once the source reaches `completed`. |
| `statuses[].message` | Detailed status message (string). |

| Status | Searchable? | Meaning |
|---|---|---|
| `queued` | No | Accepted by the server, not yet picked up by a worker. |
| `processing` | No | Content is being parsed, chunked, and embedded. |
| `graph_creation` | **Yes** | Indexed and retrievable; the knowledge graph is still being built. Already searchable via `/search`, but graph context may still be incomplete. |
| `completed` | Yes | Fully indexed and graphed. Ready for all retrieval modes. |
| `errored` | No | Processing failed. Inspect `error_code` and `message`. |
| `success` | Yes | **Alias for `completed`** that may appear in some legacy responses. |

## Polling pattern

```python
import time

source_ids = ["policy_v2", "runbook_deploy"]

while True:
    response = client.source.status(
        tenant_id="acme_corp",
        sub_tenant_id="default",
        file_ids=source_ids,
    )
    statuses = [s.indexing_status for s in response.statuses]

    if all(s in ("completed", "success") for s in statuses):
        break
    if any(s == "errored" for s in statuses):
        # inspect response.statuses for which failed
        break

    time.sleep(5)
```

Typical processing time:

- **Memories** (text, markdown, conversation pairs): seconds
- **Small documents** (under 50 pages): 1–5 minutes
- **Large documents** (50+ pages): 5–15 minutes

## Behavior notes

<Info>
**`graph_creation` is searchable.** Items in this state are already retrievable via `POST /search`. Wait for `completed` only when you specifically need full graph traversal (`graph_context: true`).
</Info>

<Info>
**Unknown IDs return as `errored`.** If you pass an ID that doesn't exist (e.g., typo), HydraDB returns an entry with `indexing_status: "errored"` rather than silently dropping it.
</Info>
