## Request body

| Name | Description |
| --- | --- |
| <Field name="tenant_id" type="string" required /> | Account-scoped tenant identifier. Use a stable, case-sensitive ID up to 25 characters; prefer lowercase letters, numbers, and underscores for portability. |
| <Field name="tenant_metadata_schema" type="array" /> | Defines tenant-level metadata fields. See the [Scoping using metadata](/essentials/v2/metadata#schema-declaration-set-at-tenant-creation) guide for detailed schema parameters. (default=`null`) |

## What happens next?

After creation, follow this sequence before serving user traffic:

1. Create the tenant with `POST /tenants`.
2. Poll [Tenant Status](/api-reference/v2/endpoint/tenant-status) until the tenant is ready.
3. Ingest files, app sources, or memories with [Ingest Content](/api-reference/v2/endpoint/ingest-content).
4. Save every returned `results[].source_id` and poll [Source Status](/api-reference/v2/endpoint/source-status).
5. Search once each recently ingested source reaches `graph_creation` or `completed`.

## Metadata schema

You can define a custom schema at tenant creation to enable fast, deterministic exact-match filtering or text-based search on metadata fields. 

For detailed parameters, valid data types, shorthand flags, and comprehensive configuration examples, see the [Scoping using metadata](/essentials/v2/metadata) guide.

<Note>
Schema field names are **immutable** after tenant creation. You can add per-document free-form metadata fields at ingestion time, but tenant-level fields cannot be renamed, removed, or have their flags changed in-place. Plan your schema carefully before creating the tenant.
</Note>

## Behavior notes

<Warning>
**Do not ingest content prematurely.** Tenant creation is asynchronous. For normal `/source/ingest` tenants, both `vectorstore_status.knowledge` and `vectorstore_status.memories`, as well as `scheduler_status` and `graph_status`, must be `true` before the tenant is ready. Always poll [`GET /tenants/status?tenant_id=...`](/api-reference/v2/endpoint/tenant-status) before your first ingestion or search call.
</Warning>

- **Default sub-tenant:** HydraDB assigns an auto-generated default sub-tenant ID for calls that omit `sub_tenant_id`. That default is created once and reused. See [Concepts → Multi-Tenant](/essentials/v2/multi-tenant) for details.
- **Reserved field names:** Some names are reserved by the indexing layer (e.g., system fields used to track ingestion and graph state). `name` values are validated server-side; reserved names return a `422 VALIDATION_ERROR`. Use prefixes like `app_` if you're worried about collisions.

## Errors

Common codes: `400 INVALID_PARAMETERS`, `403 FORBIDDEN`, `409 TENANT_ALREADY_EXISTS`, `422 VALIDATION_ERROR`. `403 FORBIDDEN` can mean the API key is not permitted to create tenants or the account has reached its tenant/plan limit. `409 TENANT_ALREADY_EXISTS` means the `tenant_id` is already visible to the current API key; use a different ID or continue with the existing tenant. See [Error Responses](/api-reference/v2/error-responses) for the full list.
