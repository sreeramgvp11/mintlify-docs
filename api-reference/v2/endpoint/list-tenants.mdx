Discover the tenants your API key has access to  -  useful for admin tooling, debug dashboards, populating a tenant picker in a UI, or selecting a target tenant before ingestion or search.

The response separates **active/provisioning** tenants (in `tenant_ids`) from tenants whose provisioning **failed** (in `failed_tenant_ids`). Re-create a failed tenant to retry provisioning.

<RequestExample>
```bash cURL
curl -X GET 'https://api.hydradb.com/tenants' \
  -H "Authorization: Bearer <your_api_key>" \
  -H "API-Version: 2"
```
```typescript TypeScript SDK
const response = await client.tenants.list();
```
```python Python SDK
response = client.tenants.list()
```
</RequestExample>

<Accordion title="HTTP request details">

- **Auth:** Bearer token + `API-Version: 2`
- **Idempotency:** Read-only
- **Async:** No

</Accordion>

## Response

<ResponseExample>

```json Response
{
  "tenant_ids": ["acme_corp", "my_first_tenant"],
  "failed_tenant_ids": null,
  "message": "Successfully retrieved tenant IDs"
}
```

</ResponseExample>

If provisioning failed for a tenant, `failed_tenant_ids` contains diagnostic entries:

```json Failed tenant example
{
  "tenant_ids": ["acme_corp"],
  "failed_tenant_ids": [
    {
      "tenant_id": "staging_import",
      "error": "Provisioning failed. Re-create the tenant to retry."
    }
  ],
  "message": "Successfully retrieved tenant IDs"
}
```

| Field | Description |
|---|---|
| <Field name="tenant_ids" /> | Array of string IDs of active or provisioning tenants. |
| <Field name="failed_tenant_ids" /> | Array of `{ tenant_id, error }` entries, or `null` when no failures. Re-create the tenant to retry provisioning. |
| <Field name="message" /> | Human-readable status string. |

## Behavior notes

<Info>
**Includes provisioning tenants.** Tenants returned in `tenant_ids` may still be provisioning. Use [Tenant Status](/api-reference/v2/endpoint/tenant-status) to confirm readiness before ingestion.
</Info>

<Info>
**Retry failed tenants.** If a tenant appears in `failed_tenant_ids`, re-create that tenant with `POST /tenants` after addressing the reported issue. Poll status again before ingestion.
</Info>
