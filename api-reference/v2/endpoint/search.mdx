The single retrieval endpoint for everything. Use it any time you need to feed an LLM with grounded context, surface user preferences, or fetch chunks ranked by relevance.

Two independent dimensions control behavior:

- **`source`** picks **what** to search: `"knowledge"` (knowledge), `"memories"` (user-scoped), or `"all"` (both, merged and re-ranked together).
- **`search_by`** picks **how** to match: `"hybrid"` (semantic + BM25, the default), `"text"` (BM25 only  -  pair with `operator`), or `"vector"` (pure dense vectors).

Use `mode: "thinking"` for higher-quality multi-query retrieval; `"fast"` for low-latency single-pass. Turn on `graph_context: true` to surface entity/relation paths alongside chunks. Tune `alpha`, `recency_bias`, and `metadata_filters` to shape ranking and narrow the candidate set.

<RequestExample>

```bash cURL
curl -X POST 'https://api.hydradb.com/search' \
  -H "Authorization: Bearer <your_api_key>" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "acme_corp",
    "sub_tenant_id": "user_alex",
    "query": "What is our refund policy, and how should I explain it to this user?",
    "source": "all",
    "search_by": "hybrid",
    "mode": "thinking",
    "max_results": 10,
    "alpha": "auto",
    "recency_bias": 0.2,
    "graph_context": true,
    "search_forceful_relations": true,
    "metadata_filters": {
      "department": "support",
      "document_metadata": {
        "source": "policy"
      }
    }
  }'
```

## Request body



| Name | Description |
| --- | --- |
| <Field name="tenant_id" type="string" required /> | Owning tenant. |
| <Field name="sub_tenant_id" type="string or null" /> | Sub-tenant scope. Required for per-user memory searches. (default=default sub-tenant) |
| <Field name="query" type="string" required /> | Search terms or natural-language question. Cannot be empty. |
| <Field name="source" type='"knowledge" or "memories" or "all"' /> | What collection to search. `"all"` runs knowledge and memories in parallel and merges by `relevancy_score`. (default=`"knowledge"`) |
| <Field name="search_by" type='"hybrid" or "text" or "vector"' /> | Retrieval method. See [Search methods](#search-methods). (default=`"hybrid"`) |
| <Field name="operator" type='"or" or "and" or "phrase"' /> | BM25 operator for `search_by: "text"`. Ignored for `hybrid` and `vector`. (default=`"or"`) |
| <Field name="mode" type='"fast" or "thinking"' /> | Retrieval pipeline. Applies to `hybrid` and `vector` only; ignored for `text`. (default=`"fast"`) |
| <Field name="max_results" type="integer or null" /> | Maximum chunks to return. Default `10`; maximum `50`. Start with `10`, use `5` for tight prompts, and increase only when reranking downstream. |
| <Field name="alpha" type='float 0.0–1.0 or "auto"' /> | Hybrid weight (`1.0` = pure semantic, `0.0` = pure BM25). Applies to `search_by: "hybrid"` only. (default=`0.8`) |
| <Field name="recency_bias" type="float 0.0–1.0" /> | Boost newer content. (default=`0.0`) |
| <Field name="graph_context" type="boolean" /> | When `true`, includes the entity/relation graph slice in the response under `graph_context`. (default=`false`) |
| <Field name="search_forceful_relations" type="boolean" /> | Pull author-declared related sources into `additional_context`. **Only takes effect when `mode: "thinking"`**  -  silently ignored in `fast` mode. (default=`true`) |
| <Field name="additional_context" type="string or null" /> | Request-time hint to guide retrieval (e.g., "user is on the billing page"). This is different from the response `additional_context` map. (default=`null`) |
| <Field name="metadata_filters" type="object or null" /> | Deterministic narrowing before ranking. See [Filters](#filters). (default=`null`) |

### Source selection

| Value | Searches | Best for |
|---|---|---|
| `"knowledge"` *(default)* | Knowledge documents, files, and app sources | Document Q&A, RAG context. |
| `"memories"` | User memories | Personalization and user preferences. |
| `"all"` | Both, merged in one ranked result set | Personalized answers grounded in both shared and user-specific context. |

### Search methods

| Method | Pipeline | Best for |
|---|---|---|
| `"hybrid"` *(default)* | Dense vectors + BM25 keyword | General-purpose retrieval and RAG. |
| `"text"` | BM25 only | Exact terms, compliance lookups, phrase search. Pair with `operator`. |
| `"vector"` | Dense vector only | Pure semantic similarity. |

### Modes

For `search_by: "hybrid"` and `"vector"`:

| Mode | Behavior | When to use |
|---|---|---|
| `"fast"` *(default)* | Single query pass | Real-time chat, autocomplete, simple lookups. |
| `"thinking"` | Multi-query expansion + reranking + forceful-relation context | Complex queries, customer-facing answers, anything where quality matters. |

### Filters

`metadata_filters` runs before ranking. The shape combines two filter scopes:

```json
{
  "metadata_filters": {
    "department": "engineering",
    "region": "us-east",
    "document_metadata": {
      "source": "account_plan",
      "author": "alex"
    }
  }
}
```

| Where | What it matches |
|---|---|
| **Top-level keys** (`department`, `region`) | The source's `tenant_metadata`. Keys must be declared in the tenant's `tenant_metadata_schema` with `enable_match: true`, otherwise they are silently ignored. |
| **Nested under `document_metadata`** | The source's free-form per-document fields. No schema declaration required. |

## Response

<ResponseExample>

```json Response
{
  "chunks": [
    {
      "chunk_uuid": "policy_main_chunk_3",
      "source_title": "Compliance Policy",
      "source_id": "policy_main",
      "chunk_content": "Refunds are issued within 30 days...",
      "relevancy_score": 0.91,
      "extra_context_ids": ["pref-tone"],
      "additional_metadata": { "author": "Legal Team" }
    }
  ],
  "sources": [
    { "id": "policy_main", "title": "Compliance Policy", "type": "pdf" }
  ],
  "graph_context": {
    "query_paths": [],
    "chunk_relations": [],
    "chunk_id_to_group_ids": {}
  },
  "additional_context": {
    "pref-tone": {
      "source_title": "User preferences",
      "chunk_content": "Prefers concise answers."
    }
  }
}
```

</ResponseExample>

| Field | Description |
|---|---|
| <Field name="chunks[]" type="SearchChunk[]" /> | Array of ranked chunks. Preserve server order in your prompt. |
| <Field name="chunks[].chunk_uuid" type="string" /> | Stable chunk identifier (string)  -  use to dedupe across calls. |
| <Field name="chunks[].source_id" type="string" /> | The source this chunk belongs to (string). |
| <Field name="chunks[].source_title" type="string" /> | Display title of the source (string). |
| <Field name="chunks[].chunk_content" type="string" /> | The actual text snippet (string). |
| <Field name="chunks[].relevancy_score" type="float" /> | Relevance score (float). Higher is more relevant within the same response; do not treat scores as stable probabilities across unrelated queries. |
| <Field name="chunks[].extra_context_ids" type="string[]" /> | Array of keys into `additional_context` for forceful-relation-linked content (`thinking` mode only). |
| <Field name="sources[]" type="SourceModel[]" /> | Deduplicated source metadata corresponding to the returned chunks. |
| <Field name="graph_context" type="GraphContext or null" /> | Graph slice object, populated when `graph_context: true`. See [GraphContext fields](#graphcontext-fields). |
| <Field name="additional_context" type="object" /> | Response-time map of related content from forceful relations (`thinking` mode only). Keys match values in `chunks[].extra_context_ids`, not necessarily chunk UUIDs. |

### `GraphContext` fields

| Field | Description |
|---|---|
| <Field name="query_paths[]" type="array" /> | Entity-traversal paths derived from the query. Each path is a list of `{ source, relation, target }` triplets. |
| <Field name="chunk_relations[]" type="array" /> | Per-chunk relations grouped by `group_id`. Attach to chunks via `chunk_id_to_group_ids`. |
| <Field name="chunk_id_to_group_ids" type="object" /> | Maps each `chunk_uuid` to one or more `group_id`s in `chunk_relations`. |

A zero-result search returns empty arrays/maps rather than an error:

```json Zero results
{
  "chunks": [],
  "sources": [],
  "graph_context": null,
  "additional_context": {}
}
```

For formatting `RetrievalResult` into an LLM prompt, see [How to Use API Results](/essentials/v2/api-results).

## Behavior notes

<Info>
**`graph_context` is on by default.** Set it to `false` if you only need ranked chunks and want to drop the graph slice from the response.
</Info>

<Info>
**`recency_bias` is off by default.** Defaults to `0.0`  -  no recency boost is applied unless you set it.
</Info>

<Warning>
**`search_forceful_relations` requires `mode: "thinking"`.** In `fast` mode the flag is silently ignored. The server does not error or warn  -  your `additional_context` will simply be empty.
</Warning>

<Warning>
**`metadata_filters` top-level keys need a schema.** Top-level keys must be declared in `tenant_metadata_schema` with `enable_match: true`. Undeclared keys are silently ignored. To filter free-form per-document fields, nest them under a `document_metadata` key.
</Warning>

<Warning>
**Common mistakes.** Check [Source Status](/api-reference/v2/endpoint/source-status) for recently ingested documents before searching. If you omit `sub_tenant_id`, HydraDB searches the default sub-tenant; use [List Sub-Tenants](/api-reference/v2/endpoint/list-sub-tenants) to discover available IDs.
</Warning>

<Info>
**Tuning heuristics.**
<ul>
  <li><strong><code>alpha</code></strong>: start at <code>0.8</code>. Lower toward <code>0.3–0.5</code> when the query contains literal tokens (error codes, SKUs, product names). Raise toward <code>0.9</code> for conceptual questions. Use <code>"auto"</code> when query shape varies.</li>
  <li><strong><code>recency_bias</code></strong>: leave at <code>0</code> for static reference material. Set <code>0.2–0.4</code> for mixed content, <code>0.6–0.8</code> for changelogs, news, or status updates.</li>
  <li><strong><code>max_results</code></strong>: start at <code>10</code>. Drop to <code>5</code> for tight context windows; raise to <code>20</code> if you rerank downstream.</li>
</ul>
</Info>
