Browse what's in a tenant or sub-tenant  -  for admin dashboards, audit views, populating a "select source" picker, or fetching a known subset of sources by ID. Pick the bucket with `kind`:

- `kind=knowledge` *(default)*  -  knowledge sources (files, app sources).
- `kind=memories`  -  user memories.

Supports pagination (`page`, `page_size`), structured filters on tenant and document metadata, source-field filters (title, type, url, etc.), and field projection (`include_fields`) for slim payloads.

<RequestExample>
```bash cURL
curl -X POST 'https://api.hydradb.com/source/list' \
  -H "Authorization: Bearer <your_api_key>" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "acme_corp",
    "kind": "knowledge",
    "page": 1,
    "page_size": 50,
    "filters": {
      "tenant_metadata": { "department": "legal" },
      "source_fields": { "type": "slack" }
    },
    "include_fields": ["title", "type", "url", "timestamp"]
  }'
```
```typescript TypeScript SDK

  kind: "knowledge",

```

## Request body



| Name | Type | Description |
|---|---|---|
| <Field name="tenant_id" required /> | string | Owning tenant. |
| <Field name="sub_tenant_id" /> | string \| null | Sub-tenant scope. If omitted, the default sub-tenant is used. (default=`null`) |
| <Field name="kind" /> | `"knowledge" \| "memories"` | Bucket to list. (default=`"knowledge"`) |
| <Field name="source_ids" /> | string[] \| null | When provided and non-empty, only items with these IDs are returned (pagination + filters still apply). (default=`null`) |
| <Field name="page" /> | integer | Page number (1-indexed). (default=`1`) |
| <Field name="page_size" /> | integer | Items per page, from `1` to `100`. (default=`50`) |
| <Field name="filters" /> | object | Structured exact-match filters. See [Filters](#filters). (default=`null`) |
| <Field name="include_fields" /> | string[] \| null | Field projection. Only the listed fields plus `id`, `tenant_id`, `sub_tenant_id` are populated. Only applies to `kind=knowledge`. (default=`null`  -  all fields) |

### Filters

`filters` is a structured object with three optional categories. Filters are exact-match constraints. **All provided pairs within and across categories combine with AND**  -  a document must satisfy every key to remain a candidate.

```json
{
  "filters": {
    "tenant_metadata": { "department": "Finance", "region": "US" },
    "document_metadata": { "author": "Alice Smith" },
    "source_fields": { "type": "slack", "title": "standup notes" }
  }
}
```

| Category | Matched against | Notes |
|---|---|---|
| <Field name="tenant_metadata" /> | Source's `tenant_metadata` payload | Keys must be declared in the tenant's `tenant_metadata_schema` with `enable_match: true`. Undeclared keys are silently ignored. |
| <Field name="document_metadata" /> | Source's `document_metadata` payload | Free-form per-document JSON. No schema declaration required. |
| <Field name="source_fields" /> | Built-in source fields (`type`, `title`, `description`, `url`, `timestamp`) | Use for app-source categories or quick title lookups. |

Filter values are matched against stored values as exact values. There are no range, contains, or OR operators on this endpoint; run multiple calls and merge client-side for OR behavior.

### `include_fields`

When you don't need every field on every row, pass `include_fields` to keep response payloads small. Only the listed fields are populated; omitted fields should be treated as unavailable in that response. `id`, `tenant_id`, and `sub_tenant_id` are always returned.

Allowed values are `title`, `type`, `description`, `url`, `timestamp`, `tenant_metadata`, `document_metadata`, and `chunk_count`. Omit or pass `null` to return everything.

`include_fields` only applies to `kind=knowledge`. It is ignored for `kind=memories`.

## Behavior notes

<Info>
**Pagination is mandatory in spirit.** Even without filters, `total` may be large. Always prefer `page` / `page_size` over fetching everything in one call.
</Info>

<Info>
**Filters AND.** All filter pairs combine with logical AND. To express OR semantics, run multiple calls and union client-side.
</Info>

<Info>
**`source_ids` + filters.** When `source_ids` is non-empty, only those IDs are considered, but `filters` still applies on top  -  useful for "show me items 1, 2, 3 that also belong to department=legal".
</Info>

<Warning>
**Undeclared filter keys are silent.** Keys in `filters.tenant_metadata` that aren't declared in `tenant_metadata_schema` are silently ignored. If a filter doesn't narrow results as expected, verify the schema field exists and has `enable_match: true`.
</Warning>

<Warning>
**Common mistakes.** `row_count` from [Tenant Stats](/api-reference/v2/endpoint/tenant-stats) counts chunks, not documents. Use this endpoint's `total` when you need distinct source or memory counts.
</Warning>
