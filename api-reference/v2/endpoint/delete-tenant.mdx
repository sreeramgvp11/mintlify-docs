Tearing down a workspace  -  for customer offboarding, end-of-life environments, or cleanup after testing. Deletes the tenant **and all of its ingested data, memories, metadata schema, vector indices, graph, and infrastructure**.

This action is irreversible. There is no soft-delete and no recovery window.

<RequestExample>
```bash cURL
curl -X DELETE 'https://api.hydradb.com/tenants?tenant_id=my_first_tenant' \
  -H "Authorization: Bearer <your_api_key>" \
  -H "API-Version: 2"
```
```typescript TypeScript SDK
const response = await client.tenants.delete({
  tenant_id: "my_first_tenant",
});
```
```python Python SDK
response = client.tenants.delete(tenant_id="my_first_tenant")
```
</RequestExample>

<Accordion title="HTTP request details">

- **Auth:** Bearer token + `API-Version: 2`
- **Idempotency:** Calling on an already-deleted tenant returns `404 TENANT_NOT_FOUND`
- **Async:** Yes  -  the response acknowledges the deletion; cleanup of vector indices, the graph, and storage objects runs in the background.

</Accordion>

<ResponseExample>

| Field | Description |
|---|---|
| <Field name="tenant_id" /> | The tenant that was deregistered. |
| <Field name="status" /> | Always `deletion_scheduled` on success. Background workers continue removing data. |
| <Field name="message" /> | Human-readable hint. |

## Deletion completion

Deletion is asynchronous. Treat deletion as complete when the tenant no longer appears in `GET /tenants`, or when `GET /tenants/status?tenant_id=...` returns `TENANT_NOT_FOUND`.

After deletion completes, the same `tenant_id` can be used in a new `POST /tenants` request. Until then, avoid recreating the tenant or retrying ingestion/search against it.

## Behavior notes

<Warning>
**Irreversible.** Ingested documents, memories, embeddings, graph nodes, metadata schema, and storage objects are removed. There is no recovery window  -  make sure you have a backup if the content matters.
</Warning>

<Info>
**Async cleanup.** The endpoint returns immediately after deregistering the tenant. Vector indices, MongoDB documents, the knowledge graph, and S3/MinIO objects are removed by background workers and may take a few minutes to fully disappear from infrastructure.
</Info>

<Warning>
**Stop in-flight work first.** Stop ingestion, polling, search, and background jobs that target this tenant before deletion. Calls made after deregistration can fail with `TENANT_NOT_FOUND` even while infrastructure cleanup is still running.
</Warning>

<Info>
**Repeat calls.** Deleting an already-deleted tenant returns `404 TENANT_NOT_FOUND`. Deleting a tenant that is still provisioning or already deleting is treated as a request to tear down that tenant; use the deletion-completion checks above before recreating the same ID.
</Info>
