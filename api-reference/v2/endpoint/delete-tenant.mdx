Permanently delete a tenant and all of its associated data, including ingested content, memories, metadata schema, vector indices, and graphs. This action is irreversible. There is no soft-delete and no recovery window.

<RequestExample>
```bash cURL
curl -X DELETE 'https://api.hydradb.com/tenants?tenant_id=my_first_tenant' \
  -H "Authorization: Bearer <your_api_key>" \
  -H "API-Version: 2"
```
```typescript TypeScript SDK
const response = await client.tenants.delete({
  tenant_id: "my_first_tenant",
});
```
```python Python SDK
response = client.tenants.delete(tenant_id="my_first_tenant")
```
</RequestExample>

## Deletion completion

Deletion is asynchronous. Treat deletion as complete when the tenant no longer appears in `GET /tenants`, or when `GET /tenants/status?tenant_id=...` returns `TENANT_NOT_FOUND`.

After deletion completes, the same `tenant_id` can be used in a new `POST /tenants` request. Until then, avoid recreating the tenant or retrying ingestion/search against it.

## Behavior notes

<Warning>
**Irreversible action.** Ingested documents, memories, embeddings, graph nodes, metadata schema, and storage objects are permanently removed. There is no recovery window  -  ensure you have a backup if the content matters.
</Warning>

- **Stop in-flight work first:** Stop all ingestion, polling, search, and background jobs targeting this tenant before deleting. Calls made after deregistration can fail with `TENANT_NOT_FOUND` even while infrastructure cleanup is still running.
- **Async cleanup:** The endpoint returns immediately after deregistering the tenant. Infrastructure cleanup of vector stores, graphs, and storage objects runs in the background and may take a few minutes to complete.
- **Repeat calls:** Deleting an already-deleted tenant returns `404 TENANT_NOT_FOUND`. Deleting a tenant that is still provisioning or deleting is treated as a request to tear down that tenant.
