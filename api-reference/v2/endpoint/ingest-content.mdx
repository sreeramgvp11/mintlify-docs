- **`type=knowledge`** - Build your knowledge base using:
  - **Files** - Use `files` for binary uploads (PDF, DOCX, CSV, MD, TXT) that HydraDB should parse. Read more about [ingesting files](/essentials/v2/knowledge).
  - **App Sources** - Use `app_knowledge` for pre-extracted JSON content (Slack threads, Notion pages, emails, tickets). Read more about [ingesting knowledge from your apps](/essentials/v2/app-sources).
- **`type=memory`** - Use `memories` for per-user content, scoped with `sub_tenant_id`. Set `infer: true` to automatically extract preferences from raw signals, or `infer: false` to store the text verbatim. Read more about [ingesting memories](/essentials/v2/memories).


<RequestExample>

```python Python SDK
import json

with open("/path/to/policy.pdf", "rb") as policy:
    knowledge_result = client.source.ingest(
        # Knowledge requests can include files, app_knowledge, or both.
        type="knowledge",
        tenant_id="acme_corp",
        sub_tenant_id="team_docs",
        upsert=True,

        # Use files when HydraDB should parse PDFs, DOCX, CSV, Markdown, or text files.
        files=[("policy.pdf", policy, "application/pdf")],
        file_metadata=json.dumps([
            {
                "source_id": "policy_main",
                "tenant_metadata": {"department": "legal"},
                "document_metadata": {"source": "policy"},
            }
        ]),

        # Use app_knowledge when your app already extracted the source text/metadata.
        app_knowledge=json.dumps([
            {
                "id": "slack_thread_001",
                "title": "Pricing discussion",
                "type": "slack",
                "content": {"text": "We agreed on three tiers..."},
                "tenant_metadata": {"department": "product"},
                "document_metadata": {"channel": "pricing"},
            }
        ]),
    )

    tenant_id="acme_corp",
    sub_tenant_id="user_alex",
    memories=json.dumps([

            "infer": True,
            "user_name": "Alex",
        }
    ]),
)

    type="memory",

    sub_tenant_id="user_alex",

)
```

## Form fields



| Name | Description |
| --- | --- |
| <Field name="type" type='"knowledge" or "memory"' /> | Use singular `"memory"` when writing memories. (default=`"knowledge"`) |
| <Field name="tenant_id" type="string" required /> | Target tenant. |
| <Field name="sub_tenant_id" type="string" /> | Logical partition inside the tenant. (default=`""`  -  default sub-tenant) |
| <Field name="upsert" type="boolean" /> | Replace existing sources with the same ID. Set to `false` to error on conflict. (default=`true`) |
| <Field name="files" type="file[]" /> | Binary uploads, **knowledge only**. Required when `type=knowledge` and you want HydraDB to parse files. Omit when ingesting memories. (default=`[]`) |
| <Field name="file_metadata" type="string (JSON array)" /> | One entry per file in `files`, in the same order. If omitted, files index with inferred defaults such as filename/title. See the `file_metadata` item shape below. |
| <Field name="app_knowledge" type="string (JSON object or array)" /> | Pre-extracted source objects (Slack, Notion, web pages, etc.), **knowledge only**. See the `app_knowledge` item shape below. |
| <Field name="memories" type="string (JSON array)" /> | Memory items, **memory only**. Required and non-empty when `type=memory`. Use plural `memories` for the form field, even though `type` is singular `memory`. See the `memories` item shape below. |

### `file_metadata` item shape

| Field | Description |
| --- | --- |
| <Field name="source_id" type="string" /> | Optional source ID. If set, becomes the `source_id` for this file (use your app's document ID for parity). |
| <Field name="tenant_metadata" type="object" /> | Tenant-schema fields for filtering and search. Keys must be declared in `tenant_metadata_schema`. (default=`{}`) |
| <Field name="document_metadata" type="object" /> | Free-form per-document fields for display or bookkeeping. To filter on these at search time, nest under `metadata_filters.document_metadata`. (default=`{}`) |
| <Field name="title" type="string" /> | Override the source title shown in search results. (default=filename) |
| <Field name="type" type="string" /> | Override the source type shown in search results. (default=inferred) |
| <Field name="url" type="string" /> | Override the canonical URL. |
| <Field name="timestamp" type="string" /> | ISO-8601 timestamp override. (default=upload time) |
| <Field name="relations" type="object" /> | Declare forceful relations to other sources. Shape: `{ "source_ids": ["...", "..."] }` (legacy alias `cortex_source_ids` still accepted). Surfaced via `additional_context` in `mode: "thinking"` search. |

### `app_knowledge` item shape

| Field | Description |
| --- | --- |
| <Field name="id" type="string" /> | Source ID. Treated as the upsert key. (default=auto-generated) |
| <Field name="title" type="string" /> | Short title or subject shown in search results. |
| <Field name="type" type="string" /> | Source category (`slack`, `notion`, `gmail`, `webpage`, etc.). Used for filtering and display. |
| <Field name="description" type="string" /> | Optional long-form description. |
| <Field name="url" type="string" /> | Canonical URL or reference link. |
| <Field name="timestamp" type="string" /> | ISO-8601 timestamp (creation or last-updated). |
| <Field name="content" type="object" required /> | Content payload. Use `{ "text": "..." }` for plain text. Required for app sources. |
| <Field name="tenant_metadata" type="object" /> | Tenant-schema fields. (default=`{}`) |
| <Field name="document_metadata" type="object" /> | Free-form per-document fields. (default=`{}`) |
| <Field name="attachments" type="array" /> | Optional related attachments. (default=`[]`) |
| <Field name="relations" type="object" /> | Forceful relations, same shape as on `file_metadata`. |

### `memories` item shape

| Field | Description |
| --- | --- |
| <Field name="source_id" type="string" /> | Optional unique ID. Acts as the upsert key. (default=auto-generated) |
| <Field name="title" type="string" /> | Short label for display in `/source/list` and search hits as `source.title`. (default=truncated `text`) |
| <Field name="text" type="string" recommended /> | Raw text or markdown content. Required unless `user_assistant_pairs` is provided. |
| <Field name="user_assistant_pairs" type="array" recommended /> | Conversation pairs `{ user, assistant }`. Required unless `text` is provided. |
| <Field name="is_markdown" type="boolean" /> | Treat `text` as markdown for chunking. (default=`false`) |
| <Field name="infer" type="boolean" /> | When `true`, HydraDB extracts the underlying preference from raw signal. (default=`false`) |
| <Field name="custom_instructions" type="string" /> | Guides extraction when `infer: true`. Ignored when `infer: false`. |
| <Field name="user_name" type="string" /> | The user's name. Feeds inference. (default=`"User"`) |
| <Field name="expiry_time" type="integer" /> | TTL in seconds. Memory stops surfacing after expiry. |
| <Field name="tenant_metadata" type="object" /> | Tenant-schema fields. (default=`{}`) |
| <Field name="document_metadata" type="object" /> | Free-form per-document fields. (default=`{}`) |
| <Field name="relations" type="object" /> | Forceful relations within the Memories store. Shape: `{ "source_ids": ["...", "..."] }` (legacy alias `cortex_source_ids` still accepted). |


<Note>
Sends `multipart/form-data`. Nested JSON fields (`file_metadata`, `app_knowledge`, `memories`) must be sent as **JSON-stringified** form values.
</Note>

## Behavior notes

- **Async indexing.** `202 Accepted` only means HydraDB **queued** the work. Content is not retrievable until `indexing_status` reaches `graph_creation` (partial) or `completed` (full). Always poll [Source Status](/api-reference/v2/endpoint/source-status) before searching.

- **`type=knowledge` + `app_knowledge` and `files` mix.** You can send both in the same request. HydraDB processes each item independently and returns one `result` per source.

- **Multipart, not JSON.** This is one of the few endpoints that uses `multipart/form-data`. Stringify the JSON arrays (`file_metadata`, `app_knowledge`, `memories`) before sending them in the form field.

- **`tenant_metadata` keys must be declared.** Keys not in the tenant's `tenant_metadata_schema` are silently ignored at search time when used in `metadata_filters`. Plan schema fields up front via [Create Tenant](/api-reference/v2/endpoint/create-tenant).

- **Common mistakes.** Use `type: "memory"` for ingestion and deletion, but use `memories` for the form field and for search/list collection selectors. If you omit `sub_tenant_id`, HydraDB writes to the default sub-tenant; list available sub-tenants with [List Sub-Tenants](/api-reference/v2/endpoint/list-sub-tenants).
