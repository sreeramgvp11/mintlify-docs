title: "Delete Source"
description: "Delete knowledge sources or memories by their IDs."
openapi: "DELETE /source"
---

import { Field } from "/snippets/field.jsx";

Remove indexed content  -  for data-retention policies, deleting a specific user's memories on request, pruning stale documents, or cleaning up after a botched ingest. Pick the bucket with `type`:

- `type=knowledge` *(default)*  -  delete knowledge sources.
- `type=memory`  -  delete memories.

Pass one or more IDs in `request.ids`. The body wraps two fields together: `request` (the actual delete payload) and `type` (the bucket).

<RequestExample>
```bash cURL
curl -X DELETE 'https://api.hydradb.com/source' \
  -H "Authorization: Bearer <your_api_key>" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d '{
    "request": {
      "tenant_id": "acme_corp",
      "sub_tenant_id": "team_docs",
      "ids": ["policy_main", "runbook_deploy"]
    },
    "type": "knowledge"
  }'
```
```typescript TypeScript SDK
await client.source.delete({
  type: "knowledge",
  request: {
    tenant_id: "acme_corp",
    sub_tenant_id: "team_docs",
    ids: ["policy_main", "runbook_deploy"],
  },
});
```
```python Python SDK
client.source.delete(
    type="knowledge",
    request={
        "tenant_id": "acme_corp",
        "sub_tenant_id": "team_docs",
        "ids": ["policy_main", "runbook_deploy"],
    },
)
```
</RequestExample>

### Memory delete example

Use `type: "memory"` and pass memory `source_id`s in the same `request.ids` array. Include the same `sub_tenant_id` you used when ingesting the memories; omitting it targets the default sub-tenant.

```json
{
  "request": {
    "tenant_id": "acme_corp",
    "sub_tenant_id": "user_alex",
    "ids": ["mem_user_alex_tone"]
  },
  "type": "memory"
}
```

<Accordion title="HTTP request details">

- **Auth:** Bearer token + `API-Version: 2`
- **Idempotency:** Deleting an already-deleted ID is a no-op for that ID  -  the rest of the batch still proceeds. `success` reflects whether *any* ID was removed.
- **Async:** Background workers may continue removing vector entries briefly after the response  -  but the source is immediately excluded from search.

</Accordion>

## Request body

The body wraps two fields together. 

| Name | Type | Description |
|---|---|---|
| <Field name="request.tenant_id" required /> | string | Owning tenant. |
| <Field name="request.sub_tenant_id" /> | string \| null | Sub-tenant scope. Use the same value used at ingestion. If omitted, the default sub-tenant is used. (default=`null`) |
| <Field name="request.ids" required /> | string[] | One or more source IDs to delete. At least one is required. |
| <Field name="type" /> | `"knowledge" \| "memory"` | Bucket selector. (default=`"knowledge"`) |

## Response

### Knowledge (`type=knowledge`)

<ResponseExample>

```json Response
{
  "success": true,
  "message": "Delete completed",
  "results": [
    { "id": "policy_main", "deleted": true, "error": null },
    { "id": "runbook_deploy", "deleted": true, "error": null }
  ],
  "deleted_count": 2
}
```

</ResponseExample>

| Field | Description |
|---|---|
| <Field name="success" /> | `true` if at least one ID was deleted. |
| <Field name="results[].id" /> | The ID being reported on. |
| <Field name="results[].deleted" /> | Whether deletion succeeded for this specific ID. |
| <Field name="results[].error" /> | Error message if deletion failed (e.g., not found). |
| <Field name="deleted_count" /> | Total IDs successfully deleted. |

### Memory (`type=memory`)

<ResponseExample>

```json Response
{
  "success": true,
  "user_memory_deleted": true
}
```

</ResponseExample>

| Field | Description |
|---|---|
| <Field name="success" /> | `true` if all listed memories were deleted. |
| <Field name="user_memory_deleted" /> | `true` only when every listed memory was deleted. `false` if any deletion failed. |

## Behavior notes

<Warning>
**Irreversible.** Chunks, embeddings, and graph nodes are removed. There is no soft-delete or undo. Keep a copy if you need to re-ingest.
</Warning>

<Info>
**Partial-success semantics.** For `type=knowledge`, each ID is reported independently in `results[]`. A failure on one ID does not stop the rest. For `type=memory`, the response reports an aggregate `user_memory_deleted` reflecting *all* listed IDs.
</Info>

<Info>
**Search drops the source immediately.** Even before background cleanup finishes, deleted IDs disappear from `/search` and `/source/list` responses.
</Info>

<Info>
**Mixed deletes need two calls.** To delete both knowledge and memory items, send two requests  -  one with `type=knowledge`, one with `type=memory`. A single call cannot mix the two buckets.
</Info>
