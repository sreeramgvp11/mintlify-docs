The SDKs wrap every endpoint in the [API Reference](/api-reference/v2) with typed methods and IDE autocomplete. They automatically set the `API-Version: 2` header on every request  -  you do not need to send it manually.

## Installation

The SDKs ship as major-version bumps of the same packages used for v1. Pin to `>=2.0.0` to use the latest method surface.

<CodeGroup>
```bash Python SDK
pip install "hydradb-sdk>=2,<3"
```
```bash TypeScript / Node.js
npm install @hydradb/sdk@^2
```
</CodeGroup>

## Client setup

<CodeGroup>
```python Python SDK
import os
from hydra_db import HydraDB

client = HydraDB(token=os.environ["HYDRA_DB_API_KEY"])
```
```python Python (Async)
import os
from hydra_db import AsyncHydraDB

async_client = AsyncHydraDB(token=os.environ["HYDRA_DB_API_KEY"])
```
```typescript TypeScript SDK
import { HydraDBClient } from "@hydradb/sdk";

const client = new HydraDBClient({
  token: process.env.HYDRA_DB_API_KEY,
});
```
</CodeGroup>

<Info>
**Python:** Both synchronous (`HydraDB`) and asynchronous (`AsyncHydraDB`) clients are available. They share an identical surface  -  choose based on your application's concurrency model.
</Info>

## Versioning

The SDKs automatically include `API-Version: 2` on every outbound request. Server-side routing resolves to the matching routes; the response includes an `X-API-Version: 2` header echoing the resolved version.

You can verify which version your client is using by inspecting the response headers in any SDK call.

## Naming conventions

The REST API uses **snake_case** for all request and response fields. The SDKs adapt this to native idioms:

| | Method naming | Parameter field naming | Example |
|---|---|---|---|
| **Raw HTTP / cURL** |  -  | snake_case | `tenant_id`, `sub_tenant_id` |
| **Python SDK** | snake_case | snake_case | method: `query()`, params: `tenant_id`, `query_by` |
| **TypeScript SDK** | camelCase | snake_case | method: `query()`, params: `tenant_id`, `query_by` |

<Info>
**TypeScript SDK uses snake_case for all parameter field names.** Only method names are camelCase (e.g., `client.source.list({tenant_id: "...", page_size: 50})`). This applies to every request object passed to an SDK method.
</Info>

The TypeScript SDK transparently maps any camelCase parameters you write to snake_case on the wire. Your IDE's autocomplete and type checking work directly off the API contract  -  if a field is optional in the API, it's optional in the SDK.

## SDK method structure

SDK methods are grouped under three top-level namespaces  -  one per `/api-reference/v2` group:

| URL prefix | SDK group | Purpose |
|---|---|---|
| `/source/*` and `/source` | `client.source` | Ingest, status, fetch, list, delete, graph relations |
| `/query` | `client.query` | Unified retrieval over knowledge, memories, or both |
| `/tenants/*` and `/tenants` | `client.tenants` | Create, list, delete, status, sub-tenants, stats |


## Method reference

### Sources

`client.source.*` covers every flow around content lifecycle  -  file uploads, app sources, memories, polling, fetching, listing, deletion, and graph inspection.

| Method | Endpoint |
|---|---|
| `client.source.ingest()` | `POST /source/ingest` |
| `client.source.status()` | `GET /source/status` |
| `client.source.fetch()` | `GET /source/fetch` |
| `client.source.list()` | `POST /source/list` |
| `client.source.delete()` | `DELETE /source` |
| `client.source.relations()` | `GET /source/relations` |

### Query

A single method covers all retrieval. Pick `type` (`"knowledge"`, `"memory"`, `"all"`) and `query_by` (`"hybrid"`, `"text"`) to control behavior.

| Method | Endpoint |
|---|---|
| `client.query()` | `POST /query` |

### Tenants

| Method | Endpoint |
|---|---|
| `client.tenants.create()` | `POST /tenants` |
| `client.tenants.list()` | `GET /tenants` |
| `client.tenants.delete()` | `DELETE /tenants` |
| `client.tenants.status()` | `GET /tenants/status` |
| `client.tenants.sub_tenants()` | `GET /tenants/sub-tenants` |
| `client.tenants.stats()` | `GET /tenants/stats` |

In Python, all method names use snake_case: `client.tenants.sub_tenants()`. In TypeScript, snake_case method names use camelCase: `client.tenants.subTenants()`.

## Migrating from v1

The v2 SDKs are a deliberate consolidation. A few common v1 → v2 method swaps:

| v1 | v2 |
|---|---|
| `client.upload.knowledge(...)` | `client.source.ingest(type="knowledge", ...)` |
| `client.upload.addMemory(...)` | `client.source.ingest(type="memory", ...)` |
| `client.upload.verifyProcessing(...)` | `client.source.status(...)` |
| `client.search.fullRecall(...)` | `client.query(type="knowledge", ...)` |
| `client.search.recallPreferences(...)` | `client.query(type="memory", ...)` |
| `client.search.booleanRecall(...)` | `client.query(query_by="text", ...)` |
| `client.fetch.listData(...)` | `client.source.list(...)` |
| `client.fetch.graphRelationsBySourceId(...)` | `client.source.relations(...)` |
| `client.fetch.content(...)` | `client.source.fetch(...)` |
| `client.data.delete(...)` | `client.source.delete(type="knowledge", ...)` |
| `client.upload.deleteMemory(...)` | `client.source.delete(type="memory", ...)` |
| `client.tenant.create(...)` | `client.tenants.create(...)` |
| `client.tenant.getTenantIds(...)` | `client.tenants.list(...)` |
| `client.tenant.deleteTenant(...)` | `client.tenants.delete(...)` |
| `client.tenant.getInfraStatus(...)` | `client.tenants.status(...)` |
| `client.tenant.getSubTenantIds(...)` | `client.tenants.sub_tenants(...)` |
| `client.tenant.monitor(...)` | `client.tenants.stats(...)` |

The v1 SDK methods remain available on the `<2.0.0` releases of `hydradb-sdk` / `@hydradb/sdk` for as long as v1 routes are supported.

## Getting started

### Create a tenant

A tenant is an isolated workspace. Most organizations create one tenant total, with sub-tenants for users or teams. See [Multi-Tenant](/essentials/v2/multi-tenant) for the full model.

<CodeGroup>
```python Python SDK
response = client.tenants.create(
    tenant_id="my_first_tenant",
    tenant_metadata_schema=[
        {"name": "department", "data_type": "string", "enable_match": True},
    ],
)
```
```python Python (Async)
response = await async_client.tenants.create(tenant_id="my_first_tenant")
```
```typescript TypeScript SDK
const response = await client.tenants.create({
  tenant_id: "my_first_tenant",
  tenant_metadata_schema: [
    { name: "department", data_type: "string", enable_match: true },
  ],
});
```
</CodeGroup>

Tenant creation is asynchronous. Poll [`tenants.status`](/api-reference/v2/endpoint/tenant-status) until provisioning completes:

<CodeGroup>
```python Python SDK
import time

while True:
    status = client.tenants.status(tenant_id="my_first_tenant")
    if (
        status.infra.scheduler_status
        and status.infra.graph_status
        and status.infra.vectorstore_status.knowledge
        and status.infra.vectorstore_status.memory
    ):
        break
    time.sleep(2)
```
```typescript TypeScript SDK
while (true) {
  const status = await client.tenants.status({
    tenant_id: "my_first_tenant",
  });
  const { scheduler_status, graph_status, vectorstore_status } = status.infra;
  if (
    scheduler_status &&
    graph_status &&
    vectorstore_status.knowledge &&
    vectorstore_status.memory
  ) break;
  await new Promise((r) => setTimeout(r, 2000));
}
```
</CodeGroup>

### Ingest knowledge

Upload documents, app sources, or both in one call:

<CodeGroup>
```python Python SDK
import json

with open("contract.pdf", "rb") as f1, open("policy.pdf", "rb") as f2:
    result = client.source.ingest(
        type="knowledge",
        tenant_id="my_first_tenant",
        files=[
            ("contract.pdf", f1, "application/pdf"),
            ("policy.pdf", f2, "application/pdf"),
        ],
        file_metadata=json.dumps([
            {
                "source_id": "contract_q4",
                "tenant_metadata": {"department": "legal"},
                "document_metadata": {"author": "Alice"},
            },
            {
                "source_id": "policy_2025",
                "tenant_metadata": {"department": "legal"},
                "document_metadata": {"author": "Bob"},
            },
        ]),
    )
```
```typescript TypeScript SDK
const result = await client.source.ingest({
  type: "knowledge",
  tenant_id: "my_first_tenant",
  files: [
    { path: "contract.pdf", filename: "contract.pdf", contentType: "application/pdf" },
    { path: "policy.pdf", filename: "policy.pdf", contentType: "application/pdf" },
  ],
  file_metadata: JSON.stringify([
    {
      source_id: "contract_q4",
      tenant_metadata: { department: "legal" },
      document_metadata: { author: "Alice" },
    },
    {
      source_id: "policy_2025",
      tenant_metadata: { department: "legal" },
      document_metadata: { author: "Bob" },
    },
  ]),
});
```
</CodeGroup>

For app sources (Slack, Notion, Gmail, webpages), pass `app_knowledge` instead of `files`. You can combine both in a single request.

### Add user memories

Use the same `ingest` method with `type: "memory"`:

<CodeGroup>
```python Python SDK
import json

result = client.source.ingest(
    type="memory",
    tenant_id="my_first_tenant",
    sub_tenant_id="user_alex",
    memories=json.dumps([
        {
            "text": "User prefers detailed technical explanations and dark mode",
            "infer": True,
            "user_name": "Alex",
            "tenant_metadata": {"team": "engineering"},
            "document_metadata": {"source": "onboarding"},
        }
    ]),
)
```
```typescript TypeScript SDK
const result = await client.source.ingest({
  type: "memory",
  tenant_id: "my_first_tenant",
  sub_tenant_id: "user_alex",
  memories: JSON.stringify([
    {
      text: "User prefers detailed technical explanations and dark mode",
      infer: true,
      user_name: "Alex",
      tenant_metadata: { team: "engineering" },
      document_metadata: { source: "onboarding" },
    },
  ]),
});
```
</CodeGroup>

<Info>
**`infer` defaults to `false`.** Set `infer: true` for conversational content where you want HydraDB to extract implicit preferences. Use `infer: false` for content that should be stored verbatim.
</Info>

### Query

A single method covers every retrieval pattern  -  switch `type` and `query_by` to control behavior:

<CodeGroup>
```python Python SDK
# Knowledge over documents
knowledge = client.query(
    tenant_id="my_first_tenant",
    query="What are the pricing tiers?",
    type="knowledge",
    query_by="hybrid",
    mode="thinking",
    graph_context=True,
)

# Personalized — searches knowledge AND user memories together
personalized = client.query(
    tenant_id="my_first_tenant",
    sub_tenant_id="user_alex",
    query="What are the pricing tiers, and how should I explain them to this user?",
    type="all",
    query_by="hybrid",
    mode="thinking",
)

# Exact-phrase BM25 search
exact = client.query(
    tenant_id="my_first_tenant",
    query="ERROR_429 rate limit",
    type="knowledge",
    query_by="text",
    operator="phrase",
)
```
```typescript TypeScript SDK
// Knowledge over documents
const knowledge = await client.query({
  tenant_id: "my_first_tenant",
  query: "What are the pricing tiers?",
  type: "knowledge",
  query_by: "hybrid",
  mode: "thinking",
  graph_context: true,
});

// Personalized — searches knowledge AND user memories together
const personalized = await client.query({
  tenant_id: "my_first_tenant",
  sub_tenant_id: "user_alex",
  query: "What are the pricing tiers, and how should I explain them to this user?",
  type: "all",
  query_by: "hybrid",
  mode: "thinking",

// Exact-phrase BM25 search

```

## Response envelope

All responses are wrapped in a consistent envelope:

```json
{
  "success": true,
  "data": {
    "message": "Success"
  },
  "error": null,
  "meta": {
    "request_id": "<uuid>",
    "latency_ms": 12.3
  }
}
```

The SDKs unwrap the envelope automatically and return the inner `data` object. Errors raise a typed exception with the envelope's `error.code` and `error.message` accessible on the exception.

## Type safety

<Check>
Both SDKs are fully typed:

- **Autocomplete** for all method names and parameters
- **Type checking** for request and response objects
- **Inline documentation** for each parameter, sourced from the OpenAPI spec
- **Compile-time validation** for required vs optional fields
- **Enum-typed values** for `type`, `query_by`, `operator` and `mode`

The SDKs provide exact type parity with the API specification:

- **Request parameters**  -  every field documented in the API reference is reflected in method signatures
- **Response objects**  -  return types match the JSON schema for each endpoint
- **Error types**  -  exception structures mirror error response formats
- **Nested objects**  -  complex parameters and responses keep their full structure

## Error handling

Both SDKs throw exceptions for non-2xx responses. Error objects expose the envelope's `error` payload:

<CodeGroup>
```python Python SDK
from hydra_db import HydraDBError

try:
    result = client.query(
        tenant_id="my_first_tenant",
        query="...",
        type="knowledge",
    )
except HydraDBError as e:
    code = e.error_code
    status = e.status_code

    if code == "TENANT_NOT_FOUND":
        # Handle missing tenant
        pass
    elif status == 429:
        # Rate limit — retry with backoff
        pass
    else:
        raise
```
```typescript TypeScript SDK
import { HydraDBError } from "@hydradb/sdk";

try {
  const result = await client.query({
    tenant_id: "my_first_tenant",
    query: "...",
    type: "knowledge",
  });
} catch (error) {
  if (error instanceof HydraDBError) {
    const code = error.body?.error?.code;

    if (code === "TENANT_NOT_FOUND") {
      // Handle missing tenant
    } else if (error.statusCode === 429) {
      // Rate limit — retry with backoff
    } else {
      throw error;
    }
  } else {
    throw error;
  }
}
```
</CodeGroup>

For the full list of error codes and retry patterns, see [Error Responses](/api-reference/v2/error-responses).

## IDE-driven discovery

Whether you're using TypeScript, Python, VS Code, PyCharm, or any modern IDE, the workflow is the same:

1. Type the method name → see all available methods
2. Open the parentheses → see all required and optional parameters
3. Press `Cmd+Space` (macOS) or `Ctrl+Space` (Windows/Linux) → get inline documentation

This works because the SDKs are fully typed with comprehensive parameter docs sourced from the OpenAPI spec.
