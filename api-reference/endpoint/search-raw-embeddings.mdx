## When to use it

Use this endpoint to perform pure vector similarity search against an embeddings tenant. You supply the query embedding (generated by the same model you used at insertion time), and HydraDB returns the nearest chunks.

This endpoint requires an **embeddings tenant**.

## Endpoint

- **Auth:** Bearer token
- **Idempotency:** Read-only
- **Async:** No

## Example

<CodeGroup>
```bash cURL
curl -X POST 'https://api.hydradb.com/embeddings/search_raw_embeddings' \
  -H "Authorization: Bearer <your_api_key>" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "my_embeddings_tenant",
    "sub_tenant_id": "default",
    "query_embedding": [0.123, -0.045, 0.789, 0.234, -0.056],
    "limit": 10,
    "filter_expr": "metadata[\"category\"] == \"engineering\""
  }'
```

## Request parameters

| Name | Type | Required | Default | Description |
|---|---|---|---|---|
| `tenant_id` | string | Yes | – | The embeddings tenant. |
| `sub_tenant_id` | string | Yes | – | Sub-tenant scope. |
| `query_embedding` | number[] | Yes | – | Query vector. Length must match `embeddings_dimension`. |
| `limit` | integer | No | `10` | Maximum results (1–1000). |
| `filter_expr` | string | No | – | Milvus filter expression for additional filtering. |
| `output_fields` | string[] | No | – | Fields to include in results. Defaults to `chunk_id`, `source_id`, `metadata`. |

## Filter expressions

`filter_expr` uses Milvus expression syntax to filter by metadata fields:

| Pattern | Example |
|---|---|
| Equality | `metadata["category"] == "engineering"` |
| Inequality | `metadata["status"] != "draft"` |
| Comparison | `metadata["year"] >= 2024` |
| Logical | `metadata["category"] == "legal" and metadata["region"] == "us"` |
| Membership | `metadata["tag"] in ["urgent", "high-priority"]` |

For full syntax, see the Milvus documentation.

## Response

```json
[
  {
    "source_id": "doc_001",
    "embedding": {
      "chunk_id": "doc_001_chunk_0",
      "embedding": [0.123, -0.045, 0.789]
    },
    "score": 0.92,
    "distance": 0.08,
    "metadata": { "category": "engineering" }
  },
  {
    "source_id": "doc_002",
    "embedding": {
      "chunk_id": "doc_002_chunk_3",
      "embedding": [0.135, -0.038, 0.801]
    },
    "score": 0.87,
    "distance": 0.13,
    "metadata": { "category": "engineering" }
  }
]
```

| Field | Description |
|---|---|
| `source_id` | The source the chunk belongs to. |
| `embedding.chunk_id` | The chunk's unique identifier. |
| `embedding.embedding` | The stored vector (only when `embedding` is in `output_fields`). |
| `score` | Similarity score. Higher = more similar. |
| `distance` | Vector distance. Lower = more similar. |
| `metadata` | Source-level metadata. |

## Behavior notes

<Info>
**Query embedding must match dimension.** The `query_embedding` length must equal the tenant's `embeddings_dimension`. Mismatched lengths return `422 VALIDATION_ERROR`.
</Info>

<Info>
**Same model in, same model out.** For meaningful results, the query embedding should come from the same model used to generate the stored vectors. Mixing models produces noise.
</Info>
