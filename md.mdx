## 1. Critical Rules for LLMs

If you read nothing else, read this section.

1. **Always poll status endpoints** -- Tenant creation and ingestion are asynchronous. Never write or read immediately after `create`.
2. **Vectorstore index mapping** -- Index `[0]` = Memories (use `recall_preferences`); index `[1]` = Knowledge (use `full_recall`).
3. **Endpoint selection** -- Use `recall_preferences` for user state, `full_recall` for documents. Mixing them returns nothing.
4. **Scope consistency** -- Write and read must use the *same* `sub_tenant_id`. Mismatch returns empty results, not an error.
5. **Schema immutability** -- Tenant metadata schema cannot be changed after creation. Plan up-front.
6. **Verify versions** -- Don't assume SDK version numbers; check PyPI / npm before pinning.
7. **`boolean_recall` is special** -- Do NOT send `alpha`, `mode`, or `graph_context` to it. It will 400.
8. **`app_knowledge` must be a string** -- When sent via multipart form, JSON-stringify it. Sending a raw object yields 422.
9. **`file_metadata` array length MUST equal `files` count** -- otherwise 422.
10. **Deletions are per-item** -- Always inspect `results[].deleted` per item rather than trusting a top-level `success`.

## 3. Core Primitives

### 3.1 Tenants & Sub-Tenants

**Tenant** -- Top-level isolated workspace. No cross-tenant data access. `tenant_id` required on every API call.

**Sub-Tenant** -- Logical partition within a tenant (user, team, workspace, project). Pass `sub_tenant_id` to scope operations. If omitted, defaults to the tenant's default sub-tenant.

**Design patterns:**

| Pattern | Configuration |
|---|---|
| B2B SaaS | One tenant per customer; sub-tenants for teams / workspaces / users |
| B2C app | One tenant for the app; sub-tenants per end user |
| Internal tools | One tenant per company; sub-tenants per department / project |
| Separate environments | Different `tenant_id` per env (prod / staging / dev) -- **NEVER** share across envs |

### 3.2 Memories

User-scoped context -- preferences, conversation history, inferred traits. Each memory is scoped to a `sub_tenant_id`.

- **Mutability:** Dynamic -- evolves with every interaction
- **Recall endpoint:** `POST /recall/recall_preferences`
- **Vectorstore index:** `[0]`
- **Inference:** Set `infer: true` to extract implicit preferences from raw input
- **Upsert:** Set `upsert: true` to update an existing memory by text match

### 3.3 Knowledge

Shared, tenant-wide context from documents and app sources.

- **Sources:** PDF, DOCX, PPTX, Markdown, CSV, TXT, HTML, Slack, Notion, Gmail, Linear, GitHub
- **Processing:** Parsed → Chunked → Embedded → Graph-linked at write time
- **Mutability:** Versioned -- replaced or deleted explicitly
- **Recall endpoint:** `POST /recall/full_recall`
- **Vectorstore index:** `[1]`

### 3.4 Recall Pipeline

Multi-stage retrieval -- not a simple vector similarity search:

1. **Metadata filtering** -- Deterministic narrowing by tenant schema fields
2. **Hybrid retrieval** -- Semantic (vector) + keyword (BM25)
3. **Graph traversal** -- Follow entity relationships
4. **Personalized ranking** -- Recency, frequency, relevance scoring

### 3.5 Metadata

| Field | Purpose | Filterable at recall? | Mutable? |
|---|---|---|---|
| `metadata` | Schema-bound fields declared at tenant creation | Yes (via `metadata_filters`) | Per-document, but schema is immutable |
| `additional_metadata` | Free-form per-document fields | **No** | Yes |

### 3.6 Hybrid Search (`alpha` parameter)

Controls the semantic-vs-keyword balance on `full_recall`:

| `alpha` | Behavior | Best for |
|---|---|---|
| `0.0` | Pure keyword (BM25) | Exact term matching |
| `0.5` | Balanced | General purpose |
| `0.8` | Mostly semantic (recommended default) | Conceptual queries |
| `1.0` | Pure semantic | Abstract queries |
| `"auto"` | Backend chooses | When unsure |

### 3.7 Recall Modes

| Mode | Behavior | Latency | Best for |
|---|---|---|---|
| `fast` | Single semantic query | Lower | Simple lookups, real-time UIs |
| `thinking` | Multi-query + graph traversal + forceful relations | Higher | Complex / multi-hop queries |

## 4. Installation

### 4.1 Prerequisites

1. An API key -- get one from [app.hydradb.com/keys](https://app.hydradb.com/keys)
2. A tenant ID -- create or copy from [app.hydradb.com/tenants](https://app.hydradb.com/tenants)

No account? Email `founders@hydradb.com` for onboarding.

All API requests require:

```
Authorization: Bearer <your_api_key>
```

### 4.2 Python SDK

**Requires Python 3.10+.**

```bash
pip install hydradb-sdk
```

<Note>
PyPI package name is `hydradb-sdk`; the importable module is `hydra_db` (with an underscore). This split is intentional -- install with the hyphenated name, import with the underscored name.
</Note>

**Initialize:**

```python
import os
from hydra_db import HydraDB  # PyPI package: hydradb-sdk; importable module: hydra_db

client = HydraDB(
    token=os.environ["HYDRA_DB_API_KEY"],
    base_url="https://api.hydradb.com",  # optional
)
```

**Async client:**

```python
import asyncio, os
from hydra_db import AsyncHydraDB

async def main():
    client = AsyncHydraDB(token=os.environ["HYDRA_DB_API_KEY"])
    results = await client.recall.full_recall(
        tenant_id="acme_corp",
        query="deployment procedures",
        max_results=10,
        alpha="auto",
    )
    print(results)

asyncio.run(main())
```

```bash
npm install @hydradb/sdk
```

## 5. API Reference

All endpoints are HTTP `POST` / `GET` / `DELETE` against `https://api.hydradb.com`, JSON-bodied unless noted, and require `Authorization: Bearer <api_key>`.

### 5.1 Tenants

#### `POST /tenants/create`

Create an isolated workspace. **Async** -- poll `/tenants/infra/status` before ingesting.

**Request:**

```json
{
  "tenant_id": "acme_corp",
  "tenant_metadata_schema": [
    {
      "name": "category",
      "data_type": "VARCHAR",
      "enable_match": true,
      "enable_dense_embedding": false,
      "enable_sparse_embedding": false
    },
    {
      "name": "timestamp",
      "data_type": "VARCHAR",
      "enable_match": true,
      "enable_dense_embedding": false,
      "enable_sparse_embedding": false
    }
  ]
}
```

**Optional fields for BYO embeddings:**

- `is_embeddings_tenant: true` -- Skip HydraDB's embedding step
- `embeddings_dimension: 1536` -- Required when `is_embeddings_tenant` is true

**Response:**

```json
{
  "status": "accepted",
  "tenant_id": "acme_corp",
  "message": "Tenant creation started in the background..."
}
```

#### `GET /tenants/infra/status`

```
GET /tenants/infra/status?tenant_id=acme_corp
```

**Response:**

```json
{
  "tenant_id": "acme_corp",
  "graph_status": true,
  "vectorstore_status": [true, true],
  "infra_status": "ready"
}
```

Wait for **all** of:

- `graph_status: true`
- `vectorstore_status: [true, true]` (both indices ready)
- `infra_status: "ready"`

#### `GET /tenants/monitor`

```
GET /tenants/monitor?tenant_id=acme_corp
```

```json

  "embedding_dimension": 1024,
  "graph_nodes": 8934,

```

#### `GET /tenants/sub_tenant_ids`

```
GET /tenants/sub_tenant_ids?tenant_id=acme_corp
```

#### `GET /tenants/tenant_ids`

```
GET /tenants/tenant_ids
```

#### `DELETE /tenants/delete`

```
DELETE /tenants/delete?tenant_id=acme_corp
```

### 5.2 Ingestion

#### `POST /ingestion/upload_knowledge`

Multipart form. Two input shapes:

**Files:**

```bash
curl -X POST https://api.hydradb.com/ingestion/upload_knowledge \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -F "tenant_id=acme_corp" \
  -F "sub_tenant_id=team_eng" \
  -F "files=@/path/to/doc.pdf" \
  -F "files=@/path/to/report.docx" \
  -F 'file_metadata=[
    {"file_id": "doc_001", "metadata": {"category": "legal", "timestamp": "2024-01-15"}},
    {"file_id": "doc_002", "metadata": {"category": "finance", "timestamp": "2024-02-20"}}
  ]'
```

**App sources (Slack, Notion, etc.):**

```bash
curl -X POST https://api.hydradb.com/ingestion/upload_knowledge \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -F "tenant_id=acme_corp" \
  -F 'app_knowledge=[
    {
      "source_id": "slack_001",
      "source_name": "Slack: #engineering",
      "knowledge_items": [
        {
          "text": "Deployment completed successfully for v2.3.1",
          "metadata": {"category": "deployment", "timestamp": "2024-03-10"},
          "additional_metadata": {
            "channel": "engineering",
            "author": "alice@acme.com",
            "thread_ts": "1710086400"
          }
        }
      ]
    }
  ]'
```

**Hard rules:**

- `file_metadata` array length **must equal** `files` count
- `app_knowledge` **must** be a JSON-encoded string when sent via multipart form
- All `metadata` keys must be declared in the tenant schema with `enable_match: true`
- `additional_metadata` is stored but **not** filterable at recall

**Response:**

```json
{
  "status": "success",
  "message": "Processing started",
  "source_ids": ["doc_001", "doc_002"]
}
```

#### `POST /ingestion/verify_processing`

```json
{
  "tenant_id": "acme_corp",
  "file_ids": ["doc_001", "doc_002"]
}
```

**Response:**

```json
{
  "results": [
    {"file_id": "doc_001", "indexing_status": "completed", "chunks_created": 47},
    {"file_id": "doc_002", "indexing_status": "processing", "chunks_created": 0}
  ]
}
```

**Status values:**

- `queued` -- waiting
- `processing` -- parsing / chunking / embedding
- `graph_creation` -- vectors done, graph still being built
- `completed` / `success` -- ready for recall
- `errored` -- failed (fail-fast in agent loops!)

Typical indexing time: 30–90 s per document.

### 5.3 Memories

#### `POST /memories/add_memory`

```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "user_alice",
  "memories": [
    {
      "text": "Prefers dark mode",
      "infer": true,
      "user_name": "Alice",
      "metadata": {"category": "preference", "timestamp": "2024-03-15"}
    },
    {
      "user_assistant_pairs": [
        {"user": "What's my coffee order?", "assistant": "Large oat milk latte with one vanilla shot"}
      ],
      "infer": true,
      "upsert": true
    }
  ]
}
```

**Two content shapes per memory:**

1. **Text-based:** `{"text": "...", "infer": true|false}`
2. **Conversation-based:** `{"user_assistant_pairs": [...], "infer": true|false}`

**Parameters:**

- `infer: true` -- Extract implicit preference (e.g., "I like dark mode" → "User prefers dark mode UI setting")
- `infer: false` -- Store verbatim
- `upsert: true` -- Update existing memory if similar text found
- `upsert: false` -- Always insert

#### `DELETE /memories/delete_memory`

Query-parameter based; **no request body**.

```
DELETE /memories/delete_memory?tenant_id=acme_corp&sub_tenant_id=user_alice&memory_id=mem_001
```

Deletes **one** memory per call. Loop over IDs for bulk.

### 5.4 Recall

#### `POST /recall/full_recall`

Knowledge search (index `[1]`) with hybrid semantic + graph + keyword.

```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "team_eng",
  "query": "What are the Q4 deployment procedures?",
  "max_results": 10,
  "alpha": 0.8,
  "recency_bias": 0.3,
  "mode": "fast",
  "graph_context": true,
  "metadata_filters": {"category": "deployment"}
}
```

**Parameters:**

- `query` (required)
- `max_results` (default `5`)
- `alpha` (default `0.8`, range `0`–`1` or `"auto"`) -- semantic vs keyword balance
- `recency_bias` (default `0.0`, range `0`–`1`) -- boost recent content
- `mode` (default `"fast"`) -- `"fast"` or `"thinking"`
- `graph_context` (default `false`) -- include entity relationships
- `metadata_filters` -- deterministic filtering by schema fields
- `forceful_relations` -- array of entity names to include explicitly (requires `mode: "thinking"`)

**Response (shape):**

```json
{
  "results": [
    {
      "chunk_uuid": "chunk_001",
      "text": "Q4 deployment requires approval from tech lead and security review...",
      "source_id": "doc_deployment_guide",
      "source_name": "Q4 Deployment Guide.pdf",
      "score": 0.94,
      "metadata": {"category": "deployment", "timestamp": "2024-10-01"},
      "additional_metadata": {"author": "alice@acme.com", "version": "2.1"},
      "graph_context": {
        "entities": ["Tech Lead", "Security Review"],
        "relationships": [
          {"source": "Deployment", "target": "Security Review", "type": "REQUIRES"}
        ]
      }
    }
  ]
}
```

#### `POST /recall/recall_preferences`

Memory search (index `[0]`).

```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "user_alice",
  "query": "What are my communication preferences?",
  "max_results": 5,
  "mode": "fast"
}
```

#### `POST /recall/boolean_recall`

Exact phrase or boolean keyword search.

```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "team_eng",
  "query": "mechanical engineer",
  "operator": "phrase",
  "search_mode": "sources",
```

## 10. Common Mistakes Checklist

| Symptom | Root cause | Fix |
|---|---|---|
| Empty recall results | Content not yet indexed | Poll `verify_processing` until `completed` |
| Memories missing from `full_recall` | Wrong endpoint | Use `recall_preferences` for memories |
| `metadata_filters` ignored | Field not declared with `enable_match: true` | Add to `tenant_metadata_schema` *before* ingestion |
| `graph_context` empty | Default is `false` | Explicitly set `graph_context: true` |
| Recency doesn't matter | `recency_bias` defaults to `0` | Set between `0.1` and `1.0` |
| Wrong data surfaces | Mismatched `sub_tenant_id` | Use the same scope on write and read |
| Forceful relations not fetching | Not in `thinking` mode | `mode: "thinking"` |
| 422 on upload | `file_metadata` length mismatch | Match array length to file count |
| Schema field can't be changed | Tenant schema is immutable | Plan up-front; use `additional_metadata` for flex |
| 422 on `app_knowledge` | Sent as object, not JSON-encoded string | `json.dumps(...)` / `JSON.stringify(...)` |
| `boolean_recall` 400 | Sent `alpha`, `mode`, or `graph_context` | Drop those params |
| Re-create tenant fails | Previous deletion still in progress | Use a different `tenant_id` or wait |
| `infer` not running | Default is `false` | Explicitly set `infer: true` |
| Partial deletion unnoticed | Trusted top-level `success` | Inspect `results[].deleted` per item |

## 11. Troubleshooting

### 11.1 Authentication failures (401)

- Header must be exactly `Authorization: Bearer <key>` (space after `Bearer`).
- Verify key in dashboard at `app.hydradb.com`.
- Check read vs read-write permissions.

```bash
curl -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  "https://api.hydradb.com/tenants/monitor?tenant_id=test"
```

### 11.2 Tenant Not Found After Creation (404)

Tenant creation is async. Always poll:

```python
import time
client.tenant.create(tenant_id="acme_corp")

deadline = time.time() + 300  # 5-minute cap
while time.time() < deadline:
    status = client.tenant.get_infra_status(tenant_id="acme_corp")
    if status["graph_status"] and status["vectorstore_status"] == [True, True]:
        break
    time.sleep(2)
else:
    raise TimeoutError("Tenant infra not ready after 5 min")

client.upload.knowledge(...)
```

### 11.3 422 on ingestion

Three common shapes -- all detected client-side:

```python
# WRONG: 2 files, 1 metadata
file_metadata = [{"file_id": "doc_001"}]
# CORRECT: lengths match
file_metadata = [{"file_id": "doc_001"}, {"file_id": "doc_002"}]

# WRONG: app_knowledge as object
app_knowledge = [{"source_id": "slack_001"}]
# CORRECT: JSON-encoded string
app_knowledge = json.dumps([{"source_id": "slack_001"}])

# WRONG: MemoryItem with no content
{"infer": True}
# CORRECT: at least text or user_assistant_pairs
{"text": "User prefers dark mode", "infer": True}
```

### 11.4 Empty recall results

Walk this list in order:

1. **Still processing?** `verify_processing` → wait for `completed`.
2. **Wrong endpoint?** Use `recall_preferences` for memories, `full_recall` for knowledge.
3. **Wrong `sub_tenant_id`?** Match the scope from write time.
4. **Filters too tight?** Re-run without `metadata_filters`.

### 11.5 Rate limiting (429)

- **Use bulk endpoints** -- pass an array to `add_memory`, not a loop.
- **Exponential backoff** -- `tenacity` `wait_exponential(min=2, max=60)`.
- **Cache** -- `lru_cache` for repeated queries (FAQ patterns, user preferences).

### 12.3 Multi-index parallel recall

`asyncio.gather` requires awaitables, so use `AsyncHydraDB` (not the sync `HydraDB` client). The sync client's `recall_preferences()` / `full_recall()` return plain dicts and will raise `TypeError` if passed to `gather`.

```python
import asyncio, os
from hydra_db import AsyncHydraDB

async_client = AsyncHydraDB(token=os.environ["HYDRA_DB_API_KEY"])

async def multi_index_recall(tenant_id, sub_tenant_id, query):
    memories, knowledge = await asyncio.gather(
        async_client.recall.recall_preferences(
            tenant_id=tenant_id, sub_tenant_id=sub_tenant_id,
            query=query, max_results=5,
        ),
        async_client.recall.full_recall(
            tenant_id=tenant_id, query=query, max_results=10,
        ),
    )
    return memories["results"] + knowledge["results"]

# If you must use the sync client, run the two calls in a thread pool instead:
from concurrent.futures import ThreadPoolExecutor

def multi_index_recall_sync(tenant_id, sub_tenant_id, query):
    with ThreadPoolExecutor(max_workers=2) as pool:
        f_mem = pool.submit(
            client.recall.recall_preferences,
            tenant_id=tenant_id, sub_tenant_id=sub_tenant_id,
            query=query, max_results=5,
        )
        f_know = pool.submit(
            client.recall.full_recall,
            tenant_id=tenant_id, query=query, max_results=10,
        )
        return f_mem.result()["results"] + f_know.result()["results"]
```

## 13. Multi-Tenant Design Patterns

### 13.1 B2B SaaS

- One `tenant_id` per customer
- `sub_tenant_id` for teams / workspaces / users
- Shared customer knowledge at tenant root (omit `sub_tenant_id` on write)
- Per-user memories at sub-tenant level

### 13.2 B2C App

- One `tenant_id` for the app
- `sub_tenant_id` per end user
- Shared knowledge (FAQs, help docs) at tenant root

### 13.3 Internal Tools

- One `tenant_id` per company
- `sub_tenant_id` per department / project
- Use metadata for fine-grained filtering

### 13.4 Environment Separation

```
my_app_prod
my_app_staging
my_app_dev
```

**Never share a tenant across environments.**

### 13.5 Scoping Hierarchy

`tenant_id` → `sub_tenant_id` → `metadata_filters`

```python
results = client.recall.full_recall(
    tenant_id="acme_corp",        # customer
    sub_tenant_id="team_mobile",  # team
    query="sprint status",
    metadata_filters={"project": "phoenix"},  # project
)
```

## 14. Performance Guidelines

### 14.1 Latency benchmarks

| Operation | p50 | p95 | p99 |
|---|---|---|---|
| Recall (`fast` mode) | ~80 ms | ~180 ms | ~250 ms |
| Recall (`thinking` mode) | ~150 ms | ~300 ms | ~400 ms |
| `add_memory` | ~50 ms | ~120 ms | -- |
| Ingestion processing | 30–90 s / doc | -- | -- |
| Tenant creation | 10–30 s | -- | -- |

### 14.2 Scaling envelope (per tenant)

- Documents: tested up to 100 K
- Chunks: tested up to 5 M
- Memories: tested up to 500 K per sub-tenant
- Sub-tenants: no hard limit (tested 10 K+)

### 14.3 Rate limits

- Recall: 1,000 req / min / tenant
- Ingestion: 100 files / min / tenant
- Memories: 500 adds / min / sub-tenant

### 14.4 Caching

- Cache repeated FAQ queries, user-preference lookups, metadata-heavy filter combos.
- TTL guidance: user preferences 5–15 min; knowledge search 1–5 min; real-time data -- don't cache.

## 15. Security & Compliance

- **Tenant isolation** -- API-level enforcement, separate graph + vector indices per tenant.
- **At rest** -- AES-256.
- **In transit** -- TLS 1.3.
- **Audit log retention** -- 90 days. Events: tenant create/delete, ingestion, recall, memory ops, auth events.
- **GDPR** -- Right-to-delete supported via `/tenants/delete` and per-source / per-memory deletion.
- **SOC 2 Type II** -- In progress.

```python
client = HydraDB(token=os.environ["HYDRA_DB_API_KEY"])

# 1) Create tenant
client.tenant.create(
    tenant_id="demo_tenant",
    tenant_metadata_schema={"category": "string"},
)

# 2) Wait for infra
deadline = time.time() + 300
while time.time() < deadline:
    s = client.tenant.get_infra_status(tenant_id="demo_tenant")
    ready_graph = s["graph_status"] is True
    ready_vs = isinstance(s["vectorstore_status"], list) \
        and len(s["vectorstore_status"]) > 0 \
        and all(s["vectorstore_status"])
    if ready_graph and ready_vs:
        break
    time.sleep(5)
else:
    raise TimeoutError("Tenant infra not ready after 5 min")

# 3) Ingest
with open("doc.pdf", "rb") as f:
    client.upload.knowledge(
        tenant_id="demo_tenant",
        files=[("doc.pdf", f, "application/pdf")],
        file_metadata=json.dumps([
            {"file_id": "doc_001", "metadata": {"category": "general"}}
        ]),
    )

# 4) Wait for processing
TERMINAL_OK = {"completed", "success"}
TERMINAL_FAIL = {"errored"}

deadline = time.time() + 300
while time.time() < deadline:
    r = client.upload.verify_processing(tenant_id="demo_tenant", file_ids=["doc_001"])
    statuses = {item["indexing_status"] for item in r["results"]}
    if statuses & TERMINAL_FAIL:
        raise RuntimeError(f"Ingestion errored: {r}")
    if statuses.issubset(TERMINAL_OK):
        break
    time.sleep(5)
else:
    raise TimeoutError("Processing did not complete in 5 min")

# 5) Recall
results = client.recall.full_recall(
    tenant_id="demo_tenant",
    query="What is this document about?",
    max_results=5,
    alpha=0.8,
)
print(results)
```

### 16.2 TypeScript

```typescript
import { HydraDBClient } from "@hydradb/sdk";
import fs from "fs";

const client = new HydraDBClient({ token: process.env.HYDRA_DB_API_KEY! });

// 1) Create tenant
await client.tenant.create({
  tenant_id: "demo_tenant",
  tenant_metadata_schema: [
    {
      name: "category",
      data_type: "VARCHAR",
      enable_match: true,
      enable_dense_embedding: false,
      enable_sparse_embedding: false,
    },
  ],
});

// 2) Wait for infra
const sleep = (ms: number) => new Promise((r) => setTimeout(r, ms));
const deadline = Date.now() + 5 * 60_000;

let infraReady = false;
while (Date.now() < deadline) {
  const s = await client.tenant.getInfraStatus({ tenant_id: "demo_tenant" });
  const readyGraph = s.graph_status === true;
  const readyVs = Array.isArray(s.vectorstore_status)
    && s.vectorstore_status.length > 0
    && s.vectorstore_status.every(Boolean);
  if (readyGraph && readyVs) { infraReady = true; break; }
  await sleep(5000);
}
if (!infraReady) {
  throw new Error("Tenant infra not ready after 5 min");
}

// 3) Ingest
await client.upload.knowledge({
  tenant_id: "demo_tenant",
  files: [{ path: "./doc.pdf", filename: "doc.pdf", contentType: "application/pdf" }],
  file_metadata: JSON.stringify([
    { file_id: "doc_001", metadata: { category: "general" } },
  ]),
});

// 4) Wait for processing
const TERMINAL_OK = new Set(["completed", "success"]);
const TERMINAL_FAIL = new Set(["errored"]);
const procDeadline = Date.now() + 5 * 60_000;

let processingComplete = false;
while (Date.now() < procDeadline) {
  const r = await client.upload.verifyProcessing({
    tenant_id: "demo_tenant",
    file_ids: ["doc_001"],
  });
  const statuses = new Set(r.results.map((x) => x.indexing_status));
  if ([...statuses].some((s) => TERMINAL_FAIL.has(s))) {
    throw new Error(`Ingestion errored: ${JSON.stringify(r)}`);
  }
  if ([...statuses].every((s) => TERMINAL_OK.has(s))) {
    processingComplete = true;
    break;
  }
  await sleep(5000);
}
if (!processingComplete) {
  throw new Error("Processing did not complete in 5 min");
}

// 5) Recall
const results = await client.recall.fullRecall({
  tenant_id: "demo_tenant",
  query: "What is this document about?",
  max_results: 5,
  alpha: 0.8,
});
console.log(results);
```

### 16.3 CLI

```bash
# 1) Login
hydradb login

# 2) Create + wait (bounded at 5 min)
hydradb tenant create --tenant-id demo_tenant
deadline=$(( $(date +%s) + 300 ))
until hydradb tenant monitor --tenant-id demo_tenant --output json \
      | jq -e '.infra_status == "ready"' >/dev/null 2>&1; do
  if [ "$(date +%s)" -ge "$deadline" ]; then
    echo "Tenant infra not ready after 5 min" >&2
    exit 1
  fi
  sleep 5
done

# 3) Ingest
hydradb knowledge upload --tenant-id demo_tenant --files doc.pdf \
  --metadata '{"category": "general"}'

# 4) Verify
hydradb knowledge verify --tenant-id demo_tenant --file-ids doc_001

# 5) Recall
hydradb recall full --tenant-id demo_tenant \
  --query "What is this document about?" --max-results 5
```
