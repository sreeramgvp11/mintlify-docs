# HydraDB Agent Integration Guide

This document is a self-contained reference designed for AI coding agents. It covers everything needed to understand, install, configure, and integrate HydraDB into any project -- from zero prior knowledge to production-ready usage.

**API Base URL:** `https://api.hydradb.com`

**Dashboard:** `https://app.hydradb.com`

**Documentation:** `https://docs.hydradb.com`

---

## 1. What Is HydraDB

HydraDB is a context infrastructure platform for AI applications. It provides persistent memory, intelligent retrieval, and knowledge graph context for LLM-powered systems.

**Core capabilities:**

- **Three memory types**: User memories (preferences, habits), Knowledge memories (documents, files), and Hive memories (organization-wide learnings)
- **Multi-stage recall**: Goes beyond vector similarity -- considers intent, context, graph relationships, recency, and personalization
- **Knowledge graphs**: Automatically extracts entities and relationships from ingested content
- **Multi-tenancy**: Complete data isolation via tenant/sub-tenant hierarchy
- **Hybrid search**: Blends semantic (dense) and keyword (sparse) search with configurable alpha parameter

**When to use HydraDB:**

- Building AI agents or chatbots that need persistent memory across sessions
- RAG (Retrieval-Augmented Generation) systems that need intelligent context retrieval
- Applications requiring per-user personalization of AI responses
- Knowledge management systems with document ingestion and search
- Any LLM application where context quality directly impacts output quality

**When NOT to use HydraDB:**

- Pure vector similarity search with no need for memory, graphs, or personalization (use a standalone vector DB)
- Structured relational data storage (use PostgreSQL, MySQL, etc.)
- Real-time streaming data (use Kafka, Redis Streams, etc.)

## 4. Core Concepts

### 4.1 Tenants and Sub-Tenants

A **tenant** is a top-level data container (like a database). A **sub-tenant** is an isolated partition within a tenant (like a schema or collection).

- Every tenant gets a DEFAULT sub-tenant automatically on creation
- No data crosses tenant boundaries
- Sub-tenants provide isolation within a tenant (e.g., per-user, per-department)

**Multi-tenancy patterns:**

| Pattern | tenant_id | sub_tenant_id | Use case |
|---|---|---|---|
| B2B SaaS | Customer org | Department/team | Enterprise clients with internal isolation |
| B2C App | Your app name | Individual user ID | Per-user personalization |
| Consulting | Your firm | Client project | Client data isolation |

### 4.2 Memory Types

| Type | What it stores | Scope | API endpoint |
|---|---|---|---|
| **User Memory** | Preferences, habits, conversation history | Per user (sub-tenant) | `POST /memories/add_memory` |
| **Knowledge** | Documents, PDFs, text files, structured data | Shared or per user | `POST /ingestion/upload_knowledge` |
| **Hive Memory** | Organization-wide learnings, collective insights | Organization-wide | `POST /memories/add_hive_memory` |

### 4.3 Recall vs Vector Search

HydraDB recall is a multi-stage retrieval pipeline, not a simple vector similarity search:

1. Understands query intent
2. Considers user context (tenant, sub-tenant)
3. Traverses knowledge graph for entity relationships
4. Weighs signals: recency, relevance, semantic similarity
5. Ranks results by usefulness for the current task

### 4.4 Metadata Architecture

Two-tier metadata system:

- **Tenant metadata**: Organization-level, immutable keys defined at tenant creation. Use for: compliance tags, business unit, region.
- **Document metadata**: Per-document, fully dynamic and mutable. Use for: author, project, status, tags.

Both can be used for filtering at recall time.

### 4.5 Hybrid Search (Alpha Parameter)

Controls the balance between semantic and keyword search:

| Alpha value | Behavior | Best for |
|---|---|---|
| `0.0` | Pure keyword search | Exact term matching |
| `0.5` | Balanced | General purpose |
| `0.8` | Mostly semantic (recommended default) | Conceptual queries |
| `1.0` | Pure semantic search | Abstract/conceptual queries |
| `"auto"` | Intelligent auto-selection | When unsure |

### 4.6 Recall Modes

| Mode | Behavior | Latency | Best for |
|---|---|---|---|
| `fast` | Single semantic query | Lower | Simple lookups, real-time UIs |
| `thinking` | Multi-query with reranking | Higher | Complex questions, higher accuracy |

### 5.1 Python Backend

```python
import os
import json
from hydra_db import HydraDB

client = HydraDB(token=os.environ["HYDRADB_API_KEY"])
TENANT_ID = os.environ["HYDRADB_TENANT_ID"]

# --- Create a tenant (one-time setup) ---
tenant = client.tenant.create(tenant_id=TENANT_ID)

# --- Ingest a document ---
with open("docs/architecture.pdf", "rb") as f:
    result = client.upload.knowledge(
        tenant_id=TENANT_ID,
        files=[("architecture.pdf", f)],
        file_metadata=json.dumps([
            {
                "id": "arch_doc_001",
                "tenant_metadata": {"department": "engineering"},
                "document_metadata": {"author": "Alice"}
            }
        ])
    )
print(f"Uploaded: {result.success_count} files")

# --- Add a user memory ---
memory_result = client.upload.add_memory(
    tenant_id=TENANT_ID,
    memories=[
        {
            "text": "User prefers concise responses with code examples",
            "infer": True,
            "user_name": "John"
        }
    ]
)

# --- Recall context ---
recall_result = client.recall.full_recall(
    query="How is authentication implemented?",
    tenant_id=TENANT_ID,

    recency_bias=0.3

for chunk in recall_result.chunks or []:

# --- Search user preferences ---
prefs = client.recall.recall_preferences(
    query="What response style does the user prefer?",
    tenant_id=TENANT_ID
)

# --- List ingested data ---

    tenant_id=TENANT_ID,
    kind="knowledge",
    page=1,
    page_size=20
)
```

### 5.3 RAG / LLM System Integration

The core pattern: retrieve context from HydraDB, inject it into your LLM prompt.

**Python with OpenAI:**

```python
import os
from hydra_db import HydraDB
from openai import OpenAI

hydra = HydraDB(token=os.environ["HYDRADB_API_KEY"])
openai_client = OpenAI()

def rag_query(user_question: str, tenant_id: str, sub_tenant_id: str = ""):
    # Step 1: Recall relevant context from HydraDB
    recall_result = hydra.recall.full_recall(
        query=user_question,
        tenant_id=tenant_id,
        sub_tenant_id=sub_tenant_id,
        mode="thinking",
        alpha=0.8,
        max_results=10,
        graph_context=True
    )

    # Step 2: Build context string from chunks
    context_parts = []
    for chunk in recall_result.chunks or []:
        context_parts.append(
            f"[Source: {chunk.source_title}]\n{chunk.chunk_content}"
        )
    context = "\n\n---\n\n".join(context_parts)

    # Step 3: Also recall user preferences
    prefs_result = hydra.recall.recall_preferences(
        query=user_question,
        tenant_id=tenant_id,
        sub_tenant_id=sub_tenant_id,
        max_results=5
    )
    user_prefs = "\n".join(
        c.chunk_content for c in (prefs_result.chunks or [])
    )

    # Step 4: Send to LLM with context
    response = openai_client.chat.completions.create(
        model="gpt-4",
        messages=[
            {
                "role": "system",
                "content": f"Answer based on the following context.\n\nContext:\n{context}\n\nUser preferences:\n{user_prefs}"
            },
            {"role": "user", "content": user_question}
        ]
    )
    return response.choices[0].message.content
```

**TypeScript with Anthropic:**

```typescript
import { HydraDBClient } from "@hydra_db/node";
import Anthropic from "@anthropic-ai/sdk";

const hydra = new HydraDBClient({
  token: process.env.HYDRADB_API_KEY,
  baseUrl: "https://api.hydradb.com",
});
const anthropic = new Anthropic();

async function ragQuery(question: string, tenant_id: string) {
  // Recall context
  const recall = await hydra.recall.fullRecall({
    tenant_id,
    query: question,
    mode: "thinking",
    alpha: 0.8,
    max_results: 10,
    graph_context: true,
  });

  const context = (recall.chunks ?? [])
    .map((c) => `[${c.source_title}]\n${c.chunk_content}`)
    .join("\n\n---\n\n");

  // Send to Claude with context
  const response = await anthropic.messages.create({
    model: "claude-sonnet-4-20250514",
    max_tokens: 1024,
    system: `Answer based on the following context.\n\nContext:\n${context}`,
    messages: [{ role: "user", content: question }],
  });

  return response.content[0].type === "text" ? response.content[0].text : "";
}
```

### 5.4 Full-Stack App (Next.js Example)

**API route** (`app/api/search/route.ts`):

```typescript
import { HydraDBClient } from "@hydra_db/node";
import { NextRequest, NextResponse } from "next/server";

const client = new HydraDBClient({
  token: process.env.HYDRADB_API_KEY,
  baseUrl: "https://api.hydradb.com",
});

export async function POST(req: NextRequest) {
  const { query, userId } = await req.json();

  const result = await client.recall.fullRecall({
    tenant_id: process.env.HYDRADB_TENANT_ID!,
    sub_tenant_id: userId,
    query,
    mode: "fast",
    alpha: 0.8,
    max_results: 10,
  });

  return NextResponse.json({
    results: (result.chunks ?? []).map((chunk) => ({
      title: chunk.source_title,
      content: chunk.chunk_content,
      score: chunk.relevancy_score,
    })),
  });
}
```

### 5.5 Data Pipeline Ingestion

```python
import os
import json
from hydra_db import HydraDB

client = HydraDB(token=os.environ["HYDRADB_API_KEY"])
TENANT_ID = os.environ["HYDRADB_TENANT_ID"]

def ingest_documents(file_paths: list[str]):
    """Batch ingest documents into HydraDB. Max 20 files per request."""
    batch_size = 20
    for i in range(0, len(file_paths), batch_size):
        batch = file_paths[i:i + batch_size]
        files = []
        metadata = []

        for path in batch:
            with open(path, "rb") as f:
                files.append((os.path.basename(path), f.read()))
            metadata.append({
                "id": os.path.basename(path),
                "document_metadata": {"source_path": path}
            })

        # Note: file_metadata must be a JSON string
        result = client.upload.knowledge(
            tenant_id=TENANT_ID,
            files=files,
            file_metadata=json.dumps(metadata)
        )
        print(f"Batch {i // batch_size + 1}: {result.success_count} uploaded, {result.failed_count} failed")
```

### 5.6 Agent Framework Integration

For agent frameworks (LangChain, LlamaIndex, CrewAI, etc.), HydraDB serves as the memory and retrieval backend:

```python
import os
from hydra_db import HydraDB

client = HydraDB(token=os.environ["HYDRADB_API_KEY"])
TENANT_ID = os.environ["HYDRADB_TENANT_ID"]

class HydraDBMemory:
    """Drop-in memory class for agent frameworks."""

    def __init__(self, tenant_id: str, sub_tenant_id: str = ""):
        self.tenant_id = tenant_id
        self.sub_tenant_id = sub_tenant_id

    def recall(self, query: str, max_results: int = 10) -> list[dict]:
        result = client.recall.full_recall(
            query=query,
            tenant_id=self.tenant_id,
            sub_tenant_id=self.sub_tenant_id,
            max_results=max_results,
            mode="thinking",
            graph_context=True
        )
        return [
            {
                "content": c.chunk_content,
                "source": c.source_title,
                "score": c.relevancy_score,
            }
            for c in (result.chunks or [])
        ]

    def store(self, text: str, user_name: str = "agent"):
        client.upload.add_memory(
            tenant_id=self.tenant_id,
            sub_tenant_id=self.sub_tenant_id,
            memories=[{"text": text, "infer": True, "user_name": user_name}]
        )

    def store_conversation(self, pairs: list[dict]):
        """Store user-assistant conversation pairs."""
        client.upload.add_memory(
            tenant_id=self.tenant_id,
```

### 6.6 Ingestion Best Practices

- **Batch limit**: Max 20 files per upload request
- **Wait between batches**: 1 second between batch requests
- **Always verify processing**: Poll `verify_processing` until status is `completed` before querying
- **Use metadata**: Rich metadata improves filtering and retrieval quality
- **Upsert**: Set `upsert=True` to update existing sources with the same ID instead of creating duplicates
- **Inference**: Set `infer=True` on user memories to let HydraDB extract insights automatically; set `infer=False` for factual/deterministic data

### 7.6 Retrieval Performance Considerations

- **`fast` mode**: Single query, lower latency. Use for real-time UIs and simple lookups.
- **`thinking` mode**: Multi-query with reranking, higher accuracy. Use for complex questions and when quality matters more than speed.
- **Alpha tuning**: Start with `0.8`. Lower toward `0.2` if users search for exact terms. Raise toward `1.0` for conceptual/abstract queries. Use `"auto"` when unsure.
- **Recency bias**: Default `0.0` (no bias). Set `0.3-0.6` for time-sensitive content. Set `0.8-1.0` when recent content is strongly preferred.
- **Metadata filters**: Narrow the search space before semantic matching for faster, more relevant results.
- **Max results**: 10-20 is typical. Higher values increase latency.

### Common Pitfalls

1. **Querying before processing completes**: After uploading, documents take 1-5 minutes to process. Always call `verify_processing` and wait for `completed` status before querying.

2. **Missing tenant creation**: You must create a tenant before any ingestion or recall operations. If you get `TENANT_NOT_FOUND`, create the tenant first.

3. **Infrastructure not ready**: After creating a tenant, poll `GET /tenants/infra/status` until both `graph_status` and `vectorstore_status` are ready before ingesting data.

4. **file_metadata must be a JSON string**: Both SDKs accept `file_metadata` as a string, not a list/array. Use `json.dumps()` in Python or `JSON.stringify()` in TypeScript.

5. **TS SDK default base URL**: The TypeScript SDK defaults to `https://api.usecortex.ai`. While this resolves to the same service, explicitly set `baseUrl: "https://api.hydradb.com"` for consistency.

6. **CLI global flags before subcommand**: In the HydraDB CLI, global options like `-o json` must precede the subcommand: `hydradb -o json memories list TENANT_ID`, not `hydradb memories list TENANT_ID -o json`.

7. **Tenant IDs are case-sensitive**: `My-Tenant` and `my-tenant` are different tenants. Use consistent casing.

8. **Rate limiting**: Implement exponential backoff on 429 responses. The SDKs have built-in retry (default: 2 retries for TS SDK).

### Retry Pattern

```python
import time

def retry_with_backoff(fn, max_retries=3):
    for attempt in range(max_retries):
        try:
            return fn()
        except TooManyRequestsError:
            if attempt == max_retries - 1:
                raise
            wait = 2 ** attempt
            time.sleep(wait)
```

## 13. Minimal End-to-End Example

A complete flow: install, configure, create tenant, ingest, verify, query.

### Python (Complete)


```python
import os
import json
import time
from hydra_db import HydraDB

# --- Step 1: Initialize client ---
client = HydraDB(token=os.environ["HYDRADB_API_KEY"])
TENANT_ID = "my-demo-tenant"

# --- Step 2: Create tenant ---
tenant = client.tenant.create(tenant_id=TENANT_ID)
print(f"Tenant created: {tenant.tenant_id}")

# --- Step 3: Wait for infrastructure ---
# Poll until infrastructure is ready (typically 10-30 seconds)
for _ in range(60):
    infra = client.tenant.get_infra_status(tenant_id=TENANT_ID)
    vs = infra.infra.vectorstore_status
    if infra.infra.graph_status and len(vs) > 0 and all(vs):
        print("Infrastructure ready.")
        break
    time.sleep(5)
else:
    raise TimeoutError("Infrastructure not ready after 5 minutes")

# --- Step 4: Ingest a user memory ---
memory_result = client.upload.add_memory(
    tenant_id=TENANT_ID,
    memories=[
        {
            "text": "The user is a backend engineer who prefers Python and uses PostgreSQL.",
            "infer": True,
            "user_name": "demo-user"
        }
    ]
)
print(f"Memory ingested: {memory_result.success_count} success")

# --- Step 5: Ingest a knowledge document ---
with open("README.md", "rb") as f:
    upload_result = client.upload.knowledge(
        tenant_id=TENANT_ID,
        files=[("README.md", f)],
        file_metadata=json.dumps([{"id": "readme_001"}])
    )
print(f"Knowledge uploaded: {upload_result.success_count} success")

# --- Step 6: Wait for processing ---
source_ids = [r.source_id for r in (upload_result.results or [])]
for _ in range(30):  # Poll for up to 5 minutes
    status = client.upload.verify_processing(
        tenant_id=TENANT_ID,
        file_ids=source_ids
    )
    all_done = all(
        s.indexing_status in ("completed", "success")
        for s in status.statuses
    )
    if all_done:
        print("Processing complete.")
        break
    print(f"Status: {[s.indexing_status for s in status.statuses]}")
    time.sleep(10)
else:
    raise TimeoutError("Processing did not complete within 5 minutes")

# --- Step 7: Recall context ---
recall_result = client.recall.full_recall(
    query="What technologies does this project use?",
    tenant_id=TENANT_ID,
    mode="thinking",
    alpha=0.8,
    max_results=5,
    graph_context=True
)

print("\n--- Recall Results ---")
for chunk in recall_result.chunks or []:
    print(f"[{chunk.relevancy_score}] {chunk.source_title}")
    print(f"  {chunk.chunk_content[:200]}")
    print()

# --- Step 8: Recall user preferences ---
prefs = client.recall.recall_preferences(
    query="What does the user prefer?",
    tenant_id=TENANT_ID
)
print("--- User Preferences ---")
for chunk in prefs.chunks or []:
    print(f"  {chunk.chunk_content[:200]}")

# --- Step 9: List all data ---
all_data = client.fetch.list_data(
    tenant_id=TENANT_ID,
    kind="knowledge"
)
print(f"\nTotal knowledge sources: {all_data.pagination.total if all_data.pagination else 'unknown'}")
```

```typescript
  const vs = infra.infra.vectorstore_status;
  if (infra.infra.graph_status && vs.length > 0 && vs.every(Boolean)) {
    console.log("Infrastructure ready.");
    break;
  }
  if (i === 59) throw new Error("Infrastructure not ready after 5 minutes");
  await new Promise((r) => setTimeout(r, 5000));
}

// Step 3: Ingest a user memory
const memResult = await client.upload.addMemory({
  tenant_id: TENANT_ID,
  memories: [
    {
      text: "The user is a backend engineer who prefers TypeScript.",
      infer: true,
      user_name: "demo-user",
    },
  ],
});
console.log(`Memory ingested: ${memResult.success_count} success`);

// Step 4: Upload a document
const uploadResult = await client.upload.knowledge({
  tenant_id: TENANT_ID,
  files: [fs.readFileSync("README.md")],
  file_metadata: JSON.stringify([{ id: "readme_001" }]),
});
console.log(`Knowledge uploaded: ${uploadResult.success_count} success`);

// Step 5: Wait for processing
const sourceIds = uploadResult.results?.map((r) => r.source_id) ?? [];
for (let i = 0; i < 30; i++) {
  const status = await client.upload.verifyProcessing({
    tenant_id: TENANT_ID,
    file_ids: sourceIds,
  });
  const allDone = status.statuses.every(
    (s) => s.indexing_status === "completed" || s.indexing_status === "success"
  );
  if (allDone) {
    console.log("Processing complete.");
    break;
  }
  if (i === 29) throw new Error("Processing did not complete within 5 minutes");
  console.log(`Status: ${status.statuses.map((s) => s.indexing_status)}`);
  await new Promise((r) => setTimeout(r, 10000));
}

// Step 6: Recall context
const recall = await client.recall.fullRecall({
  tenant_id: TENANT_ID,
  query: "What technologies does this project use?",
  mode: "thinking",
  alpha: 0.8,
  max_results: 5,
  graph_context: true,
});

console.log("\n--- Recall Results ---");
for (const chunk of recall.chunks ?? []) {
  console.log(`[${chunk.relevancy_score}] ${chunk.source_title}`);
  console.log(`  ${chunk.chunk_content?.slice(0, 200)}`);
}
```
