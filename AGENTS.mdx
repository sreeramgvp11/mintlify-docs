# HydraDB Agent Integration Guide

This document is a self-contained reference designed for AI coding agents. It covers everything needed to understand, install, configure, and integrate HydraDB into any project -- from zero prior knowledge to production-ready usage.


> **Important:** Prefer the v2 surface in this guide. Older v1 routes like `/recall/full_recall`, `/ingestion/upload_knowledge`, `/list/data`, and `/tenants/create` may still appear in the OpenAPI file for compatibility, but new integrations should use the consolidated v2 endpoints: `/tenants`, `/source/*`, and `/search`.

---

## 1. Critical rules for LLMs

### API versioning and auth

- Base URL: `https://api.hydradb.com`
- Raw HTTP calls must include:
  - `Authorization: Bearer <api_key>`
  - `API-Version: 2`
- Official SDKs set `API-Version: 2` automatically.
- Use `HYDRA_DB_API_KEY` as the environment variable in examples.

### Response envelope

Raw HTTP responses are wrapped:

```json
{
  "success": true,
  "data": {},
  "error": null,
  "meta": {
    "request_id": "...",
    "latency_ms": 12.3
  }
}
```

- Parse raw HTTP payloads from `data`.
- SDKs unwrap the envelope and return the inner `data` object.
- Log `meta.request_id` for failed requests.

### Use v2 endpoints for new work

| Task | Use this v2 endpoint / SDK method |
|---|---|
| Create tenant | `POST /tenants` · `client.tenants.create()` |
| Check tenant readiness | `GET /tenants/status` · `client.tenants.status()` |
| List tenants | `GET /tenants` · `client.tenants.list()` |
| List sub-tenants | `GET /tenants/sub-tenants` · TS `client.tenants.subTenants()` / Python `client.tenants.sub_tenants()` |
| Tenant stats | `GET /tenants/stats` · `client.tenants.stats()` |
| Ingest files, app sources, memories | `POST /source/ingest` · `client.source.ingest()` |
| Check indexing | `GET /source/status` · `client.source.status()` |
| Search knowledge/memories/both | `POST /search` · `client.search.query()` |
| List sources/memories | `POST /source/list` · `client.source.list()` |
| Fetch source content | `GET /source/fetch` · `client.source.fetch()` |
| Delete sources/memories | `DELETE /source` · `client.source.delete()` |
| Inspect graph relations | `GET /source/relations` · `client.source.relations()` |
| Indexing webhooks | `/webhooks/indexing*` |

### Async lifecycle

Two operations are asynchronous:

1. **Tenant creation**: after `POST /tenants`, poll `GET /tenants/status` until infrastructure is ready.
2. **Ingestion**: after `POST /source/ingest`, poll `GET /source/status` until the source is searchable or fully complete.

Searchable status:

- `graph_creation`: searchable, graph may still be incomplete.
- `completed`: fully indexed and graphed.

Failure status:

- Docs use `errored`; OpenAPI also exposes `failed` in one enum. Treat both as terminal failures.

### Search parameter naming

The canonical OpenAPI request field for `POST /search` is `source`:

- `source: "knowledge"`
- `source: "memory"`
- `source: "all"`

Several v2 SDK/doc examples use `type` as a friendly alias because `/source/ingest` uses `type`. For maximum raw-HTTP correctness, use `source` in cURL/fetch examples. In SDK code, follow the generated client type/autocomplete; if both are accepted, prefer the field shown by the SDK docs for that language.

### Metadata naming

- `tenant_metadata`: schema-aligned, declared at tenant creation, fast filter path.
- `additional_metadata`: canonical free-form per-source metadata.
- `document_metadata`: supported/deprecated alias for free-form metadata in many examples.
- `metadata_filters` top-level keys match `tenant_metadata`.
- `metadata_filters.additional_metadata` or `metadata_filters.document_metadata` scopes free-form metadata.

### Do not mix scopes accidentally

- `tenant_id` is the hard isolation boundary.
- `sub_tenant_id` is a logical partition inside a tenant, typically user/workspace/team.
- Use the same `sub_tenant_id` on writes and reads. Data written under one sub-tenant should not be expected to appear from another.
- Do not use `metadata_filters` as a substitute for `sub_tenant_id`.

## 2. Core primitives

### Tenants and sub-tenants

A tenant is an isolated workspace. A sub-tenant partitions data inside a tenant.

Recommended patterns:

| Use case | Pattern |
|---|---|
| B2B SaaS | one `tenant_id` per customer; `sub_tenant_id` for workspace/team/user |
| B2C app | one app/customer tenant; `sub_tenant_id = user_id` |
| Environment isolation | separate `tenant_id`s for prod/staging |
| Shared knowledge | tenant-wide/default sub-tenant or consistent shared scope |
| User memories | `sub_tenant_id = user_id` |

### Knowledge

Knowledge is shared context: documents, PDFs, Markdown, CSVs, Slack threads, Notion pages, Gmail threads, tickets, webpages, etc.

- Ingest with `POST /source/ingest`, `type=knowledge`.
- Search with `POST /search`, `type: "knowledge"`.
- Use for content that should be reusable across users.
- Mutability is explicit: re-ingest with the same ID and `upsert: true`, or delete.

### Memories

Memories are user/workspace/session-scoped context: preferences, conversation history, behavioral signals, decisions, inferred traits.

- Ingest with `POST /source/ingest`, `type=memory`.
- Search with `POST /search`, `type: "memory"` or `type: "all"`.
- Always pass the same `sub_tenant_id` used at ingestion.
- Use `infer: true` when the input is raw signal and HydraDB should extract the durable preference/fact.
- Use `infer: false` when the memory is already structured and should be stored verbatim.

### Search

`POST /search` is the single retrieval endpoint.

It can search:

- Knowledge only: `type: "knowledge"`
- Memories only: `type: "memory"`
- Both stores together: `type: "all"`

It supports:

- `search_by: "hybrid"` for semantic + BM25 retrieval.
- `search_by: "text"` for BM25 keyword/phrase search.
- `mode: "fast"` for low latency.
- `mode: "thinking"` for query expansion, reranking, richer graph traversal, and forceful-relation expansion.

### Context graph

HydraDB builds a graph of entity/relation triplets from ingested content. When `graph_context: true` (default), search can return:

- `graph_context.query_paths`
- `graph_context.chunk_relations`
- `graph_context.chunk_id_to_group_ids`

Graph context augments retrieval; `chunks` remain the primary search output.

### Forceful relations

At ingestion, sources can declare explicit relations:

```json
{
  "relations": {
    "source_ids": ["related-source-1", "related-source-2"],
    "properties": { "reason": "same thread" }
  }
}
```

At search time, set or rely on default `search_forceful_relations: true` with `mode: "thinking"` to pull related chunks into `additional_context`.

Rules:

- Forceful relation expansion only takes effect in `mode: "thinking"`.
- Prefer `source_ids`; legacy alias `cortex_source_ids` is accepted but should not be used in new code.
- Relations are store-local: memory-to-memory or knowledge-to-knowledge. Cross-store relation lookups may not surface anything.

## 4. Minimal end-to-end flow

### Python

```python
import json
import os
import time
from hydra_db import HydraDB

client = HydraDB(token=os.environ["HYDRA_DB_API_KEY"])
tenant_id = "my_first_tenant"

# 1. Create tenant.
client.tenants.create(tenant_id=tenant_id)

# 2. Wait until the tenant can accept data.
while True:
    status = client.tenants.status(tenant_id=tenant_id)
    if status.infra.ready_for_ingestion:
        break
    time.sleep(5)

# 3. Ingest a user memory.
ingest = client.source.ingest(
    type="memory",
    tenant_id=tenant_id,
    sub_tenant_id="user_alex",
    memories=json.dumps([
        {
            "source_id": "user_alex_pref_001",
            "text": "User prefers detailed technical explanations and dark mode.",
            "infer": True,
            "user_name": "Alex",
        }
    ]),
)

source_id = ingest.results[0].source_id

# 4. Wait until searchable.
while True:
    status = client.source.status(
        tenant_id=tenant_id,
        sub_tenant_id="user_alex",
        source_ids=[source_id],
    ).statuses[0]

    if status.indexing_status in ("graph_creation", "completed"):
        break
    if status.indexing_status in ("errored", "failed"):
        raise RuntimeError(getattr(status, "error_message", "Indexing failed"))
    time.sleep(2)

# 5. Search memories.
result = client.search.query(
    tenant_id=tenant_id,
    sub_tenant_id="user_alex",
    query="What does the user prefer?",
    type="memory",
    search_by="hybrid",
    mode="thinking",
)

print(result.chunks)
```

### TypeScript

```ts
import { HydraDBClient } from "@hydradb/sdk";

const client = new HydraDBClient({
  token: process.env.HYDRA_DB_API_KEY,
});

const tenantId = "my_first_tenant";

await client.tenants.create({ tenant_id: tenantId });

while (true) {
  const status = await client.tenants.status({ tenant_id: tenantId });
  if (status.infra.ready_for_ingestion) break;
  await new Promise((resolve) => setTimeout(resolve, 5_000));
}

const ingest = await client.source.ingest({
  type: "memory",
  tenant_id: tenantId,
  sub_tenant_id: "user_alex",
  memories: JSON.stringify([
    {
      source_id: "user_alex_pref_001",
      text: "User prefers detailed technical explanations and dark mode.",
      infer: true,
      user_name: "Alex",
    },
  ]),
});

const sourceId = ingest.results[0].source_id;

while (true) {
  const status = (await client.source.status({
    tenant_id: tenantId,
    sub_tenant_id: "user_alex",
    source_ids: [sourceId],
  })).statuses[0];

  if (["graph_creation", "completed"].includes(status.indexing_status)) break;
  if (["errored", "failed"].includes(status.indexing_status)) {
    throw new Error(status.error_message ?? "Indexing failed");
  }

  await new Promise((resolve) => setTimeout(resolve, 2_000));
}

const result = await client.search.query({
  tenant_id: tenantId,
  sub_tenant_id: "user_alex",
  query: "What does the user prefer?",
  type: "memory",
  search_by: "hybrid",
  mode: "thinking",
});

console.log(result.chunks);
```

## 5. Tenants API

### Create tenant

`POST /tenants` · `client.tenants.create()`

```json
{
  "tenant_id": "acme_corp",
  "tenant_metadata_schema": [
    {
      "name": "department",
      "data_type": "VARCHAR",
      "enable_match": true,
      "enable_dense_embedding": false,
      "enable_sparse_embedding": false,
      "max_length": 1024
    }
  ],
  "is_embeddings_tenant": false,
  "embeddings_dimension": 1536
}
```

Notes:

- Tenant creation is async.
- `tenant_id` should be stable; docs recommend lowercase letters, numbers, and underscores for portability.
- `tenant_metadata_schema` is effectively planned up front. If you need to change filterable metadata fields, expect to create a new tenant or re-ingest under a new schema.
- `is_embeddings_tenant: true` is for raw vector storage, not normal parsed-source ingestion.

### Check readiness

`GET /tenants/status?tenant_id=...` · `client.tenants.status()`

Ready when infrastructure reports usable status. Current docs show `infra.ready_for_ingestion`; SDK examples also check:

- `infra.scheduler_status`
- `infra.graph_status`
- `infra.vectorstore_status.knowledge`
- `infra.vectorstore_status.memory` / `memories`

Use SDK autocomplete/returned object shape for the exact field spelling.

### Other tenant endpoints

| Endpoint | Purpose |
|---|---|
| `GET /tenants` | list tenant IDs |
| `DELETE /tenants?tenant_id=...` | delete a tenant and all data |
| `GET /tenants/sub-tenants?tenant_id=...` | list active sub-tenants |
| `GET /tenants/stats?tenant_id=...` | collection counts and vector dimensions |

## 6. Source ingestion

`POST /source/ingest` is the unified write endpoint for knowledge and memories.

### Common form fields

| Field | Description |
|---|---|
| `type` | `"knowledge"` or `"memory"`; default is `"knowledge"` |
| `tenant_id` | target tenant |
| `sub_tenant_id` | optional logical partition; default sub-tenant if omitted |
| `upsert` | default `true`; replaces existing sources with same ID |
| `files` | binary uploads for knowledge |
| `file_metadata` | JSON-stringified array, one item per file |
| `app_knowledge` | JSON-stringified object/array of pre-extracted app sources |
| `memories` | JSON-stringified array of memory items |

### Knowledge from files

Use for PDFs, DOCX, Markdown, CSV, TXT, and other files HydraDB should parse.

```python
import json

with open("policy.pdf", "rb") as f:
    result = client.source.ingest(
        type="knowledge",
        tenant_id="acme_corp",
        files=[("policy.pdf", f, "application/pdf")],
        file_metadata=json.dumps([
            {
                "source_id": "policy_v2",
                "tenant_metadata": {"department": "support", "status": "approved"},
                "additional_metadata": {"author": "Alice", "version": 2},
                "title": "Refund Policy v2",
                "url": "https://example.com/policies/refunds",
            }
        ]),
    )
```

`file_metadata` item fields:

| Field | Purpose |
|---|---|
| `source_id` | stable source ID / upsert key |
| `tenant_metadata` | schema-aligned filterable fields |
| `additional_metadata` / `document_metadata` | free-form display/bookkeeping fields |
| `title`, `type`, `url`, `timestamp` | source display and citation fields |
| `relations` | explicit forceful relations, e.g. `{ "source_ids": [...] }` |

### Knowledge from app sources

Use `app_knowledge` when your connector already extracted text from Slack, Gmail, Jira, Linear, Zendesk, Notion, Confluence, Salesforce, webpages, etc.

```ts
await client.source.ingest({
  type: "knowledge",
  tenant_id: "acme_corp",
  app_knowledge: JSON.stringify([
    {
      id: "slack_C123_1715012345_000100",
      tenant_id: "acme_corp",
      sub_tenant_id: "default",
      title: "Pricing discussion - Slack #product",
      type: "slack",
      url: "https://slack.com/archives/C123/p1715012345000100",
      timestamp: "2025-01-15T12:30:00Z",
      content: {
        text: "We agreed on three tiers: Starter, Pro, and Enterprise."
      },
      tenant_metadata: {
        channel: "product",
        workspace: "acme"
      },
      additional_metadata: {
        author: "alice",
        slack_ts: "1715012345.000100"
      },
      relations: {
        source_ids: ["slack_thread_root_001"]
      }
    }
  ]),
});
```

Recommended app-source fields:

| Field | Why send it |
|---|---|
| `id` | stable HydraDB source ID and upsert key |
| `title` | readable result/citation title |
| `type` | category such as `slack`, `gmail`, `jira`, `notion`, `webpage` |
| `content.text` / `content.markdown` | primary searchable content |
| `tenant_metadata` | fast filters such as channel, project, region, workspace |
| `additional_metadata` | provider IDs, author, status, assignee, timestamps |
| `url` | citation / navigation link |
| `timestamp` | freshness and chronology |
| `relations.source_ids` | thread, parent/child, linked issue, related source expansion |

App-source object kinds in OpenAPI include:

- `email`
- `message`
- `ticket`
- `knowledge_base`
- `comment`
- `custom`

Modern docs show both a generic `content` shape and a richer app shape with `kind`, `provider`, `external_id`, `fields`, `attachments`, `comments`, and `relations`. Use the richer shape when building app-aware search; at minimum provide stable IDs, content, source type, metadata, and relations.

### Memories

Use `type="memory"` and a JSON-stringified `memories` array.

```python
import json

client.source.ingest(
    type="memory",
    tenant_id="acme_corp",
    sub_tenant_id="user_alex",
    memories=json.dumps([
        {
            "source_id": "pref_dark_mode",
            "title": "Dark mode preference",
            "text": "User prefers dark mode and concise answers.",
            "infer": True,
            "user_name": "Alex",
            "tenant_metadata": {"team": "engineering"},
            "additional_metadata": {"source": "onboarding"},
            "relations": {"source_ids": ["pref_concise"]}
        }
    ]),
)
```

Memory item fields:

| Field | Description |
|---|---|
| `source_id` | optional stable upsert key |
| `title` | label for listing/search display |
| `text` | raw text or Markdown; required unless `user_assistant_pairs` provided |
| `user_assistant_pairs` | dialogue pairs `{ user, assistant }`; useful for implied preferences |
| `is_markdown` | preserve Markdown structure during chunking |
| `infer` | extract durable preference/fact from raw signal |
| `custom_instructions` | guide inference when `infer: true` |
| `user_name` | helps inference |
| `expiry_time` | TTL in seconds |
| `tenant_metadata` | schema-aligned fields |
| `additional_metadata` / `document_metadata` | free-form fields |
| `relations` | explicit memory-to-memory links |

## 7. Source status and lifecycle

### Poll indexing

`GET /source/status` · `client.source.status()`

Parameters:

- `tenant_id` required
- `source_ids` required array
- `sub_tenant_id` optional, but should match ingestion scope

Status meanings from docs:

| Status | Searchable? | Meaning |
|---|---:|---|
| `queued` | No | accepted, not picked up |
| `processing` | No | parsing/chunking/embedding |
| `graph_creation` | Yes | chunks indexed; graph still building |
| `completed` | Yes | fully indexed and graphed |
| `errored` / `failed` | No | terminal failure |

Pattern:

```ts
while (true) {
  const status = (await client.source.status({
    tenant_id: "acme_corp",
    source_ids: ["policy_v2"],
  })).statuses[0];

  if (["graph_creation", "completed"].includes(status.indexing_status)) break;
  if (["errored", "failed"].includes(status.indexing_status)) {
    throw new Error(status.error_message ?? status.message ?? "Indexing failed");
  }
  await new Promise((r) => setTimeout(r, 2_000));
}
```

### Webhooks instead of polling

Use `/webhooks/indexing` to receive terminal indexing events.

Endpoints:

| Endpoint | Purpose |
|---|---|
| `GET /webhooks/indexing` | get current webhook registration |
| `POST /webhooks/indexing` | register or replace webhook |
| `DELETE /webhooks/indexing` | delete webhook |
| `POST /webhooks/indexing/test` | send test delivery |
| `GET /webhooks/indexing/deliveries` | list delivery attempts |
| `GET /webhooks/indexing/deliveries/{delivery_id}` | get one delivery |
| `POST /webhooks/indexing/deliveries/{delivery_id}/retry` | retry delivery |

Supported event:

- `indexing.status_changed`

Payload shape:

```json
{
  "event": "indexing.status_changed",
  "delivery_id": "<delivery_id>",
  "id": "<source_id>",
  "tenant_id": "<tenant_id>",
  "sub_tenant_id": "<sub_tenant_id>",
  "status": "completed",
  "timestamp": "<ISO-8601 timestamp>",
  "error_code": null,
  "error_message": null
}
```

Headers:

- `X-HydraDB-Delivery-ID`
- `X-HydraDB-Event`
- `X-HydraDB-Signature` when a signing secret is configured

Rules:

- Webhook URL must be public HTTPS; localhost/private networks are blocked.
- Store `delivery_id` to deduplicate retries.
- If using `signing_secret`, verify `X-HydraDB-Signature`.

## 8. Search API

`POST /search` · `client.search.query()`

### Request fields that matter

| Field | Values | Notes |
|---|---|---|
| `tenant_id` | string | required |
| `sub_tenant_id` | string/null | required for per-user memory searches |
| `query` | string | required |
| `type` | `knowledge`, `memory`, `all` | docs/SDK store selector; OpenAPI may expose alias `source` |
| `search_by` | `hybrid`, `text` | default hybrid |
| `operator` | `or`, `and`, `phrase` | only for `search_by: "text"` |
| `mode` | `fast`, `thinking` | applies to hybrid; thinking improves quality but adds latency |
| `max_results` | integer/null | default around 10; max documented as 50 |
| `alpha` | `0.0`–`1.0` or `auto` | hybrid weight; `1.0` semantic, `0.0` BM25 |
| `recency_bias` | `0.0`–`1.0` | boost newer content |
| `graph_context` | boolean | default true; include graph slice |
| `search_forceful_relations` | boolean | only effective in `thinking` mode |
| `additional_context` | string | short factual hint; not a hard filter |
| `metadata_filters` | object | deterministic filters before/around ranking |
| `search_apps` | boolean | app-aware retrieval lane for app sources |

### Recommended configurations

| Goal | Request shape |
|---|---|
| Fast document RAG | `type: "knowledge"`, `search_by: "hybrid"`, `mode: "fast"`, `graph_context: false`, `max_results: 5-10` |
| Highest-quality document RAG | `type: "knowledge"`, `search_by: "hybrid"`, `mode: "thinking"`, `graph_context: true`, `alpha: "auto"` |
| Personalized grounded answer | `type: "all"`, include `sub_tenant_id`, `search_by: "hybrid"`, `mode: "thinking"` |
| User preferences only | `type: "memory"`, include `sub_tenant_id`, `search_by: "hybrid"` |
| Exact keyword / phrase | `type: "knowledge"`, `search_by: "text"`, `operator: "phrase"` |
| Recent operational updates | `search_by: "hybrid"`, `recency_bias: 0.2-0.4`, metadata filter to doc type/status |
| App search | `type: "knowledge"`, `search_by: "hybrid"`, `mode: "thinking"`, `search_apps: true` |

### Search examples

Knowledge RAG:

```ts
const result = await client.search.query({
  tenant_id: "acme_corp",
  query: "What is our refund policy?",
  type: "knowledge",
  search_by: "hybrid",
  mode: "thinking",
  max_results: 10,
  graph_context: true,
});
```

Personalized answer using knowledge and memories:

```ts
const result = await client.search.query({
  tenant_id: "acme_corp",
  sub_tenant_id: "user_alex",
  query: "What is our refund policy, and how should I explain it to this user?",
  type: "all",
  search_by: "hybrid",
  mode: "thinking",
  alpha: "auto",
});
```

Exact phrase:

```python
result = client.search.query(
    tenant_id="acme_corp",
    query="ERROR_429 rate limit",
    type="knowledge",
    search_by="text",
    operator="phrase",
)
```

Metadata-scoped search:

```json
{
  "tenant_id": "acme_corp",
  "query": "authentication control review",
  "type": "knowledge",
  "search_by": "hybrid",
  "metadata_filters": {
    "compliance_framework": "SOC2",
    "additional_metadata": {
      "author": "alice"
    }
  }
}
```

The successful `data` object / SDK return is a `RetrievalResult`:

```json
{
  "chunks": [
    {
      "chunk_uuid": "...",
      "source_id": "...",
      "chunk_content": "...",
      "source_title": "...",
      "source_type": "...",
      "source_upload_time": "...",
      "source_last_updated_time": "...",
      "relevancy_score": 0.91,
      "metadata": {},
      "additional_metadata": {},
      "extra_context_ids": []
    }
  ],
  "sources": [
    {
      "id": "...",
      "title": "...",
      "type": "...",
      "url": "...",
      "timestamp": "...",
      "metadata": {},
      "additional_metadata": {},
      "app_kind": "message",
      "app_provider": "slack",
```

## 9. Turning search results into LLM prompts

Practical guidance from `essentials/v2/api-results.mdx`:

- Preserve the order of `chunks`; it is the server ranking.
- Use graph context only when it helps the question.
- Use `additional_context` to attach forcefully related chunks by `chunk_uuid` / `extra_context_ids`.
- For personalized answers, `type: "all"` is simplest: one result set already merges knowledge and memories.
- Do not pass raw JSON directly to the LLM if token budget matters; use the SDK formatting helper to produce a compact context string.
- Include a grounding instruction: answer only from provided context, and say when context is insufficient.

Use the SDK helper from `essentials/v2/api-results.mdx`:

- Python: `build_string(result)` from `hydra_db`
- TypeScript: `buildString(result)` from `@hydradb/sdk`

Python:

```python

from openai import OpenAI

hydra = HydraDB(token=os.environ["HYDRA_DB_API_KEY"])
openai = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

question = "How does authentication work?"

result = hydra.search.query(
    tenant_id="your-tenant",
    sub_tenant_id="your-sub-tenant",
    query=question,
    source="knowledge",
    search_by="hybrid",
    max_results=5,
)

context = build_string(result)

messages = [
    {
        "role": "system",
        "content": "Answer using only the provided HydraDB context. If the answer is not in the context, say so. Cite source titles or URLs when available.",
    },
    {
        "role": "user",
        "content": f"{context}\n\nQuestion: {question}",
    },
]
```

Common mistakes:

| Mistake | Fix |
|---|---|
| Passing raw retrieval JSON to the LLM | Use SDK `build_string(result)` / `buildString(result)` |
| Omitting `sub_tenant_id` for memories | Use the same scope as ingestion |
| Including too many chunks | Start with `max_results: 10`; reduce for tight windows |
| Re-sorting chunks client-side | Preserve HydraDB ranking |
| Setting `graph_context: false` then expecting graph fields | Leave default true or set true explicitly |
| Using `search_forceful_relations` in `fast` mode | Use `mode: "thinking"` |

## 10. Metadata guide

### Tenant schema

Declared at `POST /tenants`:

```json
{
  "tenant_metadata_schema": [
    {
      "name": "department",
      "data_type": "VARCHAR",
      "enable_match": true,
      "enable_dense_embedding": false,
      "enable_sparse_embedding": false,
      "max_length": 1024
    }
  ]
}
```

Field options:

| Field | Purpose |
|---|---|
| `name` | metadata key; must be valid and non-reserved |
| `data_type` | friendly names like `string`, `integer`, `float`, `boolean`, or backend types like `VARCHAR`, `INT*`, `JSON` |
| `enable_match` | allows exact matching in top-level `metadata_filters` |
| `enable_dense_embedding` | semantically embed `VARCHAR` field |
| `enable_sparse_embedding` | BM25 index `VARCHAR` field |
| `max_length` | max string length |
| `searchable` | shorthand for dense + sparse embedding in docs |
| `filterable` | shorthand for `enable_match` in docs |

### Where metadata belongs

| Need | Put it in | Filter with |
|---|---|---|
| Hot-path exact filters: department, region, environment, status | `tenant_metadata` | top-level `metadata_filters` |
| Free-form display/bookkeeping: author, provider IDs, version | `additional_metadata` | `metadata_filters.additional_metadata` |
| Provider app fields for UI/citations | `additional_metadata` or app `fields` | usually not filtered unless needed |
| User/workspace partition | `sub_tenant_id` | not metadata |

Rules:

- Plan hot filter fields before first ingest; undeclared scope keys can be ignored.
- Use exact equality filters; range/contains/fuzzy matching should be in the query or downstream reranker.
- To change metadata on an indexed source, re-ingest with same source ID and `upsert: true`.
- Do not rename metadata keys without re-ingesting affected sources.

## 12. Error handling

### HTTP status codes

| Code | Meaning | Retry? |
|---|---|---|
| `400` | invalid parameters / malformed request | No |
| `401` | missing/invalid API key | No |
| `403` | authenticated but not permitted | No |
| `404` | tenant/source/memory not found | No |
| `409` | conflict, e.g. existing tenant/source ID | Usually no |
| `422` | semantic validation failure | No |
| `429` | rate limited | Yes with backoff |
| `500` | internal error | Yes with backoff |
| `503` | temporary unavailability | Yes with backoff |

### Common error codes

| Code | Meaning |
|---|---|
| `INVALID_PARAMETERS` | required/malformed/incompatible parameter |
| `UNAUTHORIZED` | missing/invalid bearer token |
| `FORBIDDEN` | key lacks access |
| `TENANT_ALREADY_EXISTS` | duplicate tenant ID |
| `TENANT_NOT_FOUND` | tenant missing/not visible |
| `SOURCE_NOT_FOUND` | source/memory missing in selected scope |
| `VALIDATION_ERROR` | request shape valid but failed validation |
| `RATE_LIMITED` | rate limit exceeded |
| `INTERNAL_ERROR` | unexpected server error |
| `SERVICE_UNAVAILABLE` | dependency unavailable / under load |

Retry only `429`, `500`, `503`; use bounded exponential backoff with jitter.

SDK errors:

```python
from hydra_db import HydraDBError

try:
    result = client.search.query(

except HydraDBError as exc:
    if exc.status_code in (429, 500, 503):
        # retry with backoff
        pass
    else:
        raise
```

## 16. Common mistakes checklist

- [ ] Using v1 endpoints for new code instead of `/tenants`, `/source/*`, `/search`.
- [ ] Forgetting `API-Version: 2` in raw HTTP calls.
- [ ] Parsing raw HTTP response from the top level instead of `data`.
- [ ] Assuming SDK responses include the raw envelope; SDKs unwrap `data`.
- [ ] Searching immediately after tenant creation without polling readiness.
- [ ] Searching immediately after ingestion without polling `source.status` or using webhooks.
- [ ] Treating `processing`/`queued` content as searchable.
- [ ] Not handling both `errored` and `failed` as failure statuses.
- [ ] Omitting `sub_tenant_id` for user memories.
- [ ] Writing with one `sub_tenant_id` and reading with another.
- [ ] Using metadata filters for user partitioning instead of `sub_tenant_id`.
- [ ] Using undeclared `tenant_metadata` keys in `metadata_filters`.
- [ ] Putting hot filters in free-form metadata instead of tenant schema.
- [ ] Expecting `search_forceful_relations` to work in `mode: "fast"`.
- [ ] Setting `graph_context: false` and expecting graph fields.
- [ ] Re-sorting chunks before prompting without a deliberate reranker.
- [ ] Passing too many chunks to the LLM.
- [ ] Using `operator` without `search_by: "text"`.
- [ ] Forgetting to verify webhook signatures when a signing secret is configured.

### 17.1 Raw HTTP: create tenant, ingest memory, poll, search

Use `source` on raw `/search` requests because that is the OpenAPI field. The raw API response is wrapped; read payloads from `.data`.

```bash
export HYDRA_DB_API_KEY="your_api_key"
TENANT_ID="agent_docs_demo"
SUB_TENANT_ID="user_alex"

# 1. Create a tenant.
curl -s -X POST 'https://api.hydradb.com/tenants' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d "{\"tenant_id\":\"${TENANT_ID}\"}"

# 2. Check readiness. Poll this until .data.infra.ready_for_ingestion is true.
curl -s "https://api.hydradb.com/tenants/status?tenant_id=${TENANT_ID}" \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" | jq '.data.infra'

# 3. Ingest a memory. The memories field is JSON serialized inside multipart form data.
SOURCE_ID=$(curl -s -X POST 'https://api.hydradb.com/source/ingest' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -F "type=memory" \
  -F "tenant_id=${TENANT_ID}" \
  -F "sub_tenant_id=${SUB_TENANT_ID}" \
  -F 'memories=[{"source_id":"alex_pref_001","text":"Alex prefers concise technical answers and dark mode.","infer":true,"user_name":"Alex"}]' \
  | jq -r '.data.results[0].source_id')

# 4. Poll indexing. Stop at graph_creation if searchable is enough; wait for completed for full graph.
curl -s -G 'https://api.hydradb.com/source/status' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  --data-urlencode "tenant_id=${TENANT_ID}" \
  --data-urlencode "sub_tenant_id=${SUB_TENANT_ID}" \
  --data-urlencode "source_ids=${SOURCE_ID}" \
  | jq '.data.statuses[0]'

# 5. Search memories.
curl -s -X POST 'https://api.hydradb.com/search' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d "{\"tenant_id\":\"${TENANT_ID}\",\"sub_tenant_id\":\"${SUB_TENANT_ID}\",\"query\":\"How should I answer Alex?\",\"source\":\"memory\",\"search_by\":\"hybrid\",\"mode\":\"thinking\"}" \
  | jq '.data.chunks'
```

### 17.3 TypeScript: production helpers

The SDK unwraps `.data`. SDK docs show `type` for search; if your installed SDK exposes `source` instead, use `source`.

```ts
import { HydraDBClient, HydraDBError } from "@hydradb/sdk";

const client = new HydraDBClient({
  token: process.env.HYDRA_DB_API_KEY,
});

const sleep = (ms: number) => new Promise((resolve) => setTimeout(resolve, ms));

async function waitForTenant(tenant_id: string, timeoutMs = 300_000) {
  const deadline = Date.now() + timeoutMs;

  while (Date.now() < deadline) {
    const status = await client.tenants.status({ tenant_id });
    const infra = status.infra;

    if (infra.ready_for_ingestion) return status;

    const vectorReady =
      typeof infra.vectorstore_status === "object" &&
      Boolean(infra.vectorstore_status.knowledge) &&
      Boolean(infra.vectorstore_status.memory ?? infra.vectorstore_status.memories);

    if (infra.scheduler_status && infra.graph_status && vectorReady) return status;

    await sleep(5_000);
  }

  throw new Error(`Tenant ${tenant_id} was not ready after ${timeoutMs}ms`);
}

async function waitForSources(params: {
  tenant_id: string;
  source_ids: string[];
  sub_tenant_id?: string;
  requireCompleted?: boolean;
  timeoutMs?: number;
}) {
  const deadline = Date.now() + (params.timeoutMs ?? 300_000);
  const ok = params.requireCompleted ? new Set(["completed"]) : new Set(["graph_creation", "completed"]);
  const failed = new Set(["errored", "failed"]);

  while (Date.now() < deadline) {
    const response = await client.source.status({
      tenant_id: params.tenant_id,
      sub_tenant_id: params.sub_tenant_id,
      source_ids: params.source_ids,
    });

    const statuses = response.statuses ?? [];
    if (statuses.some((s) => failed.has(s.indexing_status))) {
      throw new Error(`Indexing failed: ${JSON.stringify(statuses)}`);
    }
    if (statuses.length === params.source_ids.length && statuses.every((s) => ok.has(s.indexing_status))) {
      return statuses;
    }

    await sleep(2_000);
  }

  throw new Error(`Sources did not finish indexing: ${params.source_ids.join(", ")}`);
}

async function withHydraRetry<T>(operation: () => Promise<T>, maxRetries = 3): Promise<T> {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await operation();
    } catch (error) {
      if (!(error instanceof HydraDBError)) throw error;
      const retryable = [429, 500, 503].includes(error.statusCode);
      if (!retryable || attempt === maxRetries) throw error;

      const delayMs = 2 ** attempt * 1000 + Math.floor(Math.random() * 250);
      await sleep(delayMs);
    }
  }
  throw new Error("unreachable");
}
```

### 17.4 Python: production helpers

```python
import random
import time
from hydra_db import HydraDBError


def wait_for_tenant(client, tenant_id: str, timeout_s: int = 300):
    deadline = time.time() + timeout_s

    while time.time() < deadline:
        status = client.tenants.status(tenant_id=tenant_id)
        infra = status.infra

        if getattr(infra, "ready_for_ingestion", False):
            return status

        vectorstore = getattr(infra, "vectorstore_status", None)
        vector_ready = False
        if vectorstore is not None:
            knowledge_ready = bool(getattr(vectorstore, "knowledge", False))
            memory_ready = bool(getattr(vectorstore, "memory", getattr(vectorstore, "memories", False)))
            vector_ready = knowledge_ready and memory_ready

        if infra.scheduler_status and infra.graph_status and vector_ready:
            return status

        time.sleep(5)

    raise TimeoutError(f"Tenant {tenant_id} was not ready after {timeout_s}s")


def wait_for_sources(client, tenant_id: str, source_ids: list[str], sub_tenant_id: str | None = None, require_completed: bool = False, timeout_s: int = 300):
    deadline = time.time() + timeout_s
    ok = {"completed"} if require_completed else {"graph_creation", "completed"}
    failed = {"errored", "failed"}

    while time.time() < deadline:
        response = client.source.status(
            tenant_id=tenant_id,
            sub_tenant_id=sub_tenant_id,
            source_ids=source_ids,
        )
        statuses = response.statuses
        values = {s.indexing_status for s in statuses}

        if values & failed:
            raise RuntimeError(f"Indexing failed: {statuses}")
        if len(statuses) == len(source_ids) and values.issubset(ok):
            return statuses

        time.sleep(2)

    raise TimeoutError(f"Sources did not finish indexing: {source_ids}")


def with_hydra_retry(operation, max_retries: int = 3):
    for attempt in range(max_retries + 1):
        try:
            return operation()
        except HydraDBError as exc:
            retryable = exc.status_code in (429, 500, 503)
            if not retryable or attempt == max_retries:
                raise
            time.sleep((2 ** attempt) + random.uniform(0, 0.25))
```

### 17.7 Webhook receiver example

```ts
import crypto from "node:crypto";
import express from "express";

const app = express();
const signingSecret = process.env.HYDRADB_WEBHOOK_SECRET;

app.post("/webhooks/hydradb", express.raw({ type: "application/json" }), (req, res) => {
  const signature = req.header("X-HydraDB-Signature");

  if (signingSecret && signature) {
    const expected = crypto
      .createHmac("sha256", signingSecret)
      .update(req.body)
      .digest("hex");

    if (!crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))) {
      return res.status(401).send("invalid signature");
    }
  }

  const event = JSON.parse(req.body.toString("utf8"));
  if (event.event === "indexing.status_changed") {
    // Store event.delivery_id to dedupe retries.
    // Mark event.id as completed/errored in your own DB.
    console.log(event.id, event.status, event.error_code, event.error_message);
  }

  res.status(204).send();
});
```
