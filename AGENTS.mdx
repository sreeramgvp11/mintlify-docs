## 1. Critical Rules for LLMs

If you read nothing else, read this section. These are the failure modes that bite agents most often.

### Endpoint & response shape (read before parsing any response)

1. **`full_recall` / `recall_preferences` return `chunks[]`, not `results[]`.** Field names are `chunk_content`, `source_title`, `relevancy_score` -- not `text`, `source_name`, `score`. A separate top-level `sources[]` array holds deduplicated source records; join on `chunk.source_id == source.id`.
2. **`infra/status` is nested.** Readiness fields live under `infra.{scheduler_status, graph_status, vectorstore_status}`. There is no top-level `infra_status: "ready"` flag.
3. **`monitor` returns collections, not totals.** Counts come from `normal_collection.row_count` (Knowledge) and `memory_collection.row_count` (Memories).
4. **`upload_knowledge` returns `results[].source_id`, not `source_ids[]`.** Each item also carries its own `status` and `error`.
5. **Deletions are per-item.** Always inspect `results[].deleted` -- a request can return top-level `success: true` while individual items fail.

### Endpoint selection

6. **Vectorstore index mapping** -- Index `[0]` = Memories (use `recall_preferences`); index `[1]` = Knowledge (use `full_recall`). Mixing endpoints returns empty.
7. **Scope consistency** -- Write and read must use the *same* `sub_tenant_id`. Mismatch returns empty results, not an error.
8. **`boolean_recall` is restricted** -- Do NOT send `alpha`, `mode`, `recency_bias`, `graph_context`, `metadata_filters`, or `additional_context`. Any of those returns 400.

### Async lifecycle

9. **Tenant creation is async.** After `POST /tenants/create`, poll `GET /tenants/infra/status` until all three booleans (`scheduler_status`, `graph_status`, both elements of `vectorstore_status`) are `true`.
10. **Ingestion is async.** After upload, poll `POST /ingestion/verify_processing` until each `indexing_status` is `completed` (or `success`). Fail fast on `errored`. `graph_creation` is already searchable -- only wait for `completed` if you need full graph context.

### Ingestion gotchas

11. **`upload_knowledge` is multipart -- always.** Even when sending only `app_knowledge` (no files), use `multipart/form-data`. JSON bodies return 422.
12. **`app_knowledge` must be a JSON-encoded string.** Sending a raw object via multipart yields 422. Same for `file_metadata`.
13. **`file_metadata` array length MUST equal `files` count.** Otherwise 422.
14. **`upsert` defaults to `true`** on both `upload_knowledge` and `add_memory`. Re-sending the same `source_id` overwrites silently. Set `upsert: false` for insert-only semantics.
15. **App sources use typed `fields`.** Each `app_knowledge` item needs `id`, `tenant_id`, `sub_tenant_id`, `kind`, `provider`, `external_id`, and searchable content under `fields`. Put channel/project/space scope in `metadata` -- `external_id` is the item ID, not the container. At recall, set `search_apps: true` on `full_recall`. See [App Sources](/essentials/app-sources).

### Schema & versioning

16. **Schema immutability** -- `tenant_metadata_schema` field names cannot be renamed or removed after tenant creation. Plan up-front; use `additional_metadata` for free-form per-document fields.
17. **`enable_match: true` is required for filtering.** A schema field without it is stored but silently ignored when passed in `metadata_filters`.
18. **`infer` defaults to `false`.** Set `infer: true` explicitly when storing conversation history or free-form notes you want extracted as preferences.
19. **Verify versions** -- Don't assume SDK version numbers; check PyPI / npm before pinning.

### Env-var prefixes are not uniform

20. **MCP, SDKs, Claude Code plugin** use `HYDRA_DB_API_KEY` and `HYDRA_DB_TENANT_ID` (underscore).
21. **CLI and Claude Code plugin** use `HYDRADB_TENANT_ID`, `HYDRADB_SUB_TENANT_ID`, `HYDRADB_API_URL` (no underscore), but `HYDRA_DB_API_KEY` (with underscore) for the key.
22. **OpenClaw plugin** uses `HYDRA_OPENCLAW_API_KEY` and `HYDRA_OPENCLAW_TENANT_ID`.
    Copy-paste exactly -- don't guess.

## 3. Core Primitives

### 3.1 Tenants & Sub-Tenants

**Tenant** -- Top-level isolated workspace. No cross-tenant data access. `tenant_id` required on every API call.

**Sub-Tenant** -- Logical partition within a tenant (user, team, workspace, project). Pass `sub_tenant_id` to scope operations. If omitted, defaults to the tenant's default sub-tenant.

**Design patterns:**

| Pattern | Configuration |
|---|---|
| B2B SaaS | One tenant per customer; sub-tenants for teams / workspaces / users |
| B2C app | One tenant for the app; sub-tenants per end user |
| Internal tools | One tenant per company; sub-tenants per department / project |
| Separate environments | Different `tenant_id` per env (prod / staging / dev) -- **NEVER** share across envs |

### 3.2 Memories

User-scoped context -- preferences, conversation history, inferred traits. Each memory is scoped to a `sub_tenant_id`.

- **Mutability:** Dynamic -- evolves with every interaction
- **Recall endpoint:** `POST /recall/recall_preferences`
- **Vectorstore index:** `[0]`
- **Inference:** Set `infer: true` to extract implicit preferences from raw input
- **Upsert:** Set `upsert: true` to update an existing memory by text match

### 3.3 Knowledge

Shared, tenant-wide context from documents and app sources.

- **Files:** PDF, DOCX, PPTX, Markdown, CSV, TXT, HTML via `files` on `upload_knowledge`
- **App sources:** Slack, Gmail, Jira, Notion, Linear, CRM, etc. via `app_knowledge` - see [App Sources](/essentials/app-sources)
- **Processing:** Parsed → Chunked → Embedded → Graph-linked at write time
- **Mutability:** Versioned -- replaced or deleted explicitly
- **Recall endpoint:** `POST /recall/full_recall` (set `search_apps: true` when querying ingested app data)
- **Vectorstore index:** `[1]`

### 3.4 Recall Pipeline

Multi-stage retrieval -- not a simple vector similarity search:

1. **Metadata filtering** -- Deterministic narrowing by tenant schema fields
2. **Hybrid retrieval** -- Semantic (vector) + keyword (BM25)
3. **Graph traversal** -- Follow entity relationships
4. **Personalized ranking** -- Recency, frequency, relevance scoring

### 3.5 Metadata

| Field | Purpose | Filterable at recall? | Mutable? |
|---|---|---|---|
| `metadata` | Schema-bound fields declared at tenant creation | Yes (via `metadata_filters`) | Per-document, but schema is immutable |
| `additional_metadata` | Free-form per-document fields | **No** | Yes |

### 3.6 Hybrid Search (`alpha` parameter)

Controls the semantic-vs-keyword balance on `full_recall`:

| `alpha` | Behavior | Best for |
|---|---|---|
| `0.0` | Pure keyword (BM25) | Exact term matching |
| `0.5` | Balanced | General purpose |
| `0.8` | Mostly semantic (recommended default) | Conceptual queries |
| `1.0` | Pure semantic | Abstract queries |
| `"auto"` | Backend chooses | When unsure |

### 3.7 Recall Modes

| Mode | Behavior | Latency | Best for |
|---|---|---|---|
| `fast` | Single semantic query | Lower | Simple lookups, real-time UIs |
| `thinking` | Multi-query + graph traversal + forceful relations | Higher | Complex / multi-hop queries; best for app-source ID/thread/relation expansion |

### 3.8 App sources (summary)

Pre-parsed records from business apps. One `app_knowledge` item = one HydraDB source.

| Layer | Fields | Purpose |
|---|---|---|
| Envelope | `id`, `tenant_id`, `sub_tenant_id` | Required on every item |
| Identity | `kind`, `provider`, `external_id` | `email`, `message`, `ticket`, `knowledge_base`, `custom` |
| Content | `fields` | Searchable body/title/status/thread IDs - match `fields.kind` to top-level `kind` |
| Scope | `metadata` | Channel, project, space, pipeline (`container_type`, `container_id`, …) |
| Structure | `relations`, `comments`, `attachments` | Cross-source links, discussion, extracted file text |

Full field reference, scenarios, and mistakes: [App Sources](/essentials/app-sources).

## 4. Installation

### 4.1 Prerequisites

1. An API key -- get one from [app.hydradb.com/keys](https://app.hydradb.com/keys)
2. A tenant ID -- create or copy from [app.hydradb.com/tenants](https://app.hydradb.com/tenants)

No account? Email `founders@hydradb.com` for onboarding.

All API requests require:

```
Authorization: Bearer <your_api_key>
```

### 4.2 Python SDK

**Requires Python 3.10+.**

```bash
pip install hydradb-sdk
```

<Note>
PyPI package name is `hydradb-sdk`; the importable module is `hydra_db` (with an underscore). This split is intentional -- install with the hyphenated name, import with the underscored name.
</Note>

**Initialize:**

```python
import os
from hydra_db import HydraDB  # PyPI package: hydradb-sdk; importable module: hydra_db

client = HydraDB(
    token=os.environ["HYDRA_DB_API_KEY"],
    base_url="https://api.hydradb.com",  # optional
)
```

**Async client:**

```python
import asyncio, os
from hydra_db import AsyncHydraDB

async def main():
    client = AsyncHydraDB(token=os.environ["HYDRA_DB_API_KEY"])
    results = await client.recall.full_recall(
        tenant_id="acme_corp",
        query="deployment procedures",
        max_results=10,
        alpha="auto",
    )
    print(results)

asyncio.run(main())
```

</Warning>

1. Environment variables (`HYDRA_DB_API_KEY`, `HYDRADB_TENANT_ID`)

## 5. API Reference

All endpoints are HTTP `POST` / `GET` / `DELETE` against `https://api.hydradb.com`, JSON-bodied unless noted, and require `Authorization: Bearer <api_key>`.

### 5.1 Tenants

#### `POST /tenants/create`

Create an isolated workspace. **Async** -- poll `/tenants/infra/status` before ingesting.

**Request:**

```json
{
  "tenant_id": "acme_corp",
  "tenant_metadata_schema": [
    {
      "name": "category",
      "data_type": "VARCHAR",
      "enable_match": true,
      "enable_dense_embedding": false,
      "enable_sparse_embedding": false
    },
    {
      "name": "timestamp",
      "data_type": "VARCHAR",
      "enable_match": true,
      "enable_dense_embedding": false,
      "enable_sparse_embedding": false
    }
  ]
}
```

**Optional fields for BYO embeddings:**

- `is_embeddings_tenant: true` -- Skip HydraDB's embedding step
- `embeddings_dimension: 1536` -- Required when `is_embeddings_tenant` is true

**Response:**

```json
{
  "status": "accepted",
  "tenant_id": "acme_corp",
  "message": "Tenant creation started in the background..."
}
```

#### `GET /tenants/infra/status`

```
GET /tenants/infra/status?tenant_id=acme_corp
```

**Response:**

```json
{
  "tenant_id": "acme_corp",
  "org_id": "free",
  "infra": {
    "scheduler_status": true,
    "graph_status": true,
    "vectorstore_status": [true, true]
  },
  "message": "Deployed infrastructure status"
}
```

Wait for **all** of:

- `infra.scheduler_status: true`
- `infra.graph_status: true`
- `infra.vectorstore_status: [true, true]` (both indices ready; `[0]` = Memories, `[1]` = Knowledge)

There is no top-level `infra_status: "ready"` field  -  readiness is the conjunction of the three fields above.

#### `GET /tenants/monitor`

```
GET /tenants/monitor?tenant_id=acme_corp
```

Returns row counts and vector dimensions per collection:

```json
{
  "tenant_id": "acme_corp",
  "normal_collection": {
    "row_count": 12847,
    "dimensions": 1536
  },
  "memory_collection": {
    "row_count": 1523,
    "dimensions": 1536
  },
  "message": "Successfully retrieved tenant collection statistics"
}
```

- `normal_collection` -- the **Knowledge** vector store (index `[1]` in `infra/status`).
- `memory_collection` -- the **Memories** vector store (index `[0]` in `infra/status`).

#### `GET /tenants/sub_tenant_ids`

```
GET /tenants/sub_tenant_ids?tenant_id=acme_corp
```

#### `GET /tenants/tenant_ids`

```
GET /tenants/tenant_ids
```

#### `DELETE /tenants/delete`

```
DELETE /tenants/delete?tenant_id=acme_corp
```

Irreversible. Tenant ID cannot be reused until cleanup completes.

### 5.2 Ingestion

#### `POST /ingestion/upload_knowledge`

Multipart form. Two input shapes:

**Files:**

```bash
curl -X POST https://api.hydradb.com/ingestion/upload_knowledge \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -F "tenant_id=acme_corp" \
  -F "sub_tenant_id=team_eng" \
  -F "files=@/path/to/doc.pdf" \
  -F "files=@/path/to/report.docx" \
  -F 'file_metadata=[
    {"file_id": "doc_001", "metadata": {"category": "legal", "timestamp": "2024-01-15"}},
    {"file_id": "doc_002", "metadata": {"category": "finance", "timestamp": "2024-02-20"}}
  ]'
```

**App sources (Slack, Notion, Jira, etc.):**

```bash
curl -X POST https://api.hydradb.com/ingestion/upload_knowledge \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -F "tenant_id=acme_corp" \
  -F "sub_tenant_id=default" \
  -F 'app_knowledge=[
    {
      "id": "slack_msg_1234567890_000100",
      "tenant_id": "acme_corp",
      "sub_tenant_id": "default",
      "kind": "message",
      "provider": "slack",
      "external_id": "1234567890.000100",
      "fields": {
        "kind": "message",
        "body": "Deployment completed successfully for v2.3.1",
        "author": "alice",
        "thread_id": "1234567890.000100",
        "created_at": "2026-05-20T10:30:00Z",
        "url": "https://acme.slack.com/archives/C01/p1234567890000100"
      },
      "metadata": {
        "container_type": "channel",
        "container_id": "C01",
        "container_name": "engineering"
      }
    }
  ]'
```

**Hard rules:**

- Always send `multipart/form-data` -- even when only sending `app_knowledge` (no files). JSON bodies return 422.
- `file_metadata` array length **must equal** `files` count.
- `app_knowledge` **must** be a JSON-encoded string when sent via multipart form (`json.dumps(...)` / `JSON.stringify(...)`).
- Each app item needs `id`, `tenant_id`, `sub_tenant_id`; put searchable content in `fields` with `kind`, `provider`, and `external_id`.
- Put channel/project/space scope in `metadata`; `external_id` is the item ID, not the container.
- `upsert` defaults to **`true`** -- re-uploading the same `source_id` overwrites the previous source. Set `upsert: false` if you need insert-only semantics.
- At recall, set `search_apps: true` on `full_recall` (`mode: "thinking"` for thread/relation expansion).
- Full contract: [App Sources](/essentials/app-sources)

**Response:**

```json
{
  "success": true,
  "message": "Upload initiated successfully",
  "results": [
    {"source_id": "doc_001", "filename": "doc.pdf",    "status": "queued", "error": null},
    {"source_id": "doc_002", "filename": "report.docx","status": "queued", "error": null}
  ],
  "success_count": 2,
  "failed_count": 0
}
```

Each `results[].source_id` is the ID to use in subsequent `verify_processing`, `delete_knowledge`, `fetch/content`, and recall calls. `filename` is `null` for app-source items.

#### `POST /ingestion/verify_processing`

```
POST /ingestion/verify_processing?tenant_id=acme_corp&sub_tenant_id=default&file_ids=doc_001&file_ids=doc_002
```

**Response:**

```json
{
  "statuses": [
    {"file_id": "doc_001", "indexing_status": "completed", "success": true},
    {"file_id": "doc_002", "indexing_status": "processing", "success": false}
  ]
}
```

Poll every few seconds until every `indexing_status` is `"completed"` or `"success"`. Fail fast on `"errored"`.

```python
import time

TERMINAL_OK = {"completed", "success"}
TERMINAL_FAIL = {"errored"}

deadline = time.time() + 300
while time.time() < deadline:
    r = client.upload.verify_processing(
        tenant_id="acme_corp",
        sub_tenant_id="default",
        file_ids=["doc_001"],
    )
    statuses = {s["indexing_status"] for s in r["statuses"]}
    if statuses & TERMINAL_FAIL:
        raise RuntimeError(f"Ingestion errored: {r}")
    if statuses.issubset(TERMINAL_OK):
        break
    time.sleep(5)
else:
    raise TimeoutError("Processing did not complete in 5 min")
```

**Status values:**

- `queued` -- waiting
- `processing` -- parsing / chunking / embedding
- `graph_creation` -- vectors done, graph still being built
- `completed` / `success` -- ready for recall
- `errored` -- failed (fail-fast in agent loops!)

Typical indexing time: under 60 s for most documents; larger items may take up to 5 minutes.

### 5.4 Recall

#### `POST /recall/full_recall`

Knowledge search (index `[1]`) with hybrid semantic + graph + keyword.

```json
{
  "tenant_id": "acme_corp",
  "sub_tenant_id": "team_eng",
  "query": "What are the Q4 deployment procedures?",
  "max_results": 10,
  "alpha": 0.8,
  "recency_bias": 0.3,
  "mode": "fast",
  "graph_context": true,
  "search_apps": true,
  "metadata_filters": {"category": "deployment"}
}
```

**Parameters:**

- `query` (required)
- `max_results` (no default -- server-determined; start with `10`)
- `alpha` (default `0.8`, range `0`–`1` or `"auto"`) -- semantic vs keyword balance
- `recency_bias` (default `0.0`, range `0`–`1`) -- boost recent content
- `mode` (default `"fast"`) -- `"fast"` or `"thinking"`
- `graph_context` (default `false`) -- include entity relationships
- `search_apps` (default `false`) -- app-aware lane: IDs, actors, threads, `parent_id`, `relations` (best with `mode: "thinking"`)
- `search_forceful_relations` (boolean, default `true`) -- in `thinking` mode, augment context with forcefully-related sources
- `additional_context` (string, optional) -- extra context to guide retrieval
- `metadata_filters` -- deterministic filtering by schema fields. Top-level keys match tenant-schema fields declared with `enable_match: true`; nest under `"additional_metadata"` for per-document fields. Exact-equality only -- no `$gte`/`$lt`/`$in`/`$regex`.

**Response (shape):**

```json
{
  "chunks": [
    {
      "chunk_uuid": "a1b2c3d4-e5f6-7890-1234-567890abcdef",
      "source_id": "doc_deployment_guide",
      "chunk_content": "Q4 deployment requires approval from tech lead and security review...",
      "source_title": "Q4 Deployment Guide.pdf",
      "source_type": "file",
      "source_upload_time": "2025-09-15T10:00:00Z",
      "relevancy_score": 0.94,
      "metadata": {"category": "deployment", "timestamp": "2024-10-01"},
      "additional_metadata": {"author": "alice@acme.com", "version": "2.1"}
    }
  ],
  "sources": [
    {
      "id": "doc_deployment_guide",
      "title": "Q4 Deployment Guide.pdf",
      "type": "file",
      "url": "https://...",
      "timestamp": "2025-09-15T10:00:00Z",
      "metadata": {"category": "deployment"},
      "additional_metadata": {"author": "alice@acme.com"}
    }
  ],
  "graph_context": {
    "query_paths": [
      {
        "triplets": [
          {
            "source": {"name": "Deployment", "type": "PROCESS"},
            "relation": {"canonical_predicate": "REQUIRES"},
            "target": {"name": "Security Review", "type": "PROCESS"}
          }
        ],
        "relevancy_score": 0.85
      }
    ],
    "chunk_relations": [],
    "chunk_id_to_group_ids": {}
  },
  "additional_context": {}
}
```

**Critical:** results live in `chunks[]` (not `results[]`). Field names are `chunk_content`, `source_title`, `relevancy_score` -- **not** `text`, `source_name`, `score`. `sources[]` is a deduplicated parallel array of source records (join on `chunk.source_id == source.id`). `graph_context` is a top-level object populated only when `graph_context: true` was requested **and** the ingested content has graph relations.

#### `POST /recall/boolean_recall`

Exact phrase or boolean keyword search.

```json
{
  "tenant_id": "acme_corp",
```

**Parameters:**

- `operator` (default `"or"`) -- `"or"` (any term), `"and"` (all terms), `"phrase"` (exact ordered phrase)

<Warning>
`boolean_recall` does **not** accept `alpha`, `mode`, `recency_bias`, `graph_context`, `metadata_filters`, or `additional_context`. Sending them returns 400. The `phrase` operator is **order-sensitive** -- "30 days refund" will not match "refund within 30 days".
</Warning>

## 9. Cookbook Patterns

### 9.1 Unified Enterprise Search ("Glean clone")

Ingest Slack, Notion, Gmail, Linear, GitHub into one tenant; recall with `search_apps: true`, `alpha=0.9`, and citations. Map each connector to a `kind` - see [App Sources](/essentials/app-sources).

```python
KIND_BY_PROVIDER = {
    "slack": "message",
    "gmail": "email",
    "notion": "knowledge_base",
    "linear": "ticket",
    "github": "ticket",
}

def app_item(provider: str, item, tenant_id: str, sub_tenant_id: str = "default"):
    kind = KIND_BY_PROVIDER[provider]
    fields = {"kind": kind}
    if kind == "message":
        fields |= {"body": item.content, "author": item.author, "thread_id": str(item.id), "url": item.url}
    elif kind == "email":
        fields |= {"subject": item.title, "body": item.content, "from": item.author, "thread_id": str(item.id), "url": item.url}
    elif kind == "knowledge_base":
        fields |= {"title": item.title, "body": item.content, "url": item.url}
    else:  # ticket
        fields |= {"title": item.title, "description": item.content, "reporter": item.author, "url": item.url}
    return {
        "id": f"{provider}_{item.id}",
        "tenant_id": tenant_id,
        "sub_tenant_id": sub_tenant_id,
        "kind": kind,
        "provider": provider,
        "external_id": str(item.id),
        "fields": fields,
        "metadata": {"container_name": getattr(item, "container", None)},
    }

sources = {"slack": slack_messages, "notion": notion_pages, "gmail": emails, "linear": issues, "github": issues}

for provider, items in sources.items():
    client.upload.knowledge(
        tenant_id="enterprise",
        app_knowledge=json.dumps([app_item(provider, item, "enterprise") for item in items]),
    )

results = client.recall.full_recall(
    tenant_id="enterprise",
    query="Q4 OKRs",
    max_results=10,
    alpha=0.9,
    mode="thinking",
    graph_context=True,
    search_apps=True,
```

## 10. Common Mistakes Checklist

| Symptom | Root cause | Fix |
|---|---|---|
| Empty recall results | Content not yet indexed | Poll `verify_processing` until `completed` |
| Memories missing from `full_recall` | Wrong endpoint | Use `recall_preferences` for memories |
| `metadata_filters` ignored | Field not declared with `enable_match: true` | Add to `tenant_metadata_schema` *before* ingestion |
| `graph_context` empty | Default is `false` | Explicitly set `graph_context: true` |
| Recency doesn't matter | `recency_bias` defaults to `0` | Set between `0.1` and `1.0` |
| Wrong data surfaces | Mismatched `sub_tenant_id` | Use the same scope on write and read |
| Forceful relations not fetching | Not in `thinking` mode, or `search_forceful_relations: false` | `mode: "thinking"` + `search_forceful_relations: true` |
| 422 on upload | `file_metadata` length mismatch | Match array length to file count |
| Schema field can't be changed | Tenant schema is immutable | Plan up-front; use `additional_metadata` for flex |
| 422 on `app_knowledge` | Sent as object, not JSON-encoded string | `json.dumps(...)` / `JSON.stringify(...)` |
| App recall misses threads/IDs | `search_apps` defaults to `false` | `search_apps: true` and `mode: "thinking"` for app data |
| Slack/Jira scoped wrong | Channel/project in `external_id` | Item ID in `external_id`; container in `metadata` |
| Legacy `knowledge_items` payload | Old app shape | Use `fields` per [App Sources](/essentials/app-sources) |
| `boolean_recall` 400 | Sent `alpha`, `mode`, or `graph_context` | Drop those params |
| Re-create tenant fails | Previous deletion still in progress | Use a different `tenant_id` or wait |
| `infer` not running | Default is `false` | Explicitly set `infer: true` |
| Partial deletion unnoticed | Trusted top-level `success` | Inspect `results[].deleted` per item |

## 11. Troubleshooting

### 11.1 Authentication failures (401)

- Header must be exactly `Authorization: Bearer <key>` (space after `Bearer`).
- Verify key in dashboard at `app.hydradb.com`.
- Check read vs read-write permissions.

```bash
curl -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  "https://api.hydradb.com/tenants/monitor?tenant_id=test"
```

### 11.2 Tenant Not Found After Creation (404)

Tenant creation is async. Always poll:

```python
import time
client.tenant.create(tenant_id="acme_corp")

deadline = time.time() + 300  # 5-minute cap
while time.time() < deadline:
    status = client.tenant.get_infra_status(tenant_id="acme_corp")
    infra = status["infra"]
    if (infra["scheduler_status"]
            and infra["graph_status"]
            and all(infra["vectorstore_status"])):
        break
    time.sleep(2)
else:
    raise TimeoutError("Tenant infra not ready after 5 min")

client.upload.knowledge(...)
```

### 11.3 422 on ingestion

Three common shapes -- all detected client-side:

```python
# WRONG: 2 files, 1 metadata
file_metadata = [{"file_id": "doc_001"}]
# CORRECT: lengths match
file_metadata = [{"file_id": "doc_001"}, {"file_id": "doc_002"}]

# WRONG: app_knowledge as object
app_knowledge = [{"id": "slack_001", "knowledge_items": [{"text": "..."}]}]
# CORRECT: JSON-encoded string with typed app-source shape
app_knowledge = json.dumps([{
    "id": "slack_msg_1234567890_000100",
    "tenant_id": "acme_corp",
    "sub_tenant_id": "default",
    "kind": "message",
    "provider": "slack",
    "external_id": "1234567890.000100",
    "fields": {"kind": "message", "body": "Deployment completed", "author": "alice"},
}])

# WRONG: MemoryItem with no content
{"infer": True}
# CORRECT: at least text or user_assistant_pairs
{"text": "User prefers dark mode", "infer": True}
```

### 11.4 Empty recall results

Walk this list in order:

1. **Still processing?** `verify_processing` → wait for `completed`.
2. **Wrong endpoint?** Use `recall_preferences` for memories, `full_recall` for knowledge.
3. **Wrong `sub_tenant_id`?** Match the scope from write time.
4. **Filters too tight?** Re-run without `metadata_filters`.
5. **App data ingested but recall is text-only?** Set `search_apps: true` (and `mode: "thinking"` for threads/relations). See [App Sources](/essentials/app-sources).

### 11.5 Rate limiting (429)

- **Use bulk endpoints** -- pass an array to `add_memory`, not a loop.
- **Exponential backoff** -- `tenacity` `wait_exponential(min=2, max=60)`.
- **Cache** -- `lru_cache` for repeated queries (FAQ patterns, user preferences).

### 12.2 Custom Chunking

Prefer **file upload** when you have a document file - HydraDB chunks internally. If you must pre-chunk, ingest **one app source per chunk** (each needs its own `id` / `external_id`; link with `fields.parent_id`):

```python

                "kind": "knowledge_base",

)
```

### 12.3 Multi-index parallel recall

`asyncio.gather` requires awaitables, so use `AsyncHydraDB` (not the sync `HydraDB` client). The sync client's `recall_preferences()` / `full_recall()` return plain dicts and will raise `TypeError` if passed to `gather`.

```python
import asyncio, os
from hydra_db import AsyncHydraDB

async_client = AsyncHydraDB(token=os.environ["HYDRA_DB_API_KEY"])

async def multi_index_recall(tenant_id, sub_tenant_id, query):
    memories, knowledge = await asyncio.gather(
        async_client.recall.recall_preferences(
            tenant_id=tenant_id, sub_tenant_id=sub_tenant_id,
            query=query, max_results=5,
        ),
        async_client.recall.full_recall(
            tenant_id=tenant_id, query=query, max_results=10,
        ),
    )
    return memories["results"] + knowledge["results"]

# If you must use the sync client, run the two calls in a thread pool instead:
from concurrent.futures import ThreadPoolExecutor

def multi_index_recall_sync(tenant_id, sub_tenant_id, query):
    with ThreadPoolExecutor(max_workers=2) as pool:
        f_mem = pool.submit(
            client.recall.recall_preferences,
            tenant_id=tenant_id, sub_tenant_id=sub_tenant_id,
            query=query, max_results=5,
        )
        f_know = pool.submit(
            client.recall.full_recall,
            tenant_id=tenant_id, query=query, max_results=10,
        )
        return f_mem.result()["results"] + f_know.result()["results"]
```

## 13. Multi-Tenant Design Patterns

### 13.1 B2B SaaS

- One `tenant_id` per customer
- `sub_tenant_id` for teams / workspaces / users
- Shared customer knowledge at tenant root (omit `sub_tenant_id` on write)
- Per-user memories at sub-tenant level

### 13.2 B2C App

- One `tenant_id` for the app
- `sub_tenant_id` per end user
- Shared knowledge (FAQs, help docs) at tenant root

### 13.3 Internal Tools

- One `tenant_id` per company
- `sub_tenant_id` per department / project
- Use metadata for fine-grained filtering

### 13.4 Environment Separation

```
my_app_prod
my_app_staging
my_app_dev
```

**Never share a tenant across environments.**

### 13.5 Scoping Hierarchy

`tenant_id` → `sub_tenant_id` → `metadata_filters`

```python
results = client.recall.full_recall(
    tenant_id="acme_corp",        # customer
    sub_tenant_id="team_mobile",  # team
    query="sprint status",
    metadata_filters={"project": "phoenix"},  # project
)
```

## 14. Performance Guidelines

### 14.1 Latency benchmarks

| Operation | p50 | p95 | p99 |
|---|---|---|---|
| Recall (`fast` mode) | ~80 ms | ~180 ms | ~250 ms |
| Recall (`thinking` mode) | ~150 ms | ~300 ms | ~400 ms |
| `add_memory` | ~50 ms | ~120 ms | -- |
| Ingestion processing | 30–90 s / doc | -- | -- |
| Tenant creation | 10–30 s | -- | -- |

### 14.2 Scaling envelope (per tenant)

- Documents: tested up to 100 K
- Chunks: tested up to 5 M
- Memories: tested up to 500 K per sub-tenant
- Sub-tenants: no hard limit (tested 10 K+)

### 14.3 Rate limits

- Recall: 1,000 req / min / tenant
- Ingestion: 100 files / min / tenant
- Memories: 500 adds / min / sub-tenant

### 14.4 Caching

- Cache repeated FAQ queries, user-preference lookups, metadata-heavy filter combos.
- TTL guidance: user preferences 5–15 min; knowledge search 1–5 min; real-time data -- don't cache.

## 16. End-to-End Examples

### 16.1 Python

```python
import os, time, json
from hydra_db import HydraDB

client = HydraDB(token=os.environ["HYDRA_DB_API_KEY"])

# 1) Create tenant
client.tenant.create(
    tenant_id="demo_tenant",
    tenant_metadata_schema={"category": "string"},
)

# 2) Wait for infra
deadline = time.time() + 300
while time.time() < deadline:
    s = client.tenant.get_infra_status(tenant_id="demo_tenant")
    infra = s["infra"]
    ready_scheduler = infra["scheduler_status"] is True
    ready_graph = infra["graph_status"] is True
    ready_vs = isinstance(infra["vectorstore_status"], list) \
        and len(infra["vectorstore_status"]) > 0 \
        and all(infra["vectorstore_status"])
    if ready_scheduler and ready_graph and ready_vs:
        break
    time.sleep(5)
else:
    raise TimeoutError("Tenant infra not ready after 5 min")

# 3) Ingest
with open("doc.pdf", "rb") as f:
    client.upload.knowledge(
        tenant_id="demo_tenant",
        files=[("doc.pdf", f, "application/pdf")],
        file_metadata=json.dumps([
            {"file_id": "doc_001", "metadata": {"category": "general"}}
        ]),
    )

# 4) Wait for processing
TERMINAL_OK = {"completed", "success"}
TERMINAL_FAIL = {"errored"}

deadline = time.time() + 300
while time.time() < deadline:
    r = client.upload.verify_processing(tenant_id="demo_tenant", file_ids=["doc_001"])
    statuses = {item["indexing_status"] for item in r["statuses"]}
    if statuses & TERMINAL_FAIL:
        raise RuntimeError(f"Ingestion errored: {r}")
    if statuses.issubset(TERMINAL_OK):
        break
    time.sleep(5)
else:
    raise TimeoutError("Processing did not complete in 5 min")

# 5) Recall
results = client.recall.full_recall(
    tenant_id="demo_tenant",
    query="What is this document about?",
    max_results=5,
    alpha=0.8,
)
print(results)
```

### 16.2 TypeScript

```typescript
import { HydraDBClient } from "@hydradb/sdk";
import fs from "fs";

const client = new HydraDBClient({ token: process.env.HYDRA_DB_API_KEY! });

// 1) Create tenant
await client.tenant.create({
  tenant_id: "demo_tenant",
  tenant_metadata_schema: [
    {
      name: "category",
      data_type: "VARCHAR",
      enable_match: true,
      enable_dense_embedding: false,
      enable_sparse_embedding: false,
    },
  ],
});

// 2) Wait for infra
const sleep = (ms: number) => new Promise((r) => setTimeout(r, ms));
const deadline = Date.now() + 5 * 60_000;

let infraReady = false;
while (Date.now() < deadline) {
  const s = await client.tenant.getInfraStatus({ tenant_id: "demo_tenant" });
  const infra = s.infra;
  const readyScheduler = infra.scheduler_status === true;
  const readyGraph = infra.graph_status === true;
  const readyVs = Array.isArray(infra.vectorstore_status)
    && infra.vectorstore_status.length > 0
    && infra.vectorstore_status.every(Boolean);
  if (readyScheduler && readyGraph && readyVs) { infraReady = true; break; }
  await sleep(5000);
}
if (!infraReady) {
  throw new Error("Tenant infra not ready after 5 min");
}

// 3) Ingest
await client.upload.knowledge({
  tenant_id: "demo_tenant",
  files: [{ path: "./doc.pdf", filename: "doc.pdf", contentType: "application/pdf" }],
  file_metadata: JSON.stringify([
    { file_id: "doc_001", metadata: { category: "general" } },
  ]),
});

// 4) Wait for processing
const TERMINAL_OK = new Set(["completed", "success"]);
const TERMINAL_FAIL = new Set(["errored"]);
const procDeadline = Date.now() + 5 * 60_000;

let processingComplete = false;
while (Date.now() < procDeadline) {
  const r = await client.upload.verifyProcessing({
    tenant_id: "demo_tenant",
    file_ids: ["doc_001"],
  });
  const statuses = new Set(r.statuses.map((x) => x.indexing_status));
  if ([...statuses].some((s) => TERMINAL_FAIL.has(s))) {
    throw new Error(`Ingestion errored: ${JSON.stringify(r)}`);
  }
  if ([...statuses].every((s) => TERMINAL_OK.has(s))) {
    processingComplete = true;
    break;
  }
  await sleep(5000);
}
if (!processingComplete) {
  throw new Error("Processing did not complete in 5 min");
}

// 5) Recall
const results = await client.recall.fullRecall({
  tenant_id: "demo_tenant",
  query: "What is this document about?",
  max_results: 5,
  alpha: 0.8,
});
console.log(results);
```
